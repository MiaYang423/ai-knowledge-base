## 一、视频原文

一张图让你看懂小龙虾（AI Agent）的核心流程，明明白白的养虾。

大家好，我是汉山。我们在微信、飞书里发的一句话，小龙虾（Agent）为什么能够理解并且还能够去执行？今天我们就用这张图给大家讲清楚。

所以我们从最开始发消息这里，比如微信、飞书，从这边发消息，发的消息会流走到我们的网关的位置里，就是这个路由或者网关。这是我们的一个数据的路口，它负责一些鉴权、限流或者一些安全检测。比如我们这个飞书或者微信怎么能够接进来，或者我们的网页是怎么能够接进来的，它需要看一些鉴权，都是通过这个进行检测。

然后网关收到的数据回到这里进行组装，它会拼上我们的历史的上下文。通常会需要我们 System Prompt 的说明，如果我们有多少次数、有哪些能力、能够做什么，都要在这里进行合成。如果我们 MCP 的外部接口能力有哪些接口可以调用，都要在这里整理进来。

我们这个历史上下文其实就是一种记忆，我们这种 Memory 的记忆。这个记忆短期的一般就存在我们内存里面，但是长期的一般通过文件或者说有一个专门的库存它。这个时候会把我们历史记忆也会拼进来，一同进行组装。

组装之后的上下文会送到交互决策模块。我们这里是一个核心模块，它收到我们组装的上下文之后，从这里最终生成的 Prompt 送给我们这个大模型，大模型进行一个处理，回复过来的内容会给到这个交互决策模块。

这个交互决策模块它会对这个回复的内容进行一个初步的分析——这个内容是最后的回复，还是回复的一些中间步骤。回复的内容如果有一个 Tool 的调用，它这个组装就会从这里去调 Tool。比如发消息、比如建一个任务、比如搜一个网页、或者读一个文件、或者执行一个命令，都是可以的。这些 Tool 都是有返回的，就是这些执行结果，比如搜网页搜到的内容要回来，回过来之后这个内容它会再进行整理，这个内容以后我们组装的这些上下文它会进行一个分析。

分析完之后如果需要模型进行再次处理，它会送到大模型里面，然后再次处理。如果还需要接着调用工具，那就再去调工具，再处理，就这么一个反复的过程，继续执行。

这就是我们说的 Agent 为什么能够持续地去做一些事情，一直做。因为它不停地去调模型，回来之后需要调什么工具，工具回来之后这个内容再送给模型，再送给工具，就是这么反复地一直去调，直到得到我们最终想要的结果。

这里有一个最终的回复，会回到网关这里，通过网关再回到我们消息的发送者，微信也好、飞书也好，回来。这样我们一个完整的数据流就结束了。

我们给 Agent 下了一个命令，然后它执行完之后我们收到一个结果，这就是我们这个 Agent 的一个完整的数据流和工作过程。

### 2.2 知识地图

```
AI Agent 完整数据流
│
├── 1. 入口层
│   └── 消息来源（微信 / 飞书 / 网页）
│       └── 网关 / 路由
│           ├── 鉴权
│           ├── 限流
│           └── 安全检测
│
├── 2. 上下文组装层
│   ├── System Prompt（角色说明 / 能力边界）
│   ├── MCP 外部接口列表（可调用的工具清单）
│   └── Memory 历史记忆
│       ├── 短期记忆（内存）
│       └── 长期记忆（文件 / 数据库）
│
├── 3. 交互决策层（核心模块）
│   ├── 接收组装好的上下文
│   ├── 生成 Prompt → 送给大模型
│   ├── 接收大模型回复
│   └── 判断回复类型
│       ├── 最终回复 → 返回结果
│       └── 中间步骤 → 调用 Tool
│
├── 4. 工具执行层（Tool）
│   ├── 发消息
│   ├── 建任务
│   ├── 搜索网页
│   ├── 读取文件
│   └── 执行命令
│       └── 执行结果回传 → 重新组装上下文
│
└── 5. 循环执行机制
    └── 模型 → 工具 → 模型 → 工具 → … → 最终结果
        └── 结果通过网关返回给消息发送者
```

### 2.3 核心概念精讲

#### 概念一：网关 / 路由

**是什么**：Agent 系统的数据入口，所有外部消息必须经过这里才能进入系统。

**为什么重要**：没有网关，任何来源的消息都可以直接打到 Agent 内部，系统既不安全也无法管控。

**详细解释**：
- **鉴权**：验证消息来源是否合法，比如飞书 / 微信的 Token 校验
- **限流**：控制请求频率，防止被刷爆
- **安全检测**：过滤恶意输入或违规内容

**实际应用场景**：你在飞书里 @ 一个 Agent 机器人，飞书会带着签名把消息推送过来，网关先验签，验证通过才往后走。

---

#### 概念二：上下文组装（Context Assembly）

**是什么**：在把消息送给大模型之前，把所有相关信息拼装成一个完整 Prompt 的过程。

**为什么重要**：大模型本身是无状态的，它只能看到你这次送进去的内容。所有的"记忆"和"能力"，都必须在这一步显式地拼进去，模型才能知道。

**详细解释**：

| 组装内容 | 说明 | 存储位置 |
|---|---|---|
| System Prompt | 告诉模型它是谁、有什么能力、有什么限制 | 代码 / 配置文件 |
| MCP 工具列表 | 列出所有可调用的外部接口，让模型知道能用什么工具 | 配置 / 注册中心 |
| 短期记忆 | 本次会话的历史对话记录 | 内存 |
| 长期记忆 | 跨会话的用户信息或知识积累 | 文件 / 数据库 |

**实际应用场景**：你第二天再问 Agent 昨天聊的事，它能记得，就是因为长期记忆被拼进了这次的上下文里。

---

#### 概念三：交互决策模块

**是什么**：Agent 的"大脑中枢"，负责把组装好的上下文送给模型、接收模型回复，并判断下一步该做什么。

**为什么重要**：这个模块决定了 Agent 是"回答完就结束"还是"继续调工具干活"，是 Agent 能自主执行复杂任务的关键。

**详细解释**：
- 收到模型回复后，判断回复类型：
  - **最终回复**：直接返回给用户
  - **工具调用指令**：解析出要调哪个工具、传什么参数，去执行
- 工具执行完毕后，把结果重新拼入上下文，再次送给模型
- 如此循环，直到模型给出最终回复

**实际应用场景**：你让 Agent 帮你"搜一下最新的 AI 新闻然后整理成表格发给我"，它会先调搜索工具，把结果拿回来，再调模型整理，最后再发消息给你，整个过程全靠这个模块调度。

---

#### 概念四：循环执行机制（Agent Loop）

**是什么**：Agent 在"调用大模型 → 调用工具 → 调用大模型"之间不断循环，直到任务完成的执行模式。

**为什么重要**：这是 Agent 区别于普通聊天机器人的本质——它不是一问一答，而是能持续自主地把一件复杂的事情做完。

**详细解释**：

```
用户下达指令
      ↓
  大模型分析
      ↓
  需要用工具？
  ┌───┴───┐
 是       否
  ↓       ↓
调工具   返回最终结果给用户
  ↓
结果拼回上下文
  ↓
再次送给大模型
  ↓
（回到"需要用工具？"继续循环）
```

**实际应用场景**：让 Agent 自动完成一个多步骤任务，比如"查数据 → 写报告 → 发邮件"，每一步都是一次循环，Agent 自己判断下一步该干什么，不需要人工介入。

---

### 2.4 速查表

| 模块 | 核心职责 | 关键技术点 |
|---|---|---|
| 网关 / 路由 | 数据入口，管控安全 | 鉴权、限流、安全检测 |
| 上下文组装 | 拼装完整 Prompt | System Prompt、Memory、MCP 工具列表 |
| 短期记忆 | 本次会话历史 | 存内存，会话结束即清除 |
| 长期记忆 | 跨会话知识积累 | 存文件或数据库，持久化 |
| 交互决策模块 | 调度模型与工具 | 判断是最终回复还是工具调用 |
| 工具层（Tool） | 执行具体动作 | 搜网页、读文件、发消息、执行命令等 |
| Agent Loop | 持续执行复杂任务 | 模型↔工具反复循环直到任务完成 |
| 网关（返回） | 把结果送回用户 | 原路返回给微信 / 飞书等消息来源 |

---

### 2.5 延伸思考

1. **循环几次才停？** Agent Loop 理论上可以无限循环，实际系统中是怎么防止它"死循环"或者跑偏的？有哪些常见的终止条件设计？

2. **记忆该存多少？** 短期记忆放内存、长期记忆放数据库，但上下文窗口是有长度限制的。当历史记忆太长塞不进去时，Agent 应该怎么做取舍？

3. **工具越多越好吗？** MCP 工具列表在上下文组装时全部塞给模型，如果注册了几百个工具，模型还能准确选择正确的工具吗？工具的数量和质量之间应该如何权衡？

### 2.6 一句话记住它

> Agent 不神秘：一条消息进来，经过网关、拼上记忆和工具清单、送给大模型、调工具、再送给模型……就这样反复循环，直到把事情做完再把结果还给你。
