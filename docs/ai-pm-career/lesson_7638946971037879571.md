# 教案：Agent Memory 面试工程细节拆解

> 来源视频：[抖音博主 · 抖音](https://www.douyin.com/video/7638946971037879571)
> 适合人群：正在准备 AI Agent / 大模型方向技术面试的产品经理、算法工程师或后端开发者。

---

## 一、视频原文

美团这场 AI 二面专门挑那种你以为懂了、其实没做过的点来问。Memory 这类题最容易答虚——你说长期记忆、短期记忆，面试官下一句就会问：怎么写入？怎么去重？冲突了谁生效？

我是小哲，这期重点看你能不能把记忆系统讲到工程细节。

**第一道题**：用一个例子解释 Memory 是怎么做的，以及怎么插入上下文的？

可以举过敏信息这个例子。用户说"我对花生过敏"，系统先抽取成结构化记忆，比如 `allergy = peanut`，再写入长期存储。下次用户问推荐餐厅时，检索到这条记忆，插到用户问题前面的 Memory 区，让模型在生成答案前先看到约束。

**第二道题**：Memory 怎么考虑上下文长度问题？

核心是不能把所有记忆都塞进去。工程上一般会做三步：先按用户场景和时间过滤；再用向量或者关键词召回相关记忆；最后按重要性和新鲜度排序，只放最相关的几条。上下文预算要提前分配，不能挤占工具结果和用户当前问题。

**第三道题**：结构化信息和文本信息之间如何区分，会不会重复？

结构化信息适合放稳定字段，比如过敏、偏好、地区、会员等级，方便精确过滤和规则判断。文本信息适合放长描述，比如用户的写作风格、项目背景、历史目标。会不会重复，关键看是否有同一个事实 ID，抽取时要做去重和归并。

**第四道题**：记忆冲突怎么解决？比如用户前后说了不同的过敏信息？

这题考的不是存储，而是冲突策略。一般要保留历史版本，但生效时按可信度、时间和来源做判断。用户明确说"我现在不对花生过敏了"，就让新记忆覆盖旧记忆，并把旧记录标成失效。高风险信息（比如过敏）最好在关键场景再追问确认。

**第五道题**：记忆是怎么写入上下文的？上下文从头到尾结构是怎样的？

常见结构是：System prompt 放规则，Developer 或业务指令放流程，Memory 放用户长期信息，Conversation 放最近对话，Tool result 放工具返回，最后是当前用户问题。Memory 一般不要直接混进历史对话里，而是单独成块，标明来源、时间和约束强度。

**第六道题**：上下文结构是不是 System prompt → Memory → Tool result → Conversation 这四个环节？

可以这么理解，但面试里要说得更严谨。真实系统里通常还会有开发者指令、检索文档、短期摘要、工具调用记录和当前任务状态。顺序也不是固定模板，关键是高优先级规则靠前，当前问题靠后，工具结果和记忆要有清晰边界，避免模型混淆来源。

**第七道题**：上下文超长了怎么办？有没有考虑过上下文压缩？

肯定要考虑。做法不是简单截断，而是分层压缩：最近几轮对话原文保留；较早对话压成任务摘要；工具结果只保留结论、参数和关键证据；长期记忆走检索注入。这样既控制 Token，又尽量保留决策链和用户约束。

**第八道题**：压缩会丢失工具调用历史，导致模型重复调用工具，这个问题怎么解决？

工程上要把工具历史变成可读的状态，而不是只依赖原始调用记录。比如记录：已经查过什么、用了什么参数、结果是什么、是否还有效。压缩后把它写进任务状态区。模型如果想重复调用工具，先检查状态，只有信息过期、缺失，或者用户要求刷新时，才再次调用。

好，这套面题我们就分享到这里。这套题表面看是在问 Memory 和上下文管理，其实真正考的是 Agent 落地能力。希望在评论区看到你的答案。

### 2.3 核心概念精讲

#### 记忆写入与检索注入

**是什么**：将用户信息抽取为结构化或文本记忆，存入长期存储，在需要时检索并注入上下文。

**为什么重要**：模型本身无状态，Memory 机制是 Agent 实现个性化、跨会话连贯性的核心基础设施。

**详细解释**：
- 抽取阶段：将自然语言（"我对花生过敏"）转为键值对（`allergy = peanut`）
- 存储阶段：写入向量数据库或结构化存储
- 注入阶段：在用户提问前，检索相关记忆，插入上下文 Memory 区块，并标明来源和约束强度

**实际应用场景**：餐厅推荐、健康管理、写作助手等需要跨会话记住用户偏好的场景。

---

#### 上下文结构设计

**是什么**：定义上下文各区块的内容、顺序和边界，确保模型能正确理解优先级和信息来源。

**为什么重要**：上下文结构混乱会导致模型混淆来源、忽略约束，直接影响生成质量和安全性。

**详细解释**：

| 区块 | 内容 | 位置建议 |
|------|------|----------|
| System prompt | 全局规则、角色定义 | 最前 |
| 开发者指令 | 业务流程、安全约束 | 靠前 |
| Memory | 用户长期信息（单独成块） | 中前 |
| 检索文档 / 短期摘要 | 相关知识、历史摘要 | 中 |
| Tool result | 工具返回结论 | 中后 |
| Conversation | 最近几轮对话 | 后 |
| 当前问题 | 用户本次输入 | 最后 |

**实际应用场景**：任何多轮对话 Agent 的 prompt 工程设计。

---

#### 记忆冲突处理

**是什么**：当用户前后提供矛盾信息时，系统决定哪条记忆生效的策略机制。

**为什么重要**：冲突处理不当会导致错误约束持续生效（如已解除的过敏限制仍影响推荐），甚至产生安全风险。

**详细解释**：
- 保留历史版本，不直接覆盖删除
- 生效判断维度：可信度（明确声明 > 隐含推断）、时间（新 > 旧）、来源（用户直接说 > 系统推断）
- 高风险信息（如过敏、医疗）在关键场景主动追问确认

**实际应用场景**：健康类 Agent、个人助理、电商推荐系统。

---

#### 上下文压缩与工具状态持久化

**是什么**：在上下文超长时，通过分层压缩减少 Token 消耗，并通过工具状态记录避免重复调用。

**为什么重要**：简单截断会丢失关键决策链；重复工具调用会增加成本和延迟，甚至导致逻辑错误。

**详细解释**：
- 分层压缩策略：
  - 最近几轮：原文保留
  - 较早对话：压成任务摘要
  - 工具结果：只保留结论 + 参数 + 关键证据
  - 长期记忆：检索注入（不全量放入）
- 工具状态持久化：记录"查过什么、用了什么参数、结果是什么、是否还有效"，写入任务状态区，模型重复调用前先检查状态

**实际应用场景**：长任务 Agent、多步骤工作流、需要工具调用的复杂对话系统。

### 2.4 速查表

| 面试题 | 核心要点 |

### 2.5 延伸思考

1. **Memory 的去重和归并在工程上应该发生在写入时还是检索时？** 两个时机各有什么优缺点，实际系统中如何权衡？

2. **上下文中 Memory 区块"标明来源、时间和约束强度"，模型真的能利用这些元信息吗？** 有没有更可靠的工程手段来确保约束被遵守？

3. **高风险信息（如过敏）在关键场景追问确认，这个"关键场景"如何识别？** 是规则判断还是模型判断，各自的风险点是什么？

---

### 2.6 一句话记住它

> 回答 Memory 面试题，别停在"长期/短期"概念层——面试官真正想听的是：你怎么写入、怎么去重、冲突了谁生效、超长了怎么压。
