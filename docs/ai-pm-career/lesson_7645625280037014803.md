## 一、视频原文

一起来看看这套美团的Agent面经，看看你能答上来几道。这套题整体难度中等，但是最后一题很多人会答得很虚。我是小哲，点赞收藏加关注，我们马上开始。

第一道题：什么是MCP，它的核心目标是什么？MCP可以理解成Agent连接外部工具和数据源的一套标准协议，核心目标是把工具发现、上下文传递和调用方式标准化，让不同客户端和服务端可以复用，而不是每个项目都重新接一遍。

第二道题：MCP主要解决了什么问题？它解决的是工具接入碎片化的问题。过去每个Agent框架都有自己的工具协议，千奇百怪，高度耦合，描述也不同。MCP想做的是一层标准接口，让工具能力更容易被发现、标用和治理。

第三道题：MCP里的host是什么？host是承载用户交互和模型能力的应用，比如IDE、桌面助手或Agent平台，它负责发起连接、管理会话、协调模型和工具结果。简单说，host是整个MCP使用体验的入口和控制面。

第四道题：MCP里的client是什么？client通常运行在host的内部，负责和某个MCP server建立连接，发送请求，接收工具列表和执行结果。它像适配层，把host的需求转换成协议消息，也把server的能力暴露给上层。

第五道题：MCP里的server是什么？server负责提供具体能力，比如文件系统、数据库、搜索、代码仓库或业务系统。它声明有哪些工具、参数怎么传、返回什么结果。工程上server还要处理鉴权、限流、日志和错误信息。

第六道题：MCP和function calling有什么区别？function calling更偏模型侧的工具调用格式，重点是让模型按schema产出参数。MCP更偏系统侧协议，重点是工具如何被发现、连接和治理。两者可以配合，不是简单替代关系。

第七道题：MCP和plugins有什么区别？plugins更偏具体平台的扩展机制，通常依赖某个生态。MCP更强调跨应用、跨工具的标准化连接。面试里可以说：plugins是一种产品形态，MCP更像底层协议和集成方式。

第八道题：生产环境落地MCP要注意什么？关键是通信安全、权限隔离、工具白名单、参数校验、审计日志和超时熔断。工具发现不能等于工具可用，敏感操作要二次确认，还要考虑多租户版本兼容和server异常时的降级。

好，这套面经我们就分享到这里。这套题表面看是在问MCP概念，其实真正考的是Agent落地能力。希望在评论区看到你的答案。

### 2.3 核心概念精讲

#### MCP（模型上下文协议）

**是什么**：Agent连接外部工具和数据源的一套标准协议，目标是把工具发现、上下文传递和调用方式标准化。

**为什么重要**：解决了过去各Agent框架工具协议碎片化的问题，让工具能力可以跨项目复用。

**详细解释**：
- 标准化三件事：工具发现、上下文传递、调用方式
- 让不同客户端和服务端可以复用，无需每个项目重新接入

**实际应用场景**：多个Agent平台共享同一套工具服务（如搜索、数据库、代码仓库）。

---

#### MCP三层架构：Host / Client / Server

**是什么**：MCP将系统分为三个角色，各司其职。

**为什么重要**：理解三层架构是回答MCP架构类面试题的核心，也是工程落地的基础。

**详细解释**：
| 角色 | 位置 | 职责 |
|------|------|------|
| Host | 最外层，用户侧 | 用户交互入口，发起连接，管理会话，协调模型和工具结果 |
| Client | Host内部 | 连接Server，发送请求，做协议适配 |
| Server | 工具侧 | 提供具体能力，声明工具列表，处理鉴权/限流/日志 |

**实际应用场景**：IDE（Host）→ MCP Client → 代码仓库MCP Server。

---

#### MCP vs Function Calling vs Plugins

**是什么**：三种不同层次的工具集成方式。

**为什么重要**：面试高频混淆点，需要能清晰区分。

**详细解释**：
| 概念 | 层次 | 重点 | 关系 |
|------|------|------|------|
| Function Calling | 模型侧 | 让模型按schema产出参数 | 与MCP可配合 |
| MCP | 系统侧协议 | 工具发现、连接、治理 | 底层协议 |
| Plugins | 平台扩展机制 | 依赖特定生态 | 产品形态 |

**实际应用场景**：Function Calling负责模型输出参数，MCP负责把参数传给正确的工具并返回结果。

---

#### 生产环境MCP落地要点

**是什么**：将MCP从原型部署到生产环境时需要关注的工程安全与稳定性要点。

**为什么重要**：这是面试最后一题，也是最难答、最能体现工程落地能力的考点。

**详细解释**：
- **安全**：通信安全、权限隔离、工具白名单、参数校验
- **可观测**：审计日志、超时熔断
- **可靠性**：敏感操作二次确认、多租户版本兼容、server异常降级
- 核心原则：**工具发现 ≠ 工具可用**

**实际应用场景**：企业级Agent平台的MCP服务治理、安全审计。

### 2.5 延伸思考

1. 视频说"工具发现不能等于工具可用"，在实际系统中如何设计工具的权限校验机制，确保Agent只能调用被授权的工具？

2. MCP和Function Calling"可以配合，不是替代关系"——在一个完整的Agent调用链中，两者分别在哪个环节发挥作用，数据流是怎样的？

3. 多租户场景下MCP的版本兼容是个难题，如果不同客户使用不同版本的MCP Server，系统应该如何设计来保证兼容性和稳定性？

### 2.6 一句话记住它

> MCP是Agent世界的"USB接口"——不管什么工具、什么平台，插进去就能用，而生产落地的难点不在于"能不能连"，而在于"安不安全、稳不稳定"。
