## 一、视频原文

一起来看看字节的 AI Agent 实习二面面经，看看你能答上来几道。这套题整体难度偏难，但是前面是基础，后面才是真正区分水平的地方。我是小哲，点赞收藏加关注，我们马上开始。

**第一道题：项目被整体拷打时应该怎么讲？**
不要只备业务背景，核心是讲清楚你解决了什么问题、系统链路怎么走、指标怎么衡量、线上遇到过什么坑。面试官更想听实质内容，比如延迟、成本、稳定性、可观测性，你有没有真的做过。

**第二道题：你对多模态大模型有哪些了解？**
可以从输入能力讲起，比如图像、语音、视频和文本的统一理解。关键不是报模型名，而是讲落地边界。多模态是和内容理解、检索增强所结合，但工程上要关注输入预处理、推理成本和结果校验。

**第三道题：Agent Skills 是怎么设计和使用的？**
Skills 本质是把可复用能力封装成工具，说明参数、约束和执行逻辑。设计时要让模型知道什么时候调用、怎么传参、失败怎么兜底。工程上一般还要做权限隔离、调用日志和结果校验。

**第四道题：如何加强大模型的记忆机制和多轮对话能力？**
短期靠上下文窗口，长期靠摘要、向量库和结构化用户画像。关键是不要把所有历史都塞回去，而是按任务检索相关记忆。多轮对话还要维护状态、意图和追问结果，避免越聊越偏。

**第五道题：多 Agent 执行策略的智能选择和切换机制怎么设计？**
可以把任务先拆成规划、执行、校验几类角色，再用路由器根据任务类型、置信度和失败次数切换 Agent。核心是要有退出条件和重试机制，否则多个 Agent 很容易循环讨论，成本高还不稳定。

**第六道题：SSE 的局限性有哪些？**
SSE 是服务端到客户端的单向流，用来做大模型打字机效果很方便，但它不支持双向通信，也存在连接数、代理超时和断线重连的问题，复杂交互场景里 WebSocket 往往更灵活。

**第七道题：如果 LoRA 效果不好，你会怎么排查和改进？**
先看数据质量和任务分布，再看基座模型是否匹配，最后看 Rank、学习率、训练轮数这些参数。LoRA 的问题是表达能力有限，容易过拟合小数据。改进可以做数据清洗、增量评测和合并策略对比。

**第八道题：RAG 的动态知识更新怎么做？**
核心是把文档更新、切片、向量化、索引刷新做成流水线。增量更新要记录版本和时间戳，旧知识要能失效。线上还要考虑缓存、召回延迟和一致性，不然用户问到的可能还是过期答案。

**第九道题：RAG 的评估方案怎么设计？**
不要只看最终回答好不好，可以拆成检索召回率、上下文相关性、答案忠实度和端到端满意度。工程上要准备标准问题集，记录命中文档、引用片段和模型输出，这样才能定位是检索错还是生成错。

**第十道题：如何评估复杂任务执行准确率的提升？**
复杂任务要拆成子步骤评估，比如规划是否合理、工具调用是否正确、最终结果是否达标。只看成功率太粗，更好的方式是建立任务级人工标注关键检查点，再对比改动前后的成功率、耗时和成本。

**第十一道题：MCP 和 Function Calling 的区别是什么？**
Function Calling 更像模型调用单个函数的协议，重点是参数结构化。MCP 更像一套连接外部工具和资源的上下文协议，适合把文件、数据库、服务能力统一暴露给 Agent。一个偏调用格式，一个偏生态连接。

### 2.3 核心概念精讲

#### 概念一：Agent Skills 设计

**是什么**：将可复用的能力封装成标准化工具，供 Agent 在执行任务时按需调用。

**为什么重要**：Skills 是 Agent 能力边界的核心载体，设计质量直接决定 Agent 能不能稳定完成任务。

**详细解释**：

| 设计维度 | 关键要点 |
|--------|--------|
| 封装内容 | 参数说明、约束条件、执行逻辑 |
| 调用时机 | 让模型明确知道什么场景触发 |
| 传参规范 | 参数结构化，避免歧义 |
| 失败兜底 | 明确失败后的降级或重试策略 |
| 工程保障 | 权限隔离、调用日志、结果校验 |

**实际应用场景**：Agent 调用搜索工具时，Skills 定义了"输入关键词、返回 Top-K 结果、超时后降级到缓存"的完整逻辑。

---

#### 概念二：RAG 的评估体系

**是什么**：对检索增强生成系统进行分层拆解评估，而不是只看最终答案质量。

**为什么重要**：RAG 出问题可能发生在检索层或生成层，不拆开评估就无法定位根因、无法针对性优化。

**详细解释**：

| 评估维度 | 评估内容 | 定位问题 |
|--------|--------|--------|
| 检索召回率 | 相关文档有没有被召回 | 检索层问题 |
| 上下文相关性 | 召回内容和问题是否匹配 | 检索层问题 |
| 答案忠实度 | 生成答案有没有幻觉 | 生成层问题 |
| 端到端满意度 | 用户最终是否满意 | 整体问题 |

**实际应用场景**：准备标准问题集，记录每次命中的文档片段和模型输出，对比改动前后各层指标变化。

---

#### 概念三：多 Agent 调度机制

**是什么**：通过路由器根据任务特征动态分配和切换不同 Agent 角色，协同完成复杂任务。

**为什么重要**：单 Agent 处理复杂任务容易出错，多 Agent 分工可以提升准确率，但如果没有调度机制，容易陷入循环讨论，成本高且不稳定。

**详细解释**：

- **角色拆分**：规划 Agent、执行 Agent、校验 Agent 各司其职
- **路由依据**：任务类型、当前置信度、历史失败次数
- **核心约束**：必须设定退出条件和重试上限，否则多 Agent 会无限循环

**实际应用场景**：复杂代码生成任务中，规划 Agent 拆解需求，执行 Agent 写代码，校验 Agent 跑测试，路由器根据测试结果决定是否重试或终止。

---

#### 概念四：MCP 与 Function Calling 的区别

**是什么**：两种让大模型连接外部能力的不同协议，定位和适用场景不同。

**为什么重要**：面试高频考点，混淆两者会暴露对 Agent 工程架构理解不深。

**详细解释**：

| 对比维度 | Function Calling | MCP |
|--------|----------------|-----|
| 定位 | 模型调用单个函数的协议 | 连接外部工具和资源的上下文协议 |
| 核心重点 | 参数结构化 | 生态连接与统一暴露 |
| 适用场景 | 单次工具调用 | 文件、数据库、服务能力统一接入 Agent |
| 一句话 | 偏调用格式 | 偏生态连接 |

**实际应用场景**：让 Agent 查询数据库用 MCP 统一暴露数据源；让模型在对话中调用计算函数用 Function Calling。

### 2.5 延伸思考

1. **多 Agent 系统的退出条件应该怎么设计才合理？** 纯靠失败次数阈值够用吗，还是需要引入任务完成度评估或时间成本约束，如何在稳定性和灵活性之间取得平衡？

2. **RAG 的评估体系建立之后，如何形成持续优化的闭环？** 线上用户的隐式反馈（如追问、不满意重问）能否自动转化为评估信号，驱动检索和生成的迭代？

3. **MCP 作为"生态连接协议"，在实际 Agent 工程中会带来哪些新的安全和权限挑战？** 当 Agent 通过 MCP 统一接入文件、数据库和外部服务时，如何防止越权访问或数据泄露？
