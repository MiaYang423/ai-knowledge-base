## 一、视频原文

来来来，你在 RAG 当中向量数据库怎么样去构建的吧，因为它的向量数据库建得好不好会直接决定 RAG 系统的检索效果。我就结合之前做过的项目说一下整个流程。

首先就是数据准备。结论，我们先要把原始的文档收集起来，可能是 PDF、Word 的，或者是数据库里的文本这些。第一步大家看着简单，但其实非常关键。比如说 PDF，我们就需要把它转变成文本，但有的 PDF 是扫描版，就得先走 OCR 识别。我们之前就遇到过，扫描版的技术文档识别出来的文本质量很差，后面就专门调了 OCR 的纯手工人工校议，才保证这个数据的质量。

第二个就是文档的切分，这个是特别重要的，因为大模型的上下文窗口有限，没办法把一整本的说明书都扔进去，所以要把整个文档切成一个一个的块。这里面有几个小策略：最简单的就是按固定大小分，比如 500 个 token 一块，但这样很容易就会把完整的段落切断，语义就不连贯了。所以更好的方式就是按语义切分，比如按照段落、标题这些自然结构来切。然后我们项目里面用的是基于文档结构的切分，比如 Markdown 按标题层级来切，HTML 按标签去切。然后我们还加了 overlap 的机制，每个块之间有 100 到 200 个 token 的重叠，这样就可以既保证上下文的连贯，也可以检索的时候不漏掉关键信息。

第三个就是向量化，也就是 embedding。我们就是要把切好的每一个块都给它转化成向量，这个时候就会用到 embedding 模型，比如说 OpenAI 的 text embedding，或者说是一些其他的开源模型。模型会把文本映射成一个高维的向量，一般是 512 维或者是 1536 维。重点是语义相似的文本，它们在向量空间的距离也会比较近。这个过程我们就是用批量处理，因为一个一个调 API 太慢了，我们之前处理 10 万条文档，做了批处理和多线程优化，速度就快了好几倍。

然后第四个就是把向量存储起来。处理完之后这里面就要做向量数据库的选型，市面上有很多选择，比如开源的、商业化的，有一些是可以支持混合检索的，既可以做向量检索，也可以做关键词检索。存的时候我们就不光存向量，还要存原始的文本、文档的来源、元数据这些，因为检索出来要把原文返回给大模型。另外还要建索引，用一些索引算法能够大幅提高检索的速度。

最后就是持续优化和维护。向量库并不是建完就可以了，还要定期更新，比如新的文档进来，我们还要重新做切分、embedding、入库。有些文档过时就要删掉或者做标记。我们项目里面设置了自动化的数据同步流程，每天晚上跑一次增量更新。还要监控检索的效果，比如看召回率、准确率这些指标，如果发现效果下滑就得调整切分策略，或者是换一个 embedding 模型。

总的来说，向量数据库的构建流程就是：数据准备、文档切分、向量化、入库检索、持续优化，每一个环节都很重要。作为产品经理的话，就不仅需要懂技术流程，还要根据业务场景去调优，比如什么样的切分粒度更合适，用哪个 embedding 模型更划算，怎么平衡检索速度和精度，这些都需要在实际项目中不断地打磨。

### 2.3 核心概念精讲

#### 概念一：文档切分（Chunking）

**是什么**：将长文档拆分成若干小块（chunk），以适配大模型有限的上下文窗口。

**为什么重要**：大模型无法一次性处理一整本说明书，切分质量直接影响后续检索的准确性，切得太碎语义丢失，切得太大又放不进去。

**详细解释**：

| 切分策略 | 方式 | 优点 | 缺点 |
|---|---|---|---|
| 固定大小切分 | 每 500 token 切一刀 | 简单易实现 | 容易切断完整段落，语义不连贯 |
| 按语义结构切分 | 按段落、标题等自然结构 | 语义完整 | 块大小不均匀 |
| 基于文档结构切分 | Markdown 按标题层级、HTML 按标签 | 贴合文档逻辑 | 需针对格式做适配 |

**Overlap 机制**：相邻块之间保留 100~200 token 的重叠区域，既保证上下文连贯，又避免检索时漏掉跨块的关键信息。

**实际应用场景**：技术文档知识库、产品说明书问答系统，切分策略需根据文档格式（PDF/Markdown/HTML）分别定制。

---

#### 概念二：向量化（Embedding）

**是什么**：用 embedding 模型将文本块转化为高维数值向量，使计算机能够度量文本之间的语义相似度。

**为什么重要**：向量化是 RAG 能做语义检索的基础，决定了"用户问的问题"和"知识库里的内容"能不能被正确匹配上。

**详细解释**：
- 常用模型：OpenAI text embedding、各类开源 embedding 模型
- 输出维度：通常为 512 维或 1536 维
- 核心原理：语义相近的文本，在向量空间中欧氏距离或余弦距离更小
- 工程优化：单条调用 API 速度极慢，应使用**批处理 + 多线程**，视频中 10 万条文档经优化后速度提升数倍

**实际应用场景**：大规模知识库构建时，embedding 的成本和速度是关键工程瓶颈，需要在模型精度和调用成本之间做权衡。

---

#### 概念三：向量数据库存储与检索

**是什么**：专门用于存储和高效检索向量数据的数据库，支持以"语义相似度"为条件进行查询。

**为什么重要**：向量数据库的选型和索引建立方式直接影响检索速度和准确率，是 RAG 系统的核心基础设施。

**详细解释**：

| 存储内容 | 说明 |
|---|---|
| 向量本身 | 用于相似度检索 |
| 原始文本 | 检索命中后返回给大模型 |
| 元数据（来源、时间等） | 用于过滤、溯源、展示 |

- **混合检索**：同时支持向量检索（语义匹配）和关键词检索（精确匹配），两者结合可提升召回质量
- **索引算法**：在向量库上建立专用索引（如 HNSW 等），大幅提升检索速度

**实际应用场景**：企业知识库、客服问答系统，需根据数据量级、是否需要混合检索、部署方式（私有化/云端）选择合适的向量数据库。

---

#### 概念四：持续优化与监控

**是什么**：向量库上线后的运营维护机制，包括增量更新、过时文档清理和检索效果监控。

**为什么重要**：知识库是动态的，文档会新增、修改、过时，检索效果也可能因数据变化而下滑，"建完不管"会导致系统逐渐失效。

**详细解释**：
- **增量更新**：新文档进来需重走切分→embedding→入库全流程，可设置自动化定时任务（如每天夜间执行）
- **过时文档处理**：删除或打标记，避免检索到错误的旧信息
- **效果监控指标**：
  - **召回率**：相关文档有没有被找到
  - **准确率**：找到的文档有没有真正回答问题
- **调优手段**：效果下滑时调整切分策略，或更换更适合业务场景的 embedding 模型

**实际应用场景**：任何生产环境中的 RAG 系统都需要这套机制，尤其是文档频繁更新的业务场景（如政策库、产品文档库）。

### 2.4 速查表

| 环节 | 核心任务 | 关键细节 / 踩坑点 |
|---|---|---|
| 数据准备 | 收集并清洗原始文档 | 扫描版 PDF 必须先过 OCR，否则文本质量极差 |
| 文档切分 | 将长文档切成小块 | 优先按语义/文档结构切，加 100~200 token overlap |
| 向量化 | 用 embedding 模型转化为向量 | 批处理 + 多线程，避免逐条调 API 太慢 |
| 向量存储 | 存入向量数据库并建索引 | 同时存向量、原文、元数据；建索引提升检索速度 |
| 持续优化 | 增量更新 + 效果监控 | 监控召回率/准确率，效果下滑及时调整策略或模型 |

### 2.5 延伸思考

1. **切分粒度如何决定？** 块太小会导致语义不完整，块太大会超出上下文窗口，对于不同类型的业务文档（法律合同 vs. 技术 FAQ vs. 产品手册），最优的切分粒度和策略应该怎么评估和选取？

2. **混合检索的适用边界在哪里？** 向量检索擅长语义匹配，关键词检索擅长精确匹配，在什么业务场景下应该偏向哪种，两者结果如何融合排序（如 RRF 算法），这对最终答案质量有多大影响？

3. **embedding 模型换代后怎么办？** 如果业务上线后发现原来的 embedding 模型效果不够好，需要更换新模型，那意味着所有历史文档都要重新向量化和入库，如何在不停服的情况下完成这次迁移，同时保证检索效果的平稳过渡？

---
