## 一、视频原文

大家好，今天给大家解读一篇来自 LangChain 的文章。

Agent 等于 Model 加 Harness。换个说法，如果你不是模型本身，那你就属于 Harness。模型只是大脑，Harness 才是让大脑干活的工具箱。光有模型只能输出文本，有了 Harness 才能变成 Agent。

从模型的角度看，还有三个无法自己解决的问题。第一，无法持久记忆，对话结束就忘了。第二，无法执行代码，只能说不能做。第三，知识有时效性，训练截止日期后的事一概不知。这三个问题必须靠 Harness 来解决。

作者从"想要什么行为"反推"需要什么 Harness"，设计梳理出六大核心组件：文件系统是持久存储基石；Bash 加代码执行是实用工具杀手锏；环境提供安全隔离；记忆与搜索实现持续学习；上下文管理对抗腐败；长期自主执行处理复杂任务。

先看基础三件套。文件系统是持久存储基石，让 Agent 有工作台，能读写代码、文档、数据，工作可增量进行，不用塞进上下文。Bash 加代码执行是实用工具，给模型一台电脑，让它自己想办法。沙箱环境提供安全隔离，不在本地跑危险代码，环境可动态创建销毁。

记忆与搜索实现持续学习，记忆文件在 Agent 启动时注入上下文，搜索工具过滤过去的实用信息。上下文管理对抗腐败——模型在上下文填满后推理能力下降，有三种策略：压缩摘要、卸载近期内容；工具输出卸载；技能尽量精简披露。长期自主执行是复杂任务的终极目标，有三个关键模式：文件系统加 Git 跟踪跨会话的工作进度；Runloop 连接 Agent 的退出，尝试用新上下文重新注入原始任务强制继续；规划与自验证，分阶段目标，检查中间结果，失败时反思重试。

一个有意思的现象：今天的 Agent 产品在训练时，模型和 Harness 是一起参与的。这形成了一个飞轮：发现有用原语，加入 Harness，用新 Harness 训练下一代模型，循环继续。但也带来过度耦合问题，模型被训练成用特定方式调用工具，改了工具逻辑性能就下降。

Harness 工程的未来有三个开放问题。第一，并行编排，几百个 Agent 同时在共享代码库上工作。第二，自我整顿，Agent 分析自己的执行轨迹，识别并修复失败。第三，动态组装，根据任务实时组装工具上下文。

Harness 不会消失，只会进化。这篇文章的核心启发是：不要把 Agent 想成一个黑盒，它是模型加框架的组合，模型负责智能，Harness 负责让智能变得有用。如果你想构建自己的 Agent，可以从这个公式出发：先想清楚想要什么行为，再反推需要什么 Harness。好的系统设计能把一的智能放大成十的生产力。

### 2.3 核心概念精讲

#### 概念一：Harness（框架/工具箱）

**是什么**：围绕模型构建的一切外部组件的总称，包括工具、存储、执行环境、记忆系统等，是让模型从"会说话"变成"能干活"的关键。

**为什么重要**：模型本身只能生成文本，无法记忆、无法执行、无法感知时间。Harness 弥补了这三大先天缺陷，是 Agent 能力的真正来源。

**详细解释**：

> 公式：**Agent = Model（大脑）+ Harness（工具箱）**
> 如果某个组件不是模型本身，它就属于 Harness。

设计 Harness 的正确姿势是**从行为反推工具**：先想清楚 Agent 要做什么，再决定需要哪些 Harness 组件，而不是堆砌工具。

**实际应用场景**：构建一个能自动写代码并提交 PR 的 Agent，需要文件系统（读写代码）、Bash 执行（运行测试）、沙箱（安全跑代码）、Git（跟踪进度）等多个 Harness 组件协同。

---

#### 概念二：六大核心组件

**是什么**：LangChain 文章梳理出的六个 Harness 必备模块，覆盖存储、执行、安全、记忆、上下文、任务管理六个维度。

**为什么重要**：这六个组件对应模型的六类能力短板，缺少任何一个都会让 Agent 在特定场景下失效。

**详细解释**：

| 组件 | 核心作用 | 解决的问题 |
|------|----------|------------|
| 文件系统 | 持久存储工作台 | 对话结束数据不丢失，工作可增量推进 |
| Bash + 代码执行 | 给模型一台"电脑" | 从"只能说"变成"真能做" |
| 沙箱环境 | 安全隔离执行 | 危险代码不在本地跑，环境可动态创建销毁 |
| 记忆与搜索 | 持续学习能力 | 记忆文件启动时注入，搜索工具过滤历史有效信息 |
| 上下文管理 | 对抗上下文腐败 | 上下文填满后推理能力下降，需压缩/卸载/精简 |
| 长期自主执行 | 处理复杂长任务 | 通过 Git 跟踪、Runloop 续接、规划自验证实现跨会话任务 |

**实际应用场景**：上下文管理中的三种策略（压缩摘要、工具输出卸载、技能精简披露）在处理超长代码库分析任务时尤为关键。

---

#### 概念三：飞轮效应与过度耦合

**是什么**：当前 Agent 产品在训练时，模型和 Harness 是联合参与的，两者相互强化形成飞轮，但同时也导致模型与特定工具深度绑定。

**为什么重要**：这解释了为什么直接修改现有 Agent 的工具逻辑往往会导致性能下降——模型已被训练成依赖特定的工具调用方式。

**详细解释**：

```
飞轮正向循环：
发现有用原语 → 加入 Harness → 用新 Harness 训练下一代模型 → 发现更多有用原语

过度耦合风险：
模型被训练成用特定方式调用工具
→ 修改工具逻辑
→ 模型行为与工具预期不匹配
→ 整体性能下降
```

**实际应用场景**：在已有 Agent 产品基础上做二次开发时，需要特别注意不要随意替换底层工具实现，否则可能破坏模型已学到的调用习惯。

---

#### 概念四：长期自主执行的三种模式

**是什么**：让 Agent 能够跨会话、持续完成复杂任务的三种工程设计模式。

**为什么重要**：真实世界的复杂任务往往无法在单次对话中完成，必须有机制保证任务状态不丢失、执行能续接、中间结果能验证。

**详细解释**：

| 模式 | 机制 | 作用 |
|------|------|------|
| 文件系统 + Git | 将工作进度写入文件并用 Git 版本控制 | 跨会话跟踪工作状态，随时可恢复 |
| Runloop 续接 | Agent 退出时，用新上下文重新注入原始任务 | 强制任务继续，不因上下文满而中断 |
| 规划与自验证 | 分阶段设定目标，检查中间结果，失败时反思重试 | 保证复杂任务质量，具备自我纠错能力 |

**实际应用场景**：让 Agent 自动完成一个需要数小时的大型重构任务，三种模式缺一不可。

### 2.5 延伸思考

1. **上下文腐败是 Harness 工程中最难解决的问题之一**——压缩摘要会损失细节，卸载工具输出会丢失中间状态，你认为在实际项目中应该优先采用哪种策略？不同任务类型（代码生成 vs. 长文写作 vs. 数据分析）的最优策略是否相同？

2. **飞轮效应带来的过度耦合**意味着模型和 Harness 越来越难以分离。随着这一趋势加深，未来"换一个更好的模型"是否会变得越来越困难？这对 Agent 系统的长期可维护性意味着什么？

3. **"动态组装"被列为 Harness 工程的未来方向之一**——Agent 根据任务实时决定用哪些工具。但工具选择本身也需要智能，这是否会引发"用 Agent 管理 Agent"的递归问题？边界应该划在哪里？

---

> 模型是大脑，Harness 是手脚——搭 Agent 不要堆工具，先想清楚要什么行为，再反推需要什么 Harness，好的设计能把一分智能放大成十分生产力。
