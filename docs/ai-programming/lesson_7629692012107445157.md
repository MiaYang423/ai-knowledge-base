## 一、视频原文

大家好，我是小鑫。在之前的分享中，很多朋友想分清楚 Claude 客户端跟 Claude 模型的区别。本期视频主要讲他们之间的区别，以及当你让 Claude 跑一个任务——比如竞品分析——这个词进来之后，Claude 客户端跟模型分别都会做什么事情。以这个东西为例去解释一下，这次请求当中都发生了什么。

那么简单来说，Claude 的 Anthropic、Claude Sonnet、Haiku 这些，他们就是大脑；然后客户端本身就是手脚，去执行各种东西。我们只有脑子、没有东西去执行是不行的，我们执行但没有聪明脑子也是不行的，这就是他们之间的一个关系。

我们具体来看，当我们去选择 Claude 客户端、给他一个任务说"帮我做一下竞品分析"，会发生什么事情？

那么首先，Claude 这个客户端进来之后，他会去加载 Claude 的 System Prompt。比如说你让他说话风格一定是说中文的，模型也可以让他的风格是简洁的，或者说让他不要省略、把东西都很详细地列出来——你在这里边会定义很多静态的一些对话定义，就像对他的要求一样。

然后这个时候会加载 Memory。比如说你之前跨会话的时候，是不是之前就已经让他做过别的竞品分析？那么之前我们在竞争中别的竞品是什么、其他的竞品是哪两个、是什么，以及我们的产品是什么——这些都会被加载进来。

然后就是 Claude 客户端自己定义的 340 种预加载的 Tools，一些工具可以供后面的模型去看，我们到底用哪个 Tools。

然后他会把这些东西打包好，作为一个上下文——把 Claude 这个客户端软件作为这个事情的打包上下文，连同用户这次的"竞品分析"请求，一块打包给模型。

好了，这些东西给了脑子之后，那脑子可以是 Claude 自己的，Anthropic 上面的，也可以是你自己接入任何模型，比如 Gemini 或者其他的模型。他在训练过程中可能不是按照这些 Tools 去训练的，所以实际的适配效果咱们也不好说。因为毕竟我们也不好说他是不是像 Claude 一样去适配得那么好。

先表达一下：模型在训练过程中可能会把对这些工具的使用作为训练的一部分，那么如果说一些其他的大模型他们不是 Claude 家的话，他可能就没有经过这些训练，不知道怎么用这些 Tools，会造成一些效果上的损失——这是有可能的。

然后我们所有上下文的内容打包好给了模型之后，模型会去 Thinking。这个我们平常在用他们的时候现在都已经见到过了，他会去想竞品是什么，他会去分析 Memory 里之前有没有做过类似的课题、之前是怎么做的，他会去看哪些工具要去调用。

想完这些事情之后，他可能会在这个过程中发出一条明确的指令，比如说去 Web Search 一下——就是他当中有工具叫 Web Search，就去往上搜索。然后模型发出这样一条指令说，我们现在要进行一个工具的调用了，这个工具是 Web Search。模型思考之后发出了这样一条指令，然后这个指令由 Claude 客户端这个软件去执行，去 Web Search、去网上查。查到这个东西之后返回给模型，模型理解完之后去生成，生成完之后再让 Claude 客户端保存到一个比如说 Root 文件夹的报告里面，然后给我们去看。

这就是整个过程。这个过程中 Claude 客户端这个软件，以及 Claude 的模型 Anthropic 上面的这些，他们怎么协同的，以及我们正常的一次请求进来之后到返回结果都发生了什么——希望这条视频能够帮大家去理解这些东西。我是小鑫，关注单击爆赞。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**当你对 Claude 说一句"帮我做竞品分析"，客户端和模型会按照一套固定流程分工协作——搞懂这个分工，你才能真正用好 AI 工具。**

---

### 2.2 知识地图

```
给 Claude 发一句话后都发生了什么
│
├── 1. 核心分工：客户端 vs 模型
│   ├── 客户端 = 手脚（执行层）
│   └── 模型 = 大脑（思考层）
│
├── 2. 客户端做的事（请求进来后）
│   ├── 加载 System Prompt（风格/语言/格式要求）
│   ├── 加载 Memory（跨会话历史记忆）
│   ├── 加载预定义 Tools（约 340 种工具）
│   └── 打包上下文 → 发给模型
│
├── 3. 模型做的事（拿到上下文后）
│   ├── Thinking（分析任务、回顾记忆、选择工具）
│   ├── 发出工具调用指令（如 Web Search）
│   └── 接收结果 → 生成回复
│
├── 4. 客户端再次介入（执行工具调用）
│   ├── 执行 Web Search
│   ├── 将结果返回给模型
│   └── 保存最终输出（如写入报告文件）
│
└── 5. 注意事项：换用第三方模型的风险
    ├── 第三方模型可能未针对这些 Tools 训练
    └── 可能导致工具调用效果下降
```

### 2.3 核心概念精讲

#### 概念一：客户端与模型的分工

**是什么**：Claude 客户端（软件）和 Claude 模型（AI 大脑）是两个不同的东西，各司其职。

**为什么重要**：很多人以为"Claude"是一个整体，搞不清楚为什么换个模型效果就变差了——理解这个分工是用好 AI 工具的基础。

**详细解释**：

| 角色 | 类比 | 负责什么 |
|------|------|----------|
| Claude 客户端 | 手脚 | 加载配置、调用工具、保存文件、管理上下文 |
| Claude 模型（如 Sonnet、Haiku） | 大脑 | 理解任务、推理思考、决定调用哪个工具、生成回复 |

**实际应用场景**：你在 Claude 客户端里说"帮我做竞品分析"，客户端负责把你的话、历史记忆、工具列表打包好交给模型；模型负责想清楚怎么做，再指挥客户端去执行搜索、保存报告等操作。

---

#### 概念二：上下文打包（Context Assembly）

**是什么**：客户端在把请求发给模型之前，会把多种信息合并成一个完整的"上下文包"再一起发送。

**为什么重要**：模型看到的不只是你说的那一句话，而是经过客户端精心组装的完整信息——这决定了模型能不能做出准确判断。

**详细解释**：上下文包通常包含三类信息：

- **System Prompt**：静态配置，比如"请用中文回答""回答要详细不要省略"
- **Memory**：跨会话记忆，比如"上次你告诉我竞品是 A 和 B，我们自己的产品是 C"
- **Tools 列表**：客户端预定义的约 340 种工具，告诉模型有哪些能力可以调用

**实际应用场景**：即使你只说了一句"帮我做竞品分析"，模型收到的实际上是一个包含历史背景、行为规范、可用工具的完整信息包，所以它才能直接开始工作而不需要你重复解释背景。

---

#### 概念三：工具调用（Tool Use）的完整循环

**是什么**：模型思考后发出"调用某个工具"的指令，客户端执行，结果再返回给模型继续处理——这是一个来回的闭环。

**为什么重要**：这解释了为什么 AI 能"上网搜索""保存文件"——这些动作不是模型自己做的，而是模型发指令、客户端代劳。

**详细解释**：

```
模型 Thinking
    ↓
模型发出指令："调用 Web Search，搜索竞品信息"
    ↓
客户端执行 Web Search
    ↓
搜索结果返回给模型
    ↓
模型生成最终回复
    ↓
客户端将报告保存到指定文件夹
    ↓
用户看到结果
```

**实际应用场景**：做竞品分析时，模型不是凭空生成内容，而是先指挥客户端去网上搜索最新信息，拿到数据后再整合生成报告。

---

#### 概念四：第三方模型的适配风险

**是什么**：在 Claude 客户端里换用非 Anthropic 的模型（如 Gemini 等），可能因为训练方式不同而导致工具调用效果变差。

**为什么重要**：很多人以为换个更便宜或更强的模型就能无缝替代，但实际上工具调用能力是需要专门训练的。

**详细解释**：

| 模型来源 | 是否针对 Claude Tools 训练 | 预期效果 |
|----------|--------------------------|----------|
| Anthropic 原生模型 | 是 | 工具调用稳定，效果好 |
| 第三方模型（如 Gemini） | 不确定 | 可能不知道如何正确使用这些 Tools，效果有损失 |

**实际应用场景**：如果你在 Claude 客户端里接入第三方模型后发现工具调用不稳定或结果变差，原因很可能就在这里。

---

### 2.4 速查表

| 环节 | 执行者 | 具体动作 |
|------|--------|----------|
| 加载 System Prompt | 客户端 | 读取语言、风格、格式等静态配置 |
| 加载 Memory | 客户端 | 读取跨会话历史（如之前分析过的竞品） |
| 加载 Tools 列表 | 客户端 | 准备约 340 种可用工具供模型选择 |
| 打包上下文 | 客户端 | 将以上内容 + 用户请求合并发给模型 |
| Thinking（推理） | 模型 | 分析任务、回顾记忆、决定调用哪个工具 |
| 发出工具调用指令 | 模型 | 明确说"现在要调用 Web Search" |
| 执行工具调用 | 客户端 | 实际去网上搜索并拿回结果 |
| 生成最终回复 | 模型 | 基于搜索结果生成分析内容 |
| 保存输出 | 客户端 | 将报告写入指定文件夹 |

---

### 2.5 延伸思考

1. **System Prompt 和 Memory 都属于"上下文"，但性质不同**——System Prompt 是你主动设定的规则，Memory 是 AI 自动积累的记忆。如果两者发生冲突（比如 System Prompt 说"简洁回答"，但 Memory 里记录了"用户喜欢详细报告"），模型会怎么处理？这对你设计 Prompt 有什么启发？

2. **视频提到客户端预定义了约 340 种 Tools**——工具越多，模型需要从中选择的成本就越高。如果工具太多导致模型选错工具，应该怎么优化？是减少工具数量，还是在 System Prompt 里做更明确的引导？

3. **第三方模型在 Claude 客户端里效果可能变差**，但第三方模型在自己原生的客户端里可能表现很好——这说明"模型能力"和"客户端适配"是两个独立的维度。在选择 AI 工具组合时，你会优先考虑哪个维度，为什么？

### 2.6 一句话记住它
