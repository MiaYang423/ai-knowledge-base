## 一、视频原文

我用了一下午做出了一个完整的网页应用，还部署上线了，任何人都能访问。关键是我一行代码都没写。今天我把整个过程拆解给你，跟着做你也行。

整个过程就四步：第一步明确需求，第二步生成原型，第三步原型转代码，第四步部署上线。接下来我一步一步跟你说清楚。

先说第一步：头脑风暴，明确需求。这一步是整个流程中最重要的一步，很多人会忽略它。但我必须告诉你，你最终做出来的东西好不好，80% 取决于你这一步做得扎不扎实。

具体怎么做呢？打开 ChatGPT、Claude 或者 Gemini，随便选一个你顺手的就行。然后把你脑海中那个模糊的想法告诉它，不用担心说得不够清楚，你就像跟一个朋友聊天一样，把你想做什么、给谁用、大概用什么功能，一股脑说出来就行。

关键在于你不要只聊一轮就结束了，你要多跟 AI 聊几轮，让它反问你。你没想到的地方它会帮你想到，你不确定的地方它会帮你分析利弊。就这样一来一回聊个十几二十轮，你会发现你的想法越来越清晰了。

聊到什么程度算够了呢？最好是能明确到：你的应用有哪些模块，每个模块有哪些功能，一共有几个页面，每个页面上大概有什么内容，用户操作流程是什么样的。当然如果你觉得一次性想不了那么全也没关系，你可以先明确两三个核心功能先做出来，后面再慢慢加。

最后让 AI 把你们所有的讨论成果汇总成一份文档，这份文档有个专业的名字叫 PRD，就是产品需求文档。

好，拿到这份 PRD 之后，我们进入第二步：生成原型界面。这一步我们需要用到一个工具叫 Google Stitch。你把刚才那份完整的 PRD 文档扔给 Stitch，它会根据你的需求自动帮你画出应用的 UI 界面。

生成之后你看一下，如果觉得布局不好看或者不合理，你就继续跟它说，让它调整。比如你想要暗黑风格你就告诉它，你觉得某个按钮位置不对你也告诉它。就这样反复微调，直到所有页面让你满意为止。

这一步结束之后，你手上应该有一套完整的、覆盖所有页面的应用原型。这个原型看起来已经十分接近最终成品了，但它还只是一个壳，接下来我们要把它变成真正能跑的代码。

进入第三步：原型转代码。在 Stitch 里面有一个功能，可以直接把你做好的原型导出到 Google IDX。导出之后，Google IDX 就会基于你的原型文件自动生成一整套可以运行的前端代码。它默认使用的技术栈是 React 加 TypeScript，你不懂这些也没关系，不影响你使用。

如果你的项目比较简单，纯前端不涉及什么后端服务器之类的，那你甚至可以在 Google IDX 里面就把所有事情做完，在里面预览、调试、修改都可以。

这里特别提醒：IDX 可以一键把代码推送到 GitHub，这一步一定要做，相当于给你的代码上了个保险，改崩了随时能回退。

如果你的项目比较复杂，需要接后端服务，或者需要更精细的调整，那你可以把代码从 GitHub 拉到本地，用专业的编程工具继续开发。这里推荐几个工具：Cursor、Windsurf、吹都可以，选你喜欢的就行。遇到报错不用慌，把报错信息直接扔给 AI，让它帮你修就行。

遵守一个原则：小步迭代，改一点测一点，不要一次性改太多东西。

好，代码搞定之后，我们来到最后一步：部署上线。这一步超级简单。打开 Vercel 这个网站，用你的 GitHub 账号登录，然后选择你刚才推送到 GitHub 的代码仓库。如果项目需要环境变量，比如一些 API Key 之类的，在部署页面填一下，然后点击部署。Vercel 会给你生成一个专属的网址，任何人通过这个网址都能访问你的应用。

到这里，你已经拥有一个完全属于自己的网页应用了。从一个模糊的想法，到一个真正上线的产品，整个过程你可能一行代码都没有手动写过。

最后快速回顾一下：第一步，用 ChatGPT 跟 AI 聊清楚需求，输出一份 PRD 文档；第二步，把 PRD 扔给 Google Stitch，生成完整应用原型；第三步，从 Stitch 导出到 Google IDX，自动生成代码，推送到 GitHub；第四步，打开 Vercel，一键部署上线。

我知道你可能觉得这些工具名字听着就头大，但你真正上手之后就会发现，每步其实只需要点几下鼠标、跟 AI 说几句话。还是先照着做一遍，做完你就懂了。

我把完整的提示词、工具链接、原态码全部整理好了，拿到就能直接用，评论区留言，免费发给大家。

### 2.3 核心概念精讲

#### 概念一：PRD（产品需求文档）

**是什么**：对你要做的应用进行完整描述的文档，包含模块、功能、页面结构和用户操作流程。

**为什么重要**：
视频明确指出，最终产品质量的 80% 取决于需求阶段做得扎不扎实。PRD 是后续所有步骤的输入源头，它越清晰，AI 生成的原型和代码就越贴近你的预期。

**详细解释**：

| PRD 应该包含的内容 | 举例说明 |
|---|---|
| 应用有哪些模块 | 用户模块、内容模块、设置模块 |
| 每个模块有哪些功能 | 用户模块：注册、登录、修改密码 |
| 一共有几个页面 | 首页、详情页、个人中心页 |
| 每个页面有什么内容 | 首页：搜索框、推荐列表、导航栏 |
| 用户操作流程 | 用户登录 → 浏览内容 → 收藏 → 查看收藏 |

**实际应用场景**：
和 AI 聊天时，不要只说"我想做一个笔记应用"就结束，要通过多轮对话让 AI 反问你、帮你补充细节，最后让 AI 把所有讨论结果汇总成这份文档。

---

#### 概念二：原型（UI Prototype）

**是什么**：用 Google Stitch 生成的应用界面视觉稿，看起来像真实产品，但还不能运行，只是一个"壳"。

**为什么重要**：
原型让你在写任何代码之前就能看到产品长什么样，可以低成本地调整布局、风格和交互逻辑，避免代码写完了才发现界面不对。

**详细解释**：
- 输入 PRD → Stitch 自动生成所有页面的 UI
- 可以用对话方式继续调整：改风格（如暗黑风格）、改布局、改按钮位置
- 反复微调直到满意，再进入下一步

**实际应用场景**：
生成原型后，逐页检查每个页面是否符合预期，用自然语言告诉 Stitch 哪里不对，让它修改，直到整套原型通过"验收"再导出。

---

#### 概念三：小步迭代原则

**是什么**：在用 AI 修改代码时，每次只改一个小地方，改完测试没问题再改下一处，不要一次性改动大量内容。

**为什么重要**：
一次改动太多，出错时很难定位问题在哪里。小步迭代可以让每次改动的影响范围可控，出错了也容易回退和排查。

**详细解释**：
- 遇到报错 → 把报错信息完整复制给 AI → 让 AI 只修这一个问题
- 修完 → 测试 → 没问题 → 再处理下一个问题
- 配合 GitHub 备份：每完成一个稳定版本就推送一次，随时可以回退

**实际应用场景**：
在 Cursor 或 Windsurf 里调试复杂项目时，不要让 AI "帮我把整个功能重写一遍"，而是"帮我修这个具体的报错"。

---

#### 概念四：GitHub + Vercel 部署链路

**是什么**：GitHub 用于存储和备份代码，Vercel 读取 GitHub 仓库并自动将代码部署为可访问的网页应用。

**为什么重要**：
这条链路让"把代码变成任何人都能访问的网站"这件事只需要几次点击，不需要自己配置服务器。

**详细解释**：

| 工具 | 作用 | 关键操作 |
|---|---|---|
| GitHub | 代码托管和版本备份 | IDX 一键推送；改崩了可回退 |
| Vercel | 自动部署和生成网址 | GitHub 授权 → 选仓库 → 填环境变量 → 部署 |

**实际应用场景**：
项目需要用到第三方 API（如 OpenAI API Key）时，不要把 Key 写死在代码里，而是在 Vercel 部署页面的环境变量一栏填入，更安全也更易管理。

### 2.4 速查表

| 步骤 | 目标 | 使用工具 | 输出物 |
|---|---|---|---|
| 第一步：明确需求 | 把模糊想法变成清晰文档 | ChatGPT / Claude / Gemini | PRD 产品需求文档 |
| 第二步：生成原型 | 把文档变成可视化界面 | Google Stitch | 覆盖所有页面的 UI 原型 |
| 第三步：原型转代码 | 把界面变成可运行的代码 | Google IDX / Cursor / Windsurf | 可运行的前端代码 + GitHub 仓库 |
| 第四步：部署上线 | 把代码变成任何人可访问的网站 | Vercel | 专属网址 |

| 项目复杂度 | 推荐路径 |
|---|---|
| 简单项目（纯前端） | Stitch → IDX → Vercel，全程在线完成 |
| 复杂项目（需要后端） | Stitch → IDX → GitHub → 本地用 Cursor/Windsurf 开发 → Vercel |

### 2.5 延伸思考

1. **需求阶段聊多少轮才算"够"？** 视频说聊 10~20 轮，但不同复杂度的项目差异很大。你怎么判断 PRD 已经详细到可以进入下一步，而不是在需求阶段过度消耗时间？

2. **这套流程的边界在哪里？** 视频覆盖的是前端网页应用，如果产品需要用户登录、数据库存储、支付功能等真正的后端能力，这套"零代码"流程还能走多远，哪个环节会最先遇到瓶颈？

3. **AI 生成的代码，长期维护会面临什么问题？** 当应用上线后需要持续迭代，或者出现复杂 bug 时，完全依赖 AI 修代码的模式是否可持续？什么情况下必须引入真正懂代码的人？
