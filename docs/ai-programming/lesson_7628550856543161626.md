# 教案：AI 回答数据类问题的产品解决方案

> 来源视频：[抖音博主 · 抖音](https://www.douyin.com/video/7628550856543161626)
> 适合人群：正在准备 AI 大模型方向面试、或负责 AI 产品落地的开发者和产品经理

---

## 一、视频原文

用户反馈我们的 AI 在回答数据类实时问题的时候会出现幻觉，大模型本身知识会有滞后性，那你对于这个产品的解决方案思路是什么样子的？

其实有两个核心想法，就是首先是大模型的幻觉问题，然后就是知识的滞后性，我会从两个方面来给出产品的方案。

先说大模型的幻觉问题吧。针对幻觉问题，我的核心逻辑就是不要让它去瞎编数字。最直接的方法就是通过工具调用，比如说当用户问今天的股价是多少，或者一个销售额是多少这一类数据问题的时候，AI 不应该直接回答，而是要先发起查数据的任务，然后调用预定义的 API 或者数据库接口，拿到真实的数据之后，再去组织语言输出。

如果工具都不可用呢？那就需要在提示词里面明确告诉模型，如果对数字没有把握，或者训练数据里面也没有相关的信息，就必须强制执行一个拒绝策略，不可以瞎猜，可以直接告诉用户"抱歉，无法确认"。听起来是简单的事情，但做出来反而更专业——用户听到"它不知道"，比看到一个看起来像知道但其实是错的数字，要好得多。

然后就是滞后性的问题，你是怎么解决这种时效性比较强的问题？比如昨天的排行榜，或者大家最新的一些动态。

我会接入一些联网的搜索能力，但是联网只能搜到公开的信息，对于企业内部的数据，外网是搜不到的。这个时候就必须接入企业的私有知识库，也就是典型的 RAG，就是把公司最新的财报、差旅文档、业务数据提前整理成一个知识源，通过现在所谓的检索增强生成来实现。

那在用户体验上，你还做过哪些优化？

在回答方面增加确定性感受。比如说有一个报表数据查询系统，同样一个问题，可以增加"从财务系统查询的数据"这样的标注，让用户一眼就知道这个数据是查出来的，不是模型编出来的，心理上会更踏实。但是对于高风险的回答，比如涉及金额还有 KPI 的数据，还会加一个可点击的引用标注，用户点开它就能看到数据来源，还有查询时间，增加可信度。

基本上都说到了。从长远来看，其实还可以建立一个用户反馈的闭环，主动在产品里面加一个"反馈错误"的按钮，鼓励用户标记不准确的内容。这样就可以定期去分析那些 bad case，看看到底是哪些环节出了问题——是工具调用失败了，还是知识库没更新，或者说是提示词设计有问题，然后针对性地进行优化。高频的问题甚至还可以自动创建工单，推动数据源或者策略的迭代。

### 2.3 核心概念精讲

#### 概念一：工具调用（Tool Calling）

**是什么**：让 AI 在回答数据类问题时，不直接生成答案，而是先触发一个外部工具（API 或数据库接口）去查询真实数据，再基于查询结果组织输出。

**为什么重要**：大模型的训练数据是静态的，它"记住"的数字随时可能是错的。工具调用让数字来源从"模型记忆"变成"实时查询"，从根源上切断幻觉。

**详细解释**：

| 场景 | 不用工具调用 | 用工具调用 |
|------|-------------|-----------|
| 用户问"今天股价多少" | 模型直接生成一个数字（可能是编的） | 先调用行情 API，拿到实时数据，再回答 |
| 用户问"这个月销售额" | 模型从训练数据中猜测 | 调用数据库接口，查真实报表数据 |

**实际应用场景**：企业内部的销售数据查询、实时行情播报、库存查询等对数字准确性要求高的场景。

---

#### 概念二：提示词强制拒绝策略

**是什么**：在系统提示词中明确约束模型，当它对某个数字没有把握、或训练数据中没有相关信息时，必须输出拒绝回答，而不是猜测或编造。

**为什么重要**：用户宁愿听到"我不知道"，也不愿看到一个看起来自信但实际上是错的数字——后者会直接损害产品信任度。

**详细解释**：
- 在提示词中加入明确规则，例如："如果你对以下问题中的数字没有确定的依据，必须回复'抱歉，无法确认该数据，建议查阅官方来源'，不得猜测。"
- 这是工具调用不可用时的兜底手段。

**实际应用场景**：工具调用失败、API 超时、数据库无对应记录时的降级处理。

---

#### 概念三：RAG（检索增强生成）

**是什么**：将企业私有数据（财报、文档、业务数据）预先整理成知识库，当用户提问时，先从知识库中检索相关内容，再把检索结果作为上下文喂给模型生成回答。

**为什么重要**：联网搜索只能覆盖公开信息，企业内部的数据外网根本搜不到。RAG 是解决企业场景知识滞后问题的标准方案。

**详细解释**：

```
用户提问
  → 检索模块在私有知识库中找到相关文档片段
  → 将文档片段 + 用户问题一起发给大模型
  → 模型基于真实文档内容生成回答
```

**实际应用场景**：企业内部知识问答、基于最新财报的数据分析、内部政策查询等。

---

#### 概念四：用户反馈闭环

**是什么**：在产品中内置"反馈错误"入口，收集用户标记的不准确回答，定期分析 bad case，定位问题环节并持续优化。

**为什么重要**：AI 产品的数据准确性问题不可能一次性解决，需要通过真实用户反馈来发现盲区，形成持续迭代的机制。

**详细解释**：

| 分析维度 | 对应优化动作 |
|---------|------------|
| 工具调用失败 | 修复 API 或增加重试机制 |
| 知识库内容过期 | 更新知识源，缩短更新周期 |
| 提示词设计有问题 | 调整约束规则或输出格式 |
| 高频 bad case | 自动创建工单，推动数据源/策略迭代 |

**实际应用场景**：所有对准确性有要求的 AI 产品，尤其是金融、医疗、企业数据分析类场景。

### 2.5 延伸思考

1. **工具调用失败时，提示词拒绝策略是最优解吗？** 还是应该有更细粒度的降级策略——比如告诉用户"数据查询暂时不可用，以下是截至上次更新的参考数据"？两种方式各自的边界在哪里？

2. **RAG 的知识库更新频率如何决定？** 如果财报每季度更新一次，但用户每天都在问，知识库的"新鲜度"和"维护成本"之间怎么做取舍？哪些数据适合实时同步，哪些适合批量更新？

3. **用户反馈闭环在实际落地中面临哪些阻力？** 用户愿意主动点"反馈错误"的比例通常很低，如何提升反馈率？除了主动反馈，还有哪些方式可以被动收集 bad case 信号？
