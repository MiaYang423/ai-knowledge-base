# 教案：氛围编程已死，Agent工程时代开启

## 一、视频原文

大家好，我是阿里长长，今天聊一个被 Karpathy 亲口宣判结束的时代——氛围编程。

前几天 Karpathy 跟红杉的 Stephanie Zhan 录了一期对话，标题就一句话：From Vibe Coding to Agentic Engineering。距离他第一次抛出 vibe coding 这个词，刚好一年。一年前他说，未来你只要描述想要啥，模型就给你写出来，当时整个圈子都嗨了，Cursor、Claude Code、Codex、Windsurf 一个个起飞，所有人都觉得程序员要消失了。

结果一年过去，他亲口说，那个时代结束了。

我自己这一年是 Claude Code 的重度用户，跟你说真的，那种我啥也不懂但功能出来的快感是真的爽。打开 Cursor 跟他说一句"帮我做个 dashboard"，回车，啪一下界面就出来了。爽完之后我就开始懵，为啥我的登录流程里有一个叫 Banana.ts 的文件，这是哪个 agent 的恶趣味，我也不知道，能跑就行。

这种状态，Karpathy 管它叫 vibe coding——靠氛围，靠感觉，靠运气。但他说这玩意儿只配做原型。

接下来他抛了个挺狠的框架，叫 Software 3.0：Software 1.0，人手写代码；Software 2.0，神经网络从数据里学；Software 3.0，LLM 本身变成计算基础，自然语言就是编程接口，agent 就是 runtime。听着挺潮对，但你回头琢磨一下，他说的是实话——你现在写代码的时候真的在写代码吗？你大部分时间是在写 spec，在配 context，在跑测试，在审 diff，在管 agent 的全流程。你写的是给模型看的提示词，模型才是真正在敲键盘的那个家伙。

这一年我最大的感受是，老程序员变成新手了。我有个朋友写了十几年，够骨灰级的那种，他前两个月跟我吐槽，用 Claude Code 打了一周，觉得自己跟刚毕业的实习生没两样。不是说写不出代码，是 agent 一次性给他吐了八百行他没写过的东西，他看半天看不懂，但又必须 code review，必须签字背锅。他说他现在不是在编程，是在签字。

这就是核心矛盾：模型越强，传统意义上的"会写代码"，越不能保证你真的懂这个系统。

所以 vibe coding 和 agentic engineering 的区别在哪？Vibe coding 是你说做个东西，模型给你做，看起来还行，可能能跑，前端可能是个大型车祸现场，但你看到像素动了，demo 能演了，你就觉得自己很牛。Agentic engineering 是另一套东西，它要 memory、要 tools、要测试、要多 agent 协作、要 sandbox、要监控、要 code review，还要 spec、安全审计，一堆其他的工程纪律。一个像魔术师从帽子里变兔子，一个像工程纪律——兔子还在，但兔子身上挂着 CI 了。

Karpathy 说了句让我印象很深的话：一个没有约束的 coding agent，不是同事，是一个拥有 root 权限、精力无限、偶尔会犯浑的初级工程师。这不是劳动力策略，这是人质危机。

这话听着有点刺耳，但是事实。你给 agent 放开了，让他能改文件、能装包、能拿 secret、能推送分支、能部署，听着像是给你配了个全能助理，但只要他有一次犯浑，你就完了。

所以未来不是一个万能 agent，是一堆专精 agent，每个有自己的工具、自己的权限、自己的成功标准，你的活是调度他们。

这事最有意思的是，市场已经在用脚投票了。微软和 GitHub 已经推出一个叫 GitHub Certified Agentic AI Developer 的认证，考试编号 GH-600。一年前 vibe coding 听着像个段子，现在已经有证书认证考试了。

但另一边，Atlassian、Duolingo、Cloudflare、IBM、Salesforce、Snap，都在裁人。Cloudflare 一次裁了 1100 多个岗位，名义上说就是为了适配 agentic AI 时代。听着挺残酷，但我想了想，AI 其实没有单纯替代谁，它做的是另一件事——它把那些看起来在工作，但其实只是在会议之间传话的岗位，暴露出来了。

我有时候觉得，这个行业正在经历一次身份重写。程序员这三个字的含义在变，以前它意味着你会写代码、调 bug、架构系统，现在它意味着你会写 spec、管 agent、做判断，知道什么该交给机器，什么必须人来。语法知识越来越不稀缺，判断力越来越稀缺。

Karpathy 最后那句话，我记到现在：Software 3.0 不是 slogan，它是程序员从代码作者，变成计算劳动力操作者的那一刻。

说真的，我也不知道是该兴奋还是该焦虑。一方面我亲手用 Claude Code 几天搞出来以前要请三个工程师做一个月的东西，那种爽感是真实的。另一方面，我也清楚这种爽感的底层是什么——是我积累的判断力在给 agent 把关。那些没积累过这些判断力的人怎么办，我也没答案。

但我至少知道一件事：光会用 Claude Code，跟会工程化地使用 agent，是两件完全不同的事儿。后者比前者难得多，但也值钱得多。

Karpathy 这次还是说对了，氛围编程已经结束了，接下来真正的游戏，才刚刚开始。

好，这期就到这里，觉得这期有用，记得点个关注收藏一下，我是阿里长长，下一期喜欢听什么，欢迎评论区告诉我。

### 2.3 核心概念精讲

#### 概念一：Vibe Coding（氛围编程）

**是什么**：靠描述需求、凭感觉和运气让 AI 生成代码，不深究实现细节，看到功能跑起来就算完成。

**为什么重要**：这是过去一年 AI 编程工具爆火的底层用法，也是 Karpathy 认为必须被淘汰的阶段。

**详细解释**：
- 典型场景：打开 Cursor，说"帮我做个 dashboard"，回车，界面出来了，不管里面多了个 Banana.ts
- 优点：速度极快，原型产出效率高，门槛极低
- 致命缺陷：只配做原型，无法支撑真实工程系统；开发者对系统内部发生了什么几乎没有掌控

**实际应用场景**：适合个人 demo、快速验证想法，但绝不适合生产环境或需要长期维护的系统。

---

#### 概念二：Agentic Engineering（Agent 工程）

**是什么**：把 AI agent 作为受约束的工程组件来使用，配套完整的工程纪律，而不是当作魔法棒挥一挥。

**为什么重要**：这是 Karpathy 指向的下一个阶段，也是市场（GitHub 认证、企业招聘方向）正在押注的方向。

**详细解释**：

| 要素 | 说明 |
|------|------|
| Memory | agent 需要有记忆机制，知道上下文 |
| Tools | 明确 agent 能用哪些工具 |
| 权限控制 | 每个 agent 只有完成任务所需的最小权限 |
| 多 agent 协作 | 不同专精 agent 各司其职，而非一个万能 agent |
| Sandbox | 隔离执行环境，防止意外破坏 |
| 监控 & CI | agent 的行为需要被持续观测和约束 |
| Code Review & Spec | 人仍然是最终的审核者和决策者 |
| 安全审计 | agent 有 root 权限时，安全边界必须明确 |

**实际应用场景**：构建需要长期运行、多人协作、对稳定性有要求的系统时，必须用 agentic engineering 的思路。

---

#### 概念三：Software 3.0

**是什么**：Karpathy 提出的软件发展阶段框架，描述 LLM 成为计算基础设施、自然语言成为编程接口的新范式。

**为什么重要**：它解释了为什么"程序员"这个身份的含义正在根本性地改变。

**详细解释**：

| 阶段 | 核心方式 | 人的角色 |
|------|----------|----------|
| Software 1.0 | 人手写显式代码 | 代码作者 |
| Software 2.0 | 神经网络从数据中学习 | 数据和模型的设计者 |
| Software 3.0 | LLM 是计算基础，自然语言是接口，agent 是 runtime | 计算劳动力的操作者/调度者 |

**实际应用场景**：理解这个框架，能帮助开发者判断自己现在真正在做的工作是什么——写给人看的代码，还是写给模型看的 spec 和提示词。

---

#### 概念四：无约束 Agent 的风险

**是什么**：给 agent 开放过多权限（改文件、装包、读取密钥、推送分支、部署）而缺乏约束，带来的系统性风险。

**为什么重要**：Karpathy 用"人质危机"来形容，一次犯错就可能造成不可逆的损害。

**详细解释**：
- Karpathy 原话：一个没有约束的 coding agent，不是同事，是一个拥有 root 权限、精力无限、偶尔会犯浑的初级工程师
- 风险不在于 agent 能力弱，恰恰在于它能力强但判断边界不清
- 解决方向：专精 agent + 最小权限原则 + 人负责调度和最终判断

**实际应用场景**：在给 agent 配置工具和权限时，必须像设计系统角色权限一样谨慎，而不是"给它最大权限省事"。

### 2.4 速查表

| 维度 | Vibe Coding | Agentic Engineering |
|------|-------------|---------------------|
| **核心逻辑** | 靠感觉、靠运气 | 工程纪律 |
| **适用场景** | 原型、demo、验证想法 | 生产系统、长期维护项目 |
| **对开发者的要求** | 会描述需求 | 会写 spec、管 agent、做判断 |
| **对 agent 的态度** | 魔术师，变出来就行 | 受约束的工程组件 |
| **风险** | 系统一团糟但自己不知道 | 管理不当则一次犯浑全盘皆输 |
| **稀缺能力** | 语法和代码技巧 | 判断力和系统调度能力 |
| **市场信号** | 圈子狂欢期（已过） | GitHub 认证、企业转型方向 |

### 2.5 延伸思考

1. **判断力从哪里来？** 视频里提到，vibe coding 的爽感底层是"积累的判断力在给 agent 把关"。那些没有经历过传统工程训练的新人，在 agent 时代如何建立这种判断力？还是说这种判断力本身的获取路径也需要被重新设计？

2. **"签字而非编程"是危机还是进化？** 视频里那位骨灰级程序员觉得自己沦为"签字机器"，但换个角度看，架构师、技术 lead 一直在做的不也是审核和决策而非亲手写代码？这是降级，还是整个行业在向更高抽象层移动？

3. **最小权限原则能真正约束 agent 吗？** Karpathy 的建议是专精 agent + 权限隔离，但现实中复杂任务往往需要跨越权限边界。如何在"给 agent 足够能力完成任务"和"防止它犯浑造成不可逆损害"之间找到可操作的平衡点？
