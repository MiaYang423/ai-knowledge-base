## 一、视频原文

如果你经历过去年内卷的DeepSeek热潮，也被今年铺天盖地的Agent洗脑过，那你一定会和我有同样的疑惑，就是怎么才短短一年的时间，AI就从一个全能聊天工具变成了能自动干活的数字员工？我这才刚刚弄明白大模型适合啥，怎么又冒出来一个智能体？

没关系，今天这期视频我就跟大家好好聊聊AI到底是怎么干活的。

智能体，也就是常说的Agent，它和豆包这种AI聊天工具最大的区别就是它能够自己拆解各种任务并自动执行，直到取得具体的结果。我在之前的视频里面有仔细讲过，感兴趣的可以去看一看。

想让Agent顺畅工作，其核心还是要依赖大模型。而目前市面上主流的Agent工具都会自带大模型，比如公认世界最强的Claude Code代理，就是自家的Claude Sonnet模型。虽然它的模型也是世界顶尖水平，但价格也同样顶尖。像我这种用不起他们家模型，但是又眼馋Claude Code这个Agent工具的，怎么办呢？最一劳永逸的方法就是选择国产模型替代。

那怎么样才能用上国产模型呢？这里又涉及到一个名叫API的技术。API全称应用程序接口，你可以把它理解为一条跨平台的模型直连通道，而连接这个通道的东西是一串名叫API Key的代码。只要你购买了国产模型的付费计划，拿到专属的API Key，把它填到特定的位置，就能够让Agent调用国内模型了。

但是不同的工具配置方法是不一样的。有的时候会提供一个设置面板，你填进去就完事了；而像Claude Code这种就需要你去修改它的设置文件。一般情况下每一个国产模型的付费计划后台都会提供对应的文件修改方式，但这对新手小白来说还是有点困难，像我当初准备这个都花了小半天。不过现在GitHub上面有一个名叫CCS之类的项目，能够非常方便地完成API Key的配置和各种模型的接入。

那搞定了Agent和模型的匹配，是不是又能愉快地起飞了呢？也不是。因为你会发现几个非常现实的问题，比如你想让它联网搜索东西，它会告诉你没这个能力；截个图给它看，它能回你一堆驴唇不对马嘴的内容；一个链接给它分析，它也根本分析不了。

如果想解决这个问题，就必须要聊到我们今天的第三个核心概念了——MCP，全称是Model Context Protocol，翻译过来就是模型上下文协议。大家千万别被"协议"这两个字吓到了，简单来说它就像AI界共同打造的一个万能接口。

如果不懂这个协议，你的Agent其实是个偏科生。比如你让它去扒一个网页，网页底层用的是复杂的HTTP代码，而Agent脑子里只能处理纯文本。为了让它们两个交流，以前必须针对这一个网页苦哈哈地手写一套定制化的数据转换代码，换一个网站还得重写一套。而MCP的出现直接颠覆了这种落后的模式。

当你把MCP接入了你的Agent之后，后台会发生这三个积极智能的步骤：

第一步，MCP工具会主动给你的Agent递上一份技能清单，比如清单上会写明"我们现在掌握了三个技能：提取网页正文、调用联网搜索、读取数据库"；

第二步，当你在聊天框下达了"帮我搜索一下最新新闻"的指令后，Agent会自动去查内部清单，发现技能二正好对口，于是它会按照协议规定的统一格式把你的需求打成一个标准包，发送给MCP工具；

第三步，MCP工具拿到需求后，自己转身去外部网络里执行抓取动作，完成之后会把乱七八糟的网页代码全部翻译成大模型能看懂的结构化纯文本，直接塞进它的上下文里。

那MCP工具去哪里获取呢？GitHub上面一搜一大把。

好，现在咱们搞定了大模型的接入，又接入了MCP的外部工具，咱们Agent的底层框架就算彻底搭建完毕了。但框架再好，如果它每次干活就得像带新人一样把复杂的业务逻辑从头到尾事无巨细地交代一遍，不仅你累得够呛，跑出来的token账单也能让你肉疼。

那怎么才能让它把这些复杂的流程记下来，变成一个高效稳定的自动功能？这就涉及到了我们今天的最后一个核心概念——SOP。简单来说，SOP就是针对高频复杂任务的标准作业流程。它的核心作用就是把你日常重复的、专业要求极高的工作流程封装成一个固定的模块。等你下次再需要处理同类任务时，Agent就可以直接调用这个现成的SOP去自动执行，再也不用你每次苦哈哈地输入几千字的提示词了。这不仅把自动化执行的效率拿到了极致，还能把你大模型的token消耗成本狠狠地降下来。

### 2.3 核心概念精讲

#### 概念一：Agent（智能体）

**是什么**：Agent是能够自主拆解任务、自动执行步骤、直到得出具体结果的AI工具，不同于豆包这类被动回答问题的聊天AI。

**为什么重要**：
聊天AI需要你一步步指挥，Agent可以接到一个目标后自己想办法完成，相当于从"工具"升级为"数字员工"。

**详细解释**：

| 对比维度 | 聊天AI（如豆包） | Agent（智能体） |
|---|---|---|
| 工作方式 | 被动回答，一问一答 | 主动拆解任务，自动执行 |
| 输出形式 | 文字回复 | 具体结果（文件、操作、数据等） |
| 依赖 | 大模型本身 | 大模型 + API + MCP + SOP |

**实际应用场景**：
让Agent帮你搜集最新行业新闻、整理成报告并保存到指定位置，全程无需人工干预。

---

#### 概念二：API 与 API Key（模型接入）

**是什么**：API（应用程序接口）是跨平台连接大模型的直连通道；API Key是购买付费计划后获得的专属密钥，填入工具后即可完成接入。

**为什么重要**：
主流Agent工具（如Claude Code）自带的模型价格昂贵，通过API接入国产模型是降低使用成本的核心手段。

**详细解释**：

- 部分工具提供**图形设置面板**，直接粘贴API Key即可
- 部分工具（如Claude Code）需要**手动修改配置文件**，对新手有一定门槛
- GitHub上有辅助配置项目（视频中提到"CCS之类的项目"），可简化这一过程

**实际应用场景**：
买了国内某大模型的付费套餐，拿到API Key后，将Claude Code的底层模型替换为国产模型，大幅降低使用成本。

---

#### 概念三：MCP（模型上下文协议）

**是什么**：MCP（Model Context Protocol）是AI界统一的万能接口协议，让Agent能够调用联网搜索、读取网页、访问数据库等外部能力，而不需要为每个工具单独写转换代码。

**为什么重要**：
没有MCP，Agent只能处理纯文本，无法联网、无法读图、无法分析链接——相当于一个"偏科生"。MCP让Agent具备了真正实用的外部感知能力。

**详细解释——MCP工作三步骤**：

| 步骤 | 发生了什么 |
|---|---|
| Step 1：技能注册 | MCP工具向Agent递交技能清单（如：联网搜索、提取网页正文、读取数据库） |
| Step 2：需求匹配 | Agent收到用户指令后，查清单找到对口技能，按统一格式打包请求发给MCP |
| Step 3：执行返回 | MCP去外部执行操作，把结果翻译成大模型能读懂的结构化文本，塞入上下文 |

**实际应用场景**：
用户说"帮我搜一下今天的AI新闻"，Agent通过MCP的联网搜索技能自动抓取网页内容，返回可读的文本摘要。

---

#### 概念四：SOP（标准作业流程）

**是什么**：SOP是把高频、复杂的工作流程提前封装成固定模块，Agent下次遇到同类任务时直接调用，无需用户重新输入长篇提示词。

**为什么重要**：
- 省去每次写几千字提示词的重复劳动
- 大幅降低token消耗，节省使用成本
- 保证每次执行质量稳定、流程标准化

**详细解释**：

```
没有SOP的情况：
  用户每次 → 输入完整复杂提示词 → Agent执行 → 消耗大量token

有SOP的情况：
  提前封装流程模块
  用户每次 → 一句话调用SOP → Agent自动执行 → token消耗大幅降低
```

**实际应用场景**：
将"每周竞品分析报告"的完整流程封装成SOP，之后只需对Agent说"跑一下竞品分析"，它就能自动完成所有步骤。

### 2.4 速查表

| 概念 | 全称 | 解决什么问题 | 类比理解 |
|---|---|---|---|
| **Agent** | 智能体 | AI从聊天工具升级为自动执行工具 | 数字员工 |
| **API / API Key** | 应用程序接口 / 密钥 | 让Agent接入你选择的大模型（含国产） | 模型直连通道 + 门禁卡 |
| **MCP** | Model Context Protocol | 让Agent具备联网、读图、读数据库等外部能力 | AI界的万能插座 |
| **SOP** | 标准作业流程 | 把复杂重复流程封装成模块，省时省token | 工作手册/自动化脚本 |

### 2.5 延伸思考

1. **MCP的"统一协议"价值在哪里？** 在MCP出现之前，每接入一个外部工具就要手写一套转换代码。MCP统一了接口标准之后，谁来维护这个生态、谁来保证MCP工具的质量和安全性？这个协议的治理机制是什么？

2. **SOP封装的边界在哪里？** SOP适合高频、标准化的任务，但现实工作中很多任务是非标的、需要随机应变的。什么样的工作流值得封装成SOP，什么样的场景反而不适合？如何判断？

3. **国产模型替代的代价是什么？** 视频提到用API Key接入国产模型来降低成本，但不同模型的能力存在差异。在Agent场景下，模型的推理能力、上下文长度、工具调用稳定性会直接影响任务完成质量——如何在成本和效果之间找到平衡点？
