> 适合人群：有一定 AI 基础、正在准备大模型/智能体方向面试或从事相关开发的工程师

## 一、视频原文

面试官问你智能体意图识别的方法，你要是直接说写个 prompt 调用大模型就行，那这场面试基本就凉了。

大家好，我是 Jack。遇到意图识别，很多人第一反应就是把用户的话直接丢给大模型，让它给个分类标签。比如用户说"帮我查邮件"，模型返回"查询邮件"的标签，简单粗暴。但这在面试官眼里就是灾难。

为啥呢？用户说话很随意。像老板说"帮我看看上个月谁没交周报，顺便发邮件催一下"，这是查数据和发邮件两个意图，要是系统只能输出一个分类，漏掉哪个都得挨骂。更可怕的是，用户说"把客户资料全删了"，单靠语义模型判断，万一它出错真删了，就出大事了。

那到底怎么解决呢？面试官想听的大招，不是一把模型乱顿，而是一套可落地的意图分类体系。

首先你得明白，真实的智能体产品，意图一定是个有限集合。比如你做一个办公助手，意图无非就是查邮件、发邮件、查日程、删文件这么十几二十类。定义清楚之后，意图识别就变成了一个分类任务。

怎么让分类又准又快、成本还扛得住？工业界的标准打法是：规则加模型并行，安全后置。

别一上来就全甩给大模型。像查邮件、发周报这种高频、说法固定的意图，直接用规则做关键词匹配，速度极快，成本几乎为零。只有规则兜不住的模糊说法，才交给语义模型去理解。语义模型可以是一个微调过的小参数模型，也可以是大模型 few-shot 的分类。但调用之前先让规则过一遍，能挡掉大半流量，成本一下子就控制住了。

这两路并行跑，结果汇总产生最终意图。如果结果冲突，再用一个简单的仲裁逻辑：比如语义模型置信度高就听语义模型的；规则命中但语义模型置信度低，就走人工兜底或反问澄清。这个思路既保证了准确率，又没有盲目消耗算力。

那上下文里出现"不对不对"这种自我纠错怎么办？这就是多轮对话里意图识别的必备能力了。实际处理时，会把前一轮或前几轮发言作为历史信息一起传给意图分类模型，模型自然能学到用户刚刚改口了，现在意图是查风险条款，不再是找合同。这比专门训练一个上下文专家轻量得多，也更符合端到端落地的习惯。

至于安全方面，比如"删客户资料"这种高危操作，它确实不属于意图识别这个环节。工程上安全的做法是：意图识别只管输出"删除资料"这个标签，后续会有一个独立的 Guardrail 安全策略层，在动作真正执行前对指令做全量校验、二次确认和风险拦截。这样职责清晰，不会把安全逻辑硬塞进意图分类模型，导致模型又慢又难调。

咱们还用之前那个例子过一遍。用户说"帮我把昨天的合同找出来，不对不对，还是先查查有没有风险条款吧"。规则引擎瞬间抓到"合同""风险条款"，初步命中"合同风险分析"这个意图。同时语义小模型读取上文，理解到用户已经改口，也给出了"合同风险分析"的结果，而且置信度很高。两路结果一致，意图直接确定为"合同风险分析"。整个识别过程只调用了一次轻量模型，规则又凌成本，既快又省。

识别完成后，后面才是工具调度：查文件、调法务分析接口。而安全护栏在工具执行前会最后审一遍："合同风险分析"是读操作，不涉及删除，放行。

你看，这样分阶段设计，每一步都清晰可控。

回头看这道题，智能体做意图识别，核心就是：收敛到有限意图的分类；落地时用规则加小模型并行提效降成本；把安全校验后置到策略层，而不是一开始就塞进意图识别。

最后留个思考题：并行时规则和模型给出的意图标签不一致，你觉得仲裁逻辑应该怎么设计？欢迎在评论区聊聊。今天视频就到这，咱们下期见，拜拜。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**智能体意图识别不能只靠大模型，工业级方案是"规则+小模型并行、安全后置"的分层架构，这也是面试中区分普通候选人和有工程经验者的关键答题思路。**

### 2.3 核心概念精讲

#### 概念一：意图有限集合

**是什么**：在真实智能体产品中，用户的意图不是无限开放的，而是可以被预先定义为一个有限的分类集合（如办公助手场景下的十几到二十类意图）。

**为什么重要**：
把"理解用户想做什么"这个模糊问题，转化为"从有限选项中分类"这个工程问题，才能让后续的规则匹配和模型训练有的放矢，也让系统行为可预期、可测试。

**详细解释**：

| 做法 | 问题 |
|------|------|
| 意图开放、不预定义 | 模型输出不可控，无法穷举测试 |
| 预定义有限意图集合 | 变成分类任务，可量化准确率，可针对性优化 |

**实际应用场景**：办公助手预定义意图集合示例——查邮件、发邮件、查日程、删文件、查周报、发周报、合同风险分析等。

---

#### 概念二：规则 + 模型并行架构

**是什么**：意图识别不走单一路径，而是同时运行规则引擎和语义模型两条通道，最后由仲裁层融合结果。

**为什么重要**：
单纯用大模型，延迟高、成本高、不可控；单纯用规则，无法处理模糊表达。并行架构取两者之长，用规则拦截大部分流量，只把难以判断的请求交给模型，大幅降低成本和延迟。

**详细解释**：

| 维度 | 规则引擎 | 语义模型 |
|------|----------|----------|
| 适用场景 | 高频、表达固定的意图 | 模糊、多样化表达 |
| 实现方式 | 关键词匹配 | 微调小模型 / 大模型 few-shot |
| 速度 | 极快（毫秒级） | 相对慢 |
| 成本 | 几乎为零 | 有推理成本 |
| 处理顺序 | 先跑，能拦截大半流量 | 规则兜不住时才调用 |

**仲裁逻辑**：

- 两路结果一致 → 直接确定意图
- 语义模型置信度高 → 采用语义模型结果
- 规则命中但语义置信度低 → 人工兜底或向用户反问澄清

**实际应用场景**：用户说"帮我把昨天合同找出来，不对，先查风险条款"——规则抓到关键词，语义模型理解改口，两路均指向"合同风险分析"，一次轻量模型调用完成识别。

---

#### 概念三：多轮对话中的意图识别

**是什么**：在多轮对话场景下，将前几轮的历史对话一并传入意图分类模型，使模型能感知用户的自我纠错和意图变化。

**为什么重要**：
用户在对话中频繁改口（如"不对不对，还是……"），如果只看当前一句话，意图识别会出错。引入历史上下文是最轻量的解决方式。

**详细解释**：

- **输入**：当前用户输入 + 前 N 轮对话历史
- **效果**：模型自动学到"用户已改口，新意图优先"
- **对比方案**：专门训练上下文理解模型（成本高、周期长）vs. 直接拼接历史传入（轻量、易落地）

**实际应用场景**：用户先说"帮我找合同"，再说"不对，查风险条款"——历史上下文传入后，模型输出"合同风险分析"而非"查找合同"。

---

#### 概念四：安全后置（独立 Guardrail 层）

**是什么**：意图识别模块只负责输出意图标签，不承担安全判断职责；安全校验由独立的 Guardrail 策略层在工具/动作真正执行前完成。

**为什么重要**：
把安全逻辑塞进意图识别模型，会让模型职责混乱、难以调试，且安全边界不清晰。分层设计使每个模块职责单一，安全可独立迭代和加强。

**详细解释**：

```
用户输入
    ↓
意图识别层（只输出标签，如"删除资料"）
    ↓
工具调度层
    ↓
Guardrail 安全策略层（执行前校验）
    ├── 读操作 → 放行
    └── 高危操作（删除/修改） → 二次确认 / 拦截
    ↓
工具/动作执行
```

**实际应用场景**：用户说"把客户资料全删了"——意图识别正常输出"删除资料"标签，Guardrail 层识别为高危写操作，触发二次确认流程，防止误删。

### 2.4 速查表

| 对比维度 | 新手做法 | 工业级做法 |
|----------|----------|------------|
| 意图范围 | 开放式，不预定义 | 有限集合，预先定义 |
| 识别方式 | 直接调用大模型分类 | 规则引擎 + 语义模型并行 |
| 多意图处理 | 只输出一个标签 | 支持同时识别多个意图 |
| 多轮对话 | 只看当前输入 | 携带历史上下文传入模型 |
| 安全处理 | 塞进意图识别模型 | 独立 Guardrail 层后置校验 |
| 成本控制 | 每次都调大模型 | 规则先过滤，模型只处理模糊请求 |
| 冲突处理 | 无仲裁机制 | 置信度 + 规则命中情况联合仲裁 |

### 2.5 延伸思考

1. **仲裁逻辑如何设计才能既自动化又留有人工干预空间？**
   视频提到"置信度低时走人工兜底"，但在高并发场景下人工兜底不现实。如何设定置信度阈值、设计自动降级策略，同时保留低频边界案例的人工复核通道，是实际落地中的核心工程问题。

2. **意图有限集合如何随业务演化动态扩展，而不破坏现有规则和模型？**
   视频强调意图是有限集合，但真实产品会不断新增功能。新增意图时，如何最小化对规则引擎和已训练模型的改动，同时保证新旧意图之间不互相干扰？

3. **Guardrail 安全层的校验规则应该由谁来定义和维护？**
   视频将安全后置为独立策略层，但"哪些操作算高危""二次确认的触发条件是什么"本身就是复杂的业务判断。这套规则是硬编码、规则引擎还是再引入一个模型来判断？不同选择的边界和风险在哪里？

---

### 2.6 一句话记住它

> 意图识别的面试答案不是"调大模型"，而是：有限意图做分类、规则加小模型并行降成本、安全校验独立后置不越界。
