## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**LangChain 是单向流水线，适合固定流程的 RAG 任务；LangGraph 是循环状态机，适合需要纠错、人工审批和多智能体协作的复杂 Agent——搞清楚这个区别，能让你在架构选型时少踩大坑。**

### 2.3 核心概念精讲

#### 概念一：LangChain 的 DAG 流水线模式

**是什么**：LangChain 默认采用有向无环图（DAG）结构，数据从输入端单向流向输出端，每一步执行完即释放，不保留中间状态。

**为什么重要**：这决定了它天然适合流程固定的任务，但也决定了它无法原生支持"发现错误后回退重试"的逻辑。

**详细解释**：
以标准 RAG 流程为例，执行顺序是：
1. 接收用户输入
2. 去向量库检索相关资料
3. 将资料喂给大模型生成回答
4. 解析并输出结果

整个过程单向推进，如同流水线上的零件，经过一个工位就移向下一个，不会回头。记忆和上下文需要开发者手动在每一步之间传递。

**实际应用场景**：企业知识库问答（RAG）、文档翻译、固定模板的文本生成任务。

---

#### 概念二：LangGraph 的全局持久化状态与 Checkpoint 机制

**是什么**：LangGraph 引入了图计算结构和一个贯穿整个任务生命周期的全局状态对象，并通过 Checkpoint（存档点）机制实现任务的随时暂停与恢复。

**为什么重要**：这是 LangGraph 能支撑工业级 Agent 的核心原因——任务不怕中断，出错可以回退，人类可以随时介入。

**详细解释**：

| 能力 | 说明 |
|---|---|
| 全局持久化状态 | 整个任务共享一个状态对象，所有节点都能读写，不需要手动传递上下文 |
| Checkpoint（存档点） | 任务执行到任意节点都可以保存快照，断电或中断后从快照恢复，不必从头跑 |
| 时间旅行 | 可以手动将任务回退到任意历史节点重新执行，用于调试或纠错 |
| 循环与回退 | 当工具调用失败或结果不满足条件时，Agent 可以自动转回上一步重新规划 |
| 人工审批断点 | 在关键节点设置红色断点，AI 计算完成后暂停，等待人类确认后再继续执行 |

**实际应用场景**：跨部门复杂审批流、代码自动修复（写代码→运行→报错→重新规划→修复）、金融/法律领域需要人工复核的 AI 决策流程。

---

#### 概念三：人机协同（Human-in-the-Loop）

**是什么**：在 AI 完成计算后，在特定节点强制暂停，等待人类审核确认，再由 AI 继续执行后续步骤。

**为什么重要**：视频指出这是大模型进入核心生产线的"最后一关"——在金融、法律等高风险领域，纯自动化 AI 不被信任，必须有人类把控核心决策节点。

**详细解释**：LangGraph 通过在图中设置"审批断点"实现这一能力。AI 负责繁重的数据处理和推理，人类只需在关键决策点点击确认，兼顾了效率与可控性。

**实际应用场景**：合同审核 Agent（AI 起草→人类审批→AI 发送）、金融风控（AI 识别风险→人类确认→AI 执行处置）。

---

#### 概念四：多智能体协作网络

**是什么**：多个专职 Agent 分工协作，共同完成一个复杂任务，例如编程 Agent 负责写代码、测试 Agent 负责跑测试。

**为什么重要**：视频指出"多智能体协作已成标配"，而这种协作网络的调度和状态管理，需要 LangGraph 的图计算框架才能支撑，LangChain 的单向流水线无法胜任。

**详细解释**：在 LangGraph 中，每个 Agent 是图中的一个节点，节点之间通过边（条件路由）连接，全局状态在所有节点间共享，使得多个 Agent 可以协作、等待、并行或串行执行。

**实际应用场景**：自动化软件开发流水线（需求分析 Agent → 编码 Agent → 测试 Agent → 部署 Agent）、复杂客服工单处理系统。

---

### 2.4 速查表

| 对比维度 | LangChain | LangGraph |
|---|---|---|
| 底层结构 | 有向无环图（DAG） | 图计算 + 全局状态机 |
| 数据流向 | 单向，不回头 | 支持循环、回退、分支 |
| 状态管理 | 逐步消毁，手动传递 | 全局持久化，自动共享 |
| 断点恢复 | 不支持 | 支持（Checkpoint） |
| 时间旅行 | 不支持 | 支持 |
| 人工审批介入 | 不支持 | 支持（审批断点） |
| 多智能体协作 | 难以支撑 | 原生支持 |
| 上手复杂度 | 低，代码量少 | 较高，需定义节点和边 |
| 执行速度 | 快（无循环损耗） | 相对较慢（有状态维护开销） |
| **典型场景** | RAG 问答、文本翻译 | 代码自修复、跨部门审批、复杂中台 |
| **选型口诀** | 线性 + 固定 + 一次性 → 选它 | 纠错 + 循环 + 人工干预 → 选它 |

### 2.5 延伸思考

1. **LangChain 和 LangGraph 是互斥关系吗？** 实际工程中，能否将两者结合使用——用 LangChain 处理固定的子流程模块，再用 LangGraph 在更高层做流程编排和状态管理？这种混合架构有哪些潜在的优势和风险？

2. **Checkpoint 机制的代价是什么？** 全局持久化状态意味着需要额外的存储和序列化开销，在高并发、低延迟的生产环境中，这种开销是否会成为瓶颈？应该如何设计存储策略来平衡可靠性与性能？

3. **"人工审批断点"会不会成为效率瓶颈？** 视频认为人机协同是工业级落地的关键，但在实际业务中，如果审批节点依赖人类响应，整个 Agent 流程就会被人类的响应速度拖慢。如何设计审批粒度和超时策略，才能在安全性和效率之间取得平衡？

---

### 2.6 一句话记住它

> 流程固定就用 LangChain 跑流水线，任务复杂需要兜底和人工拍板就上 LangGraph——技术选型没有最好的，只有最合适的。
