所以接下来就可以看到 Claude Code 在这里会先制定一个方案，这里方案已经制作出来了，它知道我需要它来给我做一个用于 Survey 也就是问卷调研的小程序。然后我同意了以后，左边我们可以看到 Terminal 的界面当中 Claude Code 在进行代码的输写，然后右边其实是我们可以实时观测到 Claude Code 修改或者新建代码以后模拟器的一个情况。

所以还是非常高效的，可以看到右边这个界面颜色已经发生了改变，从最开始微信自带的基本模板，现在已经有了一个雏形，就是我需要的几个模块：一个是参与调研的注册，然后继续填写新的问卷。

然后这里其实是我给了 Claude Code 一些新的需求再进行修改，因为第一版的时候有一些图标的地方显示了代码的形式。所以这里其实是一个人机互动的过程，修改了大概两三轮以后，现在在模拟器当中运行，错误慢慢地减少了。

接下来我把它进行了真正的真机模拟，这部分是在我的手机上进行的录频，刚才前面更大一些的画面是在电脑上。运行到这一步基本没有什么问题了，完成。

### 2.3 核心概念精讲

#### 概念一：Claude Code

**是什么**：Anthropic 出品的 AI 编程助手，可以在 Terminal（命令行）中直接读写本地文件、生成和修改代码。

**为什么重要**：不只是"给你看代码片段"，而是直接操作文件系统，相当于一个会写代码的协作者在你的项目文件夹里干活。

**详细解释**：
- 需要提前指定工作目录（本视频中是一个新建的空文件夹）
- 需要授予文件修改权限才能自动写入
- 工作过程在 Terminal 界面可见，透明可追踪

**实际应用场景**：从零新建项目、批量修改已有代码文件、配合其他开发工具（如微信开发者工具）联动使用。

---

#### 概念二：先出方案再执行的工作模式

**是什么**：Claude Code 在动手写代码之前，会先输出一份执行方案，等用户确认后再开始。

**为什么重要**：给用户一个"检查点"，避免 AI 跑偏方向后大量返工，降低试错成本。

**详细解释**：
- 用户输入 prompt → Claude Code 理解需求并列出方案
- 用户审阅方案，觉得没问题后"同意"
- Claude Code 开始在 Terminal 中逐步写代码

**实际应用场景**：需求复杂或文件较多时，这个确认步骤尤其重要，可以在方案阶段就纠正理解偏差。

---

#### 概念三：人机多轮迭代

**是什么**：AI 生成第一版后，用户发现问题，追加新需求，AI 再修改，如此循环直到满意。

**为什么重要**：一次 prompt 很难把所有细节说清楚，多轮对话是实际落地的常态，不是失败。

**详细解释**：

| 轮次 | 发生了什么 |
|------|------------|
| 第1轮 | Claude Code 生成基础页面，出现图标显示为代码字符的问题 |
| 第2-3轮 | 用户补充需求，Claude Code 修复问题，错误逐渐减少 |
| 最终 | 模拟器稳定运行，进入真机测试 |

**实际应用场景**：任何 AI 辅助开发项目，都建议预留"迭代时间"，不要期望一次生成完美结果。

---

#### 概念四：双屏联动开发体验

**是什么**：左侧 Terminal 看 Claude Code 写代码的过程，右侧微信开发者工具模拟器实时反映代码变化效果。

**为什么重要**：所见即所得，不需要手动刷新或切换，大幅降低调试的心理负担，也方便及时发现视觉问题。

**详细解释**：
- Terminal（左）：展示 Claude Code 的操作日志和代码输出
- 微信开发者工具模拟器（右）：代码文件一变动，界面立刻更新
- 最后一步：真机预览，在手机上验证实际运行效果

**实际应用场景**：适合边看边调的开发习惯，尤其对非专业开发者友好，可以直观判断效果对不对。

### 2.5 延伸思考

1. **prompt 质量决定上限**：视频中一开始的 prompt 内容没有展示细节，如果需求描述不清晰，AI 方案会偏差多少？怎么写一个对 Claude Code 友好的小程序需求 prompt？

2. **多轮迭代的边界在哪里**：本次 2-3 轮就基本完成，但如果功能更复杂（比如加入数据库同步、多用户权限管理），迭代成本会怎么变化？Claude Code 在哪类问题上容易卡住？

3. **AI 生成的代码可维护性如何**：小程序跑起来了，但后续如果需要改需求、加功能，AI 生成的代码结构是否便于人工介入？团队协作时应该怎么管理这类"AI 写的项目"？
