## 一、视频原文

大家好，我是奔走。在各种 AI 软件神仙打架的时代，Claude 好像在慢慢被大家遗忘了。很多人对它的印象还停留在几个老问题上：记忆烂、上下文短、文件管理像个垃圾堆。事实是大家的认知已经配不上它的进化速度了。

上期视频我们讲了 Claude 的一些使用技巧，今天我用三条干货手把手带你打造自我进化的 Claude。来，走着学着进化。

**第一步：学会用 Standing Orders 长驻指令**

当你发现自己每天都在对 Claude 说同一类话，这时候不要再写一次性 Prompt，你要把它升级成 Standing Orders。

举一个简单的例子，比如你想让 Claude 帮你每天收各个平台里跟 AI 有关的信息，你只需要做三件事：

第一步，给目标。告诉 Claude：我需要你帮我写一个 Standing Orders，用 OpenCLI 工具收多个平台有关 AI 的信息。

第二步，给资料。去 GitHub 找到 OpenCLI 的用法，把链接发给它。Claude 会自己抓信息、自己消化，并提取关键命令和最佳实践，然后写进 Standing Order。

第三步，看成品。当你让 Claude 写完一条高质量的 Standing Order 后，遇到新的指令可以让它直接参考这个模式去写。

我后面让它帮我写的"文档去 AI 化指令 Humanizer Standing Order"就是这样：先抓了 Copy.ai 的完整 Humanizer 文档，全部内化成一个美观的自动过检查清单。这条 Order 写完以后，我就不需要再提"说话别太 AI 了"，因为 Claude 已经被训练好了。

**第二步：灵活使用 Heartbeat 和 Cron 自动干活机制**

两者的区别在于：

Heartbeat 是隔一阵子看一眼，有事说事，没事继续睡，最短 30 分钟一次，能把好几件事叠到一起跑。

Cron 是闹钟响了，到这个点必须干活，精确到秒，一条任务一个坑。

如果一个任务要做到分钟级的响应就用 Cron，如果要和其他事情一起跑任务就用 Heartbeat。

你不需要自己去研究 Heartbeat 命令怎么写，不需要去背 Cron 表达式的语法，你只需要对 Claude 说一句话："帮我监控我的自媒体账号数据，每天自动查一次播放量和点赞数。"然后 Claude 自己会干这些事儿：去读 OpenCLI 的文档，写 heartbeat.md，订好每条命令的间隔和步骤，剩下来它自己就能跑通了。

**第三步：外接专业软件，搭建低摩擦操作系统**

Claude 自带了四种接入方式，包括 MCP、CLI、Harness 和内置工具。

第一种，MCP：可以把它理解成 AI 工具的标准接口，只要某个软件有 MCP 服务，Claude 就有机会对接它。像 Notion、GitHub、Airtable、Figma、Canva、Slack、GitLab，这些有明确 MCP 接口的软件优先用 MCP 对接。它适合查资料、跑项目、改任务、写记录，或者上下文查数据库这类事情。

但很多平台没有这么开放的接口，比如各类自媒体平台、实时数据源。这些地方我们要的不是调用功能，而是获取平台内容：评论、字幕、热榜、创作者数据。这时候就用 CLI 工具，比如我常用的 OpenCLI，它能把原本只能手动操作的网站变成 Claude 可以调用的命令。比如做选题调研时，它能搜英文趋势、找同类视频、看高赞回答和用户痛点，再把结果统一汇总回来。

第三种，Harness：适合对接专业 Agent。比如代码任务可以交给 Codex，Claude 负责接任务、拆任务、判断结果，Codex 负责进仓库、改代码、跑测试，最后结果再回到 Claude。

第四种，内置工具：Claude 本身就有浏览器、文件读写、终端命令、搜索等功能，适合硬实任务、自 Agent 系统等场景。

什么时候开始用？很简单，只要任务不是"回答我一个问题"，而是要真的操作网页、文件、代码、表格、文档、项目系统，就该用外接工具。

最后几个例子让你一看就会。比如 Claude 帮我做这期视频：低级玩法是让模型自己乱搞，高级玩法是——Claude 用 OpenCLI 抓平台数据，用浏览器读官方文档，用 OpenCLI 存资料，用 Airtable 管理选题状态，用 Canva 做封面，需要代码 Demo 就把任务交给 Codex，用 HyperFrames 生成视频动效。剩下的就是你的任务：把所有结果按脚本逻辑剪辑成片进视频，最后就成了今天你看到的这期视频内容了。

### 2.3 核心概念精讲

#### 概念一：Standing Orders（长驻指令）

**是什么**：把你反复对 Claude 说的同类要求，提前写成一条结构化的持久指令，让 Claude 每次都自动遵守，不需要重复交代。

**为什么重要**：每次重新写 Prompt 既低效又容易遗漏细节，Standing Orders 相当于给 Claude 做了一次"岗前培训"，之后它就记住规则了。

**详细解释**：

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1. 给目标 | 用自然语言描述需求 | 例："帮我写一个 Standing Order，用 OpenCLI 收集多平台 AI 资讯" |
| 2. 给资料 | 提供工具官方文档链接 | Claude 自动抓取、消化文档，提炼关键命令 |
| 3. 看成品 | 审核 Claude 生成的 Standing Order | 可作为模板，复用到其他类似指令 |

**实际应用场景**：写作风格规范（如"去 AI 味"检查清单）、每日信息收集任务、固定格式的内容输出要求。

---

#### 概念二：Heartbeat 与 Cron 自动调度机制

**是什么**：Claude 内置的两种定时自动执行机制，让任务无需人工触发、按计划自动运行。

**为什么重要**：把 Claude 从"被动回答"变成"主动执行"，实现真正的自动化工作流。

**详细解释**：

| 对比维度 | Heartbeat | Cron |
|----------|-----------|------|
| 触发方式 | 间隔轮询（有事处理，没事等待） | 精确时间点触发（闹钟式） |
| 最小间隔 | 30 分钟 | 精确到秒 |
| 任务并行 | 支持多任务叠加运行 | 一条任务一个坑 |
| 适用场景 | 监控类、汇总类任务 | 需要精确时间响应的任务 |

**实际应用场景**：用 Heartbeat 每天自动统计自媒体账号播放量和点赞数；用 Cron 在每天固定时刻触发内容发布或数据抓取。

---

#### 概念三：四种外接工具接入方式

**是什么**：Claude 连接外部软件和服务的四条通道，让它能真正"操作"而不只是"回答"。

**为什么重要**：绝大多数真实工作不是问答，而是操作——读文件、改代码、抓数据、管项目。外接工具是 Claude 从聊天机器人变成工作系统的关键。

**详细解释**：

| 接入方式 | 适用对象 | 典型软件/场景 |
|----------|----------|---------------|
| MCP | 有标准接口的主流软件 | Notion、GitHub、Airtable、Figma、Canva、Slack、GitLab |
| CLI 工具 | 无开放接口的平台（自媒体/实时数据） | OpenCLI：抓评论、热榜、字幕、创作者数据、英文趋势 |
| Harness | 专业 Agent 编排 | Claude 拆任务 → Codex 进仓库改代码跑测试 → 结果回 Claude |
| 内置工具 | 通用操作 | 浏览器、文件读写、终端命令、搜索 |

**实际应用场景**：做视频选题时，用 OpenCLI 抓平台数据 + 浏览器读官方文档 + Airtable 管理选题状态 + Codex 处理代码 Demo，Claude 统筹全流程。

### 2.4 速查表

| 功能需求 | 用哪个 | 关键特点 |
|----------|--------|----------|
| 固化重复 Prompt | Standing Orders | 一次训练，永久生效 |
| 定时监控/轮询任务 | Heartbeat | 最短30分钟，可并行多任务 |
| 精确时间点触发任务 | Cron | 精确到秒，单任务专用 |
| 对接 Notion/GitHub 等主流软件 | MCP | 需软件支持 MCP 服务 |
| 抓取自媒体平台数据 | CLI 工具（OpenCLI） | 把手动操作变成可调用命令 |
| 代码任务自动化 | Harness + Codex | Claude 拆任务，Codex 执行 |
| 读文件/开网页/跑终端 | 内置工具 | Claude 原生支持，无需额外配置 |

### 2.5 延伸思考

1. **Standing Orders 的边界在哪里？** 长驻指令能覆盖的是"规则固定"的重复任务，但如果业务需求经常变化，频繁修改 Standing Orders 的成本是否会超过直接写 Prompt？什么类型的任务最适合沉淀成 Standing Orders？

2. **Heartbeat 和 Cron 的可靠性如何保障？** 自动化任务一旦出错（比如平台接口变化、网络超时），Claude 能否自动检测并告警？在没有人工监督的情况下，如何设计"自我纠错"机制？

3. **四种外接方式如何组合才最高效？** 视频展示了多工具协作的理想状态，但实际搭建中，MCP、CLI、Harness、内置工具之间的数据如何流转？上下文长度限制会不会成为多工具协作的瓶颈？
