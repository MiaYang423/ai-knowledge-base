## 一、视频原文

这么多 coding 模型到底应该怎么去选？2025 年人人都在玩 vibe coding，但用对模型才是关键。模型选错了效果不好，又分分钟烧光你的钱包。

很多人都用 AI 写代码还停留在一个模型从头写到尾的阶段，但这样做真的效率非常低。毕竟模型也有术业有专攻的问题。我现在用的是分层写作模式，让不同的模型做自己最擅长的内容。

第一层，我的前端 Demo 使用 Gemini 2.0 Flash 或 Gemini 2 Pro，这个模型有个特点就是 UI 的界面做得特别漂亮，效果设计也很棒。我一般会用它去快速搭一个原型出来，把界面的效果和基础的交互跑通。但它有个明显短板，逻辑推理能力不行，复杂的后端逻辑容易出问题。

所以到第二层，我会切到 Claude 系列。这里分两种情况：如果是常规的编码，我会用 Claude 3.5，执行效率很高，性价比也很不错；但遇到复杂架构设计或者奇怪的 bug，我会直接上 Claude Opus 4.5，算贵一点，但解决复杂问题的能力确确实实要强很多。

举个例子，三个月前我在做我们 AI 视频剪辑工具的后台管理系统。我先用 Gemini 搭了个管理界面，花了半个小时，有了不错的视觉效果。然后用 Claude Opus 去设计整体的架构，包括数据库的设计、API 接口规划等等。架构定义好以后，具体的代码实现就交给 Claude 3.5 去跑，跑了一天时间，一整个系统就跑下来了。

另外，如果你不差钱，你全程用 Claude Opus 开 agentic 模式，效果肯定是最好的。但对大部分人来说，性价比还是需要考虑的。

那这里我再推荐一个平替方案：Gemini 2.5 Pro，这个模型我最近用下来感觉很不错，水平跟 Claude 3.5 差不多，有些推理的处理甚至更好，关键是便宜很多，主打就是量大管饱。唯一的缺点就是有时候你可能要多跑几轮去验证，但问题都能解决。

而 Gemini 2.0 Pro 其实也挺有潜力的，价格也很划算，但 Gemini 的安全审查能力我觉得还是有待提高，跟 Claude 的代码质量相比还是会有一些差距。

至于 GPT 的话，我真的基本不用它去写代码，不是说质量不行，就是它的整个性价比较低，同样的钱能够买到更好的服务，那为什么还要选它呢？

所以总结一下我自己的模型选用策略：前端原型用 Gemini 2.0 Pro，快速出效果，UI 设计美观；架构设计用 Claude Opus，保证方向不跑偏，推理能力很强；具体编码在 Claude 3.5 和 Gemini 2.5 Pro 之间看预算去决定。

另外也别迷信某一个模型，AI 工具真的更新太快了，根据具体需求来选择才是最务实的做法。

### 2.3 核心概念精讲

#### 概念一：分层写作模式

**是什么**：将一个完整的开发任务拆分为多个阶段，每个阶段选用最擅长该任务的 AI 模型，而不是全程用同一个模型。

**为什么重要**：
不同模型在 UI 生成、逻辑推理、架构设计上的能力差距显著；用单一模型会在它的短板上浪费时间和金钱，分层使用可以让每一分钱都花在刀刃上。

**详细解释**：

| 层级 | 任务类型 | 推荐模型 | 核心理由 |
|------|----------|----------|----------|
| 第一层 | 前端原型 / UI 搭建 | Gemini 2.0 Pro | UI 美观，出效果快，适合快速验证 |
| 第二层 | 系统架构 / 数据库设计 / API 规划 | Claude Opus 4.5 | 推理能力强，架构方向把控准 |
| 第三层 | 具体代码实现 | Claude 3.5 / Gemini 2.5 Pro | 执行效率高，性价比好 |

**实际应用场景**：
视频中举例：做 AI 视频剪辑工具的后台管理系统——先用 Gemini 搭界面（30 分钟出视觉效果），再用 Claude Opus 做架构设计，最后用 Claude 3.5 跑具体代码实现，一天内完成整个系统。

---

#### 概念二：性价比优先的模型选择逻辑

**是什么**：在效果相近的前提下，优先选择单位成本更低的模型；只在真正需要时才动用高价模型。

**为什么重要**：
高价模型（如 Claude Opus）并非在所有任务上都比平价模型强出足够多，盲目使用会快速烧光预算，尤其在大量代码生成场景下成本差异极为显著。

**详细解释**：

| 场景 | 推荐选择 | 理由 |
|------|----------|------|
| 常规编码 | Claude 3.5 | 效率高，性价比好 |
| 复杂 bug / 复杂架构 | Claude Opus 4.5 | 贵但值，解决复杂问题能力明显更强 |
| 预算有限的编码 | Gemini 2.5 Pro | 价格更低，水平接近 Claude 3.5 |
| 不差钱追求最优 | Claude Opus + agentic 模式 | 效果最好，但成本最高 |
| 不推荐用于写代码 | GPT 系列 | 同等价格可买到更好服务 |

**实际应用场景**：
对大多数独立开发者或小团队，Gemini 2.5 Pro 是"量大管饱"的平替首选，多跑几轮验证即可弥补偶尔的不稳定性。

---

#### 概念三：模型各自的能力边界

**是什么**：每个主流 coding 模型都有明确的优势区间和短板，了解边界才能合理分配任务。

**为什么重要**：
把不擅长的任务交给错误的模型，不仅出错率高，还会浪费大量 token 成本在反复修正上。

**详细解释**：

| 模型 | 优势 | 短板 |
|------|------|------|
| Gemini 2.0 Pro | UI 界面美观，前端效果好，出图快 | 复杂后端逻辑推理能力弱 |
| Claude Opus 4.5 | 复杂推理、架构设计、疑难 bug 定位 | 价格贵，日常编码性价比低 |
| Claude 3.5 | 常规编码执行效率高，性价比好 | 极复杂问题不如 Opus |
| Gemini 2.5 Pro | 价格便宜，推理能力接近 Claude 3.5 | 偶尔需要多轮验证才稳定 |
| GPT 系列 | 质量尚可 | 性价比在当前竞争格局下偏低 |

**实际应用场景**：
遇到"奇怪的 bug 怎么都解决不了"时，直接升级到 Claude Opus，而不是在低价模型上反复挣扎浪费时间。

### 2.5 延伸思考

1. **分层策略的切换成本如何控制？** 在不同模型之间传递上下文（代码、架构文档、需求）时，如何设计 prompt 和交接格式，才能减少信息损耗、避免"换了模型就忘了前面做了什么"的问题？

2. **Gemini 2.5 Pro"多跑几轮验证"的隐性成本值得吗？** 如果把多轮验证的时间成本折算进去，它相对于 Claude 3.5 的价格优势还剩多少？什么类型的项目更适合用它，什么类型的项目反而不划算？

3. **"别迷信某一个模型"在实操中如何落地？** 随着各家模型快速迭代，如何建立一套自己的周期性评测机制，而不是靠零散的感受来判断哪个模型当前最值得用？

---

### 2.6 一句话记住它
