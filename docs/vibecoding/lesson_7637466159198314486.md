### 2.3 核心概念精讲

#### 概念一：Expo Blank TypeScript 模板

**是什么**：Expo 是一个专门用来开发手机 APP 的工具框架，Blank TypeScript 是它提供的一个最简单的空白起步模板，使用 TypeScript 语言。

**为什么重要**：
用这个模板初始化项目，可以让 Codex 在一个标准、干净的结构里开始写代码，避免从零手动配置环境的麻烦。

**详细解释**：

| 术语 | 含义 |
|---|---|
| Expo | 做手机 APP 的开发工具，支持 iOS 和 Android |
| Blank | 空白模板，没有多余的预置页面和功能 |
| TypeScript | JavaScript 的加强版，有类型检查，适合正式项目 |
| 初始化后生成的内容 | APP 入口文件、配置文件、依赖声明、脚本 |

**实际应用场景**：
在 Codex 对话框里发送一句话——"请用 Expo Blank TypeScript 模板初始化这个空目录"——Codex 就会自动执行命令，把整个项目骨架生成好，不需要自己手动敲命令行。

---

#### 概念二：Codex 的任务排队 vs 插队机制

**是什么**：当 Codex 正在执行一个任务时，你又发了一条新消息，Codex 有两种处理方式：排队（等主任务完成再回答）或插队（暂停主任务先回答你）。

**为什么重要**：
理解这个机制，可以避免在 Codex 干活的时候误操作打断任务，也可以主动利用插队功能随时问问题。

**详细解释**：

| 模式 | 行为 |
|---|---|
| 排队模式 | 主任务执行完毕后，再处理你插进来的问题 |
| 插队模式 | 跳出主任务，先回答插入的问题，再继续主任务 |
| 如何切换 | 在 Codex 设置里直接配置 |

**实际应用场景**：
视频中，作者在 Codex 初始化项目的同时，插了一条"请用通俗语言解释 Expo"，Codex 可以根据设置决定是先解释还是等初始化完再解释。

---

#### 概念三：项目规范文件（Agent STANDARDS）

**是什么**：一个放在项目里的简短说明文件，告诉 Codex 这个项目的基本规则，比如项目类型、技术栈、开发原则、输出语言。

**为什么重要**：
有了这个文件，Codex 在整个项目过程中都会遵守这些规则，不会乱用云服务、不会突然切换语言，保持行为一致。

**详细解释**：

视频中生成的 Agent STANDARDS 只有五行核心内容：

| 字段 | 内容 |
|---|---|
| 项目类型 | Expo 移动应用 |
| 技术栈 | （按实际项目填写） |
| 开发方式 | 本地优先，不上云 |
| 依赖原则 | 不依赖外部云服务 |
| 输出语言 | 全程中文 |

**实际应用场景**：
项目初始化完成后，让 Codex 自动生成这个文件，后续所有任务都会以此为基准执行，不需要每次都重复说明规则。

---

#### 概念四：Codex 首次使用的关键设置

**是什么**：第一次打开 Codex 时需要完成的基础配置，决定 Codex 的工作方式和权限范围。

**为什么重要**：
设置不对，Codex 可能会在没有确认的情况下执行危险操作，或者每次都要手动审批，影响效率。

**详细解释**：

| 设置项 | 推荐值 | 说明 |
|---|---|---|
| 常规工作模式 | 按需选择 | 决定 Codex 主动程度 |
| 权限安全访问 | 默认关闭 | 防止 Codex 随意访问系统 |
| 批准策略 | Auto-approve | 减少每步手动确认 |
| 作用范围 | Workspace-wide | 限定在当前工作区内操作 |
| 个性化自定义指引 | 填写核心偏好 | 先填重要的，后续慢慢补充 |

**实际应用场景**：
第一次打开 Codex 时先把这几项配好，后续跟 Codex 磨合的过程中再逐步调整，不需要一次性全部设置完美。

### 2.5 延伸思考

1. **模型一致性的代价是什么？** 视频中提到项目确定用哪个模型后就不要换，但如果新模型能力更强，中途换模型会带来哪些具体风险？代码风格不一致、上下文丢失，还是其他问题？

2. **Agent STANDARDS 只有五行够用吗？** 视频里的规范文件非常精简，随着项目复杂度增加，什么时候需要扩充这个文件？哪些内容值得加进去，哪些又会变成噪音干扰 Codex？

3. **插队和排队机制背后的效率权衡是什么？** 插队可以随时得到解释，但会打断主任务；排队保证任务连续，但你得等。在实际项目开发中，什么情况下应该用插队，什么情况下应该让 Codex 专心把任务跑完？
