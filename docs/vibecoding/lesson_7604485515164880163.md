# 教案：零代码一下午做出网页应用并上线

> 适合人群：想用 AI 工具独立做出产品但完全不会写代码的普通人

## 一、视频原文

我用了一下午做出了一个完整的网页应用，还部署上线了，任何人都能访问，关键是我一行代码都没写。今天我把整个过程拆解给你，跟着做你也行。

整个过程就四步：第一步明确需求，第二步生成原型，第三步原型转代码，第四步部署上线。接下来我一步一步跟你说清楚。

先说第一步，头脑风暴，明确需求。这一步是整个流程中最重要的一步，很多人会忽略它，但我必须告诉你，你最终做出来的东西好不好，80% 取决于你这一步做得扎不扎实。

具体怎么做呢？打开 ChatGPT、Claude 或者 Gemini，随便选一个你顺手的就行。然后把你脑海中那个模糊的想法告诉它，不用担心说得不够清楚，你就像跟一个朋友聊天一样，把你想做什么、给谁用、大概用什么功能，一股脑说出来就行。

关键在于你不要只聊一轮就结束了，你要多跟 AI 聊几轮，让它反问你。你没想到的地方它会帮你想到，你不确定的地方它会帮你分析利弊。就这样一来一回聊个十几二十轮，你会发现你的想法越来越清晰了。

聊到什么程度算够了呢？最好是能明确到你的应用有哪些模块、每个模块有哪些功能、一共有几个页面、每个页面上大概有什么内容、用户操作流程是什么样的。当然如果你觉得一次性想不了那么全也没关系，你可以先明确两三个核心功能先做出来，后面再慢慢加。

最后让 AI 把你们所有的讨论成果汇总成一份文档，这份文档有个专业的名字叫 PRD，就是产品需求文档。

好，拿到这份 PRD 之后，我们进入第二步，生成原型界面。这一步我们需要用到一个工具叫 Google Stitch。你把刚才那份完整的 PRD 文档扔给 Stitch，它会根据你的需求自动帮你画出应用的 UI 界面。生成之后你看一下，如果觉得布局不好看或者不合理，你就继续跟它说，让它调整。比如你想要暗黑风格你就告诉它，你觉得某个按钮位置不对你也告诉它，就这样反复微调，直到所有页面让你满意为止。

这一步结束之后，你手上应该有一套完整的、覆盖所有页面的应用原型。这个原型看起来已经十分接近最终成品了，但它还只是一个壳，接下来我们要把它变成真正能跑的代码。

进入第三步，原型转代码。在 Stitch 里面有一个功能，可以直接把你做好的原型导出到 Google IDX。导出之后，Google IDX 就会基于你的原型文件自动生成一整套可以运行的前端代码。它默认使用的技术栈是 React 加 TypeScript，你不懂这些也没关系，不影响你使用。

如果你的项目比较简单，纯前端不涉及什么后端服务器之类的，那你甚至可以在 Google IDX 里面就把所有事情做完，在里面预览、调试、修改都可以。

这里特别提醒，IDX 可以一键把代码推送到 GitHub，这一步一定要做，相当于给你的代码上了个保险，改崩了随时能回退。

如果你的项目比较复杂，需要接后端服务，或者需要更精细的调整，那你可以把代码从 GitHub 拉到本地，用专业的编程工具继续开发。这里推荐几个工具：Cursor、Claude Code、Cline，都可以，选你喜欢的就行。遇到报错不用慌，把报错信息直接扔给 AI 让它帮你修就行。记住一个原则：小步迭代，改一点测一点，不要一次性改太多东西。

好，代码搞定之后，我们来到最后一步，部署上线。这一步超级简单，打开 Vercel 这个网站，用你的 GitHub 账号登录，然后选择你刚才推送到 GitHub 的代码仓库。如果项目需要环境变量，比如一些 API Key 之类的，在部署页面填一下，然后点击部署。Vercel 会给你生成一个专属的网址，任何人通过这个网址都能访问你的应用。

到这里，你已经拥有一个完全属于自己的网页应用了。从一个模糊的想法到一个正式上线的产品，整个过程你可能一行代码都没有手动写过。

最后快速回顾一下：第一步，用 ChatGPT、Claude 跟 AI 聊清楚需求，输出一份 PRD 文档；第二步，把 PRD 扔给 Google Stitch，生成完整的 UI 原型；第三步，从 Stitch 导出到 Google IDX，自动生成代码，推送到 GitHub；第四步，打开 Vercel，一键部署上线。

我知道你可能觉得这些工具名字听着就头大，但你真正上手之后就会发现，每步其实只需要点击几下鼠标、跟 AI 说几句话。还是先照着做一遍，做完你就懂了。我把完整的提示词、工具链接、原型代码全部整理好了，拿到就能直接用，评论"Cursor"免费发给大家。

### 2.3 核心概念精讲

#### 概念一：PRD（产品需求文档）

**是什么**：把你对产品的所有想法和需求整理成一份结构化文档，是后续所有步骤的"蓝图"。

**为什么重要**：视频明确说"最终做出来的东西好不好，80% 取决于这一步做得扎不扎实"。后续的原型生成、代码生成都依赖这份文档的质量。

**详细解释**：

| 要素 | 说明 |
|------|------|
| 模块划分 | 应用由哪几个大模块组成 |
| 功能列表 | 每个模块下有哪些具体功能 |
| 页面清单 | 一共几个页面，每页有什么内容 |
| 操作流程 | 用户从进入到完成任务的完整路径 |

**实际应用场景**：用 ChatGPT/Claude/Gemini 多轮对话，最终让 AI 把聊天结果汇总输出成这份文档，直接复制扔给下一步工具使用。

---

#### 概念二：原型（UI Prototype）

**是什么**：用 Google Stitch 根据 PRD 自动生成的应用界面视觉稿，看起来接近真实产品，但还不能运行。

**为什么重要**：它是需求文字和可运行代码之间的"翻译层"，让你在写代码之前就能看到并调整产品长什么样，避免做出来才发现不对。

**详细解释**：
- 输入 PRD → Stitch 自动生成所有页面的 UI 布局
- 可以继续用自然语言指令调整：风格（如暗黑风格）、布局、按钮位置等
- 反复微调直到满意，才进入下一步

**实际应用场景**：把 PRD 粘贴进 Google Stitch，告诉它"我想要暗黑风格""这个按钮移到右上角"，迭代到满意为止。

---

#### 概念三：GitHub 版本控制

**是什么**：一个代码托管平台，可以保存代码的每一次历史版本。

**为什么重要**：视频特别强调"这一步一定要做，相当于给代码上了个保险，改崩了随时能回退"。对于不懂代码的人来说，这是防止把项目搞坏又无法恢复的关键保障。

**详细解释**：
- Google IDX 支持一键将代码推送到 GitHub
- 每次改动前推送一次，出问题可以回到上一个可用版本
- 同时也是 Vercel 部署的代码来源

**实际应用场景**：在 IDX 里点一下"推送到 GitHub"，之后 Vercel 直接连接这个仓库完成部署。

---

#### 概念四：Vercel 一键部署

**是什么**：一个专门用于部署前端网页应用的平台，连接 GitHub 仓库后可以自动构建并生成可访问的网址。

**为什么重要**：让"本地能跑"变成"全世界都能访问"，是整个流程的最后一公里，且操作极简。

**详细解释**：

| 步骤 | 操作 |
|------|------|
| 登录 | 用 GitHub 账号登录 Vercel |
| 选仓库 | 选择刚推送的代码仓库 |
| 填环境变量 | 如有 API Key 等敏感信息在此填写 |
| 点击部署 | Vercel 自动构建，生成专属网址 |

**实际应用场景**：整个操作不超过5分钟，部署完成后得到一个形如 `yourapp.vercel.app` 的网址，发给任何人都能打开使用。

### 2.4 速查表

| 步骤 | 目标 | 工具 | 输入 | 输出 |
|------|------|------|------|------|
| 第一步：明确需求 | 把模糊想法变成清晰文档 | ChatGPT / Claude / Gemini | 你的想法（自然语言） | PRD 产品需求文档 |
| 第二步：生成原型 | 把文档变成可视化界面 | Google Stitch | PRD 文档 | 所有页面的 UI 原型 |
| 第三步：原型转代码 | 把界面变成可运行程序 | Google IDX → GitHub（复杂项目再用 Cursor/Claude Code/Cline） | UI 原型文件 | 可运行的前端代码 + GitHub 仓库 |
| 第四步：部署上线 | 让任何人都能访问 | Vercel | GitHub 仓库 | 专属访问网址 |

| 场景 | 推荐方案 |
|------|------|
| 项目简单（纯前端） | 全程在 Google IDX 内完成 |
| 项目复杂（需要后端/精细调整） | 从 GitHub 拉到本地，用 Cursor / Claude Code / Cline |
| 遇到报错 | 把报错信息直接发给 AI，让它修 |
| 担心改崩 | 每次改动前先推送到 GitHub |

### 2.5 延伸思考

1. **需求阶段的质量瓶颈**：视频说"80% 的质量取决于第一步"，但如果你本身对自己想做什么都不清楚，AI 的反问和引导能补足多少？有没有什么方法可以在开始聊之前先做一些准备，让第一步效率更高？

2. **这条工具链的边界在哪里**：视频介绍的流程主要适合前端网页应用，如果产品需要用户登录、数据存储、支付等后端功能，这套流程还能走多远？哪个环节会最先遇到障碍？

3. **Vibe Coding 的可维护性问题**：一行代码都没写就做出来的产品，如果上线后需要持续迭代、修复 bug、增加功能，长期维护的成本和风险会是什么样的？和自己写代码的项目相比有哪些本质差异？
