# 教案：Claude 发一句话后到底发生了什么

> 适合人群：刚开始用 Claude 或 AI 编程工具、想搞清楚"软件"和"模型"区别的初学者

## 一、视频原文

大家好，我是小鑫。在之前的分享中，很多朋友没分清楚 Claude 客户端跟 Claude 模型的区别。本期视频主要讲他们之间的区别，以及当你让 Claude 跑一个任务——比如竞品分析——这个词进来之后，Claude 客户端跟模型分别都会做什么事情。以这个东西为例，去解释一下这次调用过程中都发生了什么。

那么简单来说，Claude 的 Anthropic、Sonnet、Haiku，他们就是大脑，然后 Claude 客户端本身就是手脚，去执行各种东西。我们只有脑子没有东西去执行是不行的，我们执行但没有聪明脑子也是不行的，这就是他们之间的一个关系。

我们具体来看，当我们去选择 Claude 客户端，给他一个任务说"帮我做一下竞品分析"，会发生什么事情。

那么首先，Claude 客户端这个请求进来之后，他会去加载 Claude 的 System Prompt。比如说你让他说话风格一定是说中文的，模型也可以让他的风格是简洁的，或者说让他不要省略，把东西都写得很详细——你在这里边会定义很多静态的一些定义，就像对他的要求一样。

然后这个时候会加载 Memory。比如说你之前跨会话的时候，是不是之前就已经让他做过别的竞品分析？那么之前我们在某次对话中别的竞品是什么，其他的竞品是哪两个，以及我们的产品是什么。

然后就是 Claude 客户端他自己定义的三四十种预加载的 Tools，一些工具，可以供后面的模型去看我们到底用哪个工具。

然后他会把这些东西打包好，作为一个上下文，由 Claude 这个软件把这个事情打包，连同用户这次"竞品分析"的请求一块打包给模型。

好了，这些东西给了脑子之后，那我们脑子可以是 Claude 自己的 Anthropic 模型，上面也可以是你自己接入的任何模型，比如 Gemini 或者其他的模型。他在训练过程中可能不是按照这些 Tools 去训练的，所以说他实际的适配效果也不好说，因为他未必像 Claude 一样去适配得那么好。

我想表达的是，模型在训练过程中可能会把对这些工具的使用作为训练的一部分。那么如果说一些其他的大模型，他们不是 Claude 家的话，他可能就没有经过这些训练，不知道可能要用这些 Tools 干嘛，会造成一些效果上的损失。这是有关系的。

然后我们所有上下文打包好给了模型之后，模型会去 Thinking，这个东西我们平常在使用的时候都已经见到过了。他会去想竞品是什么，他会去分析 Memory 里之前有没有做过类似的课题，之前是怎么做的，他会去看哪些工具要去调用。

他想完这些事情之后，会在这个过程中发出一条明确的指令，比如说去 Web Search 一下——就是他有一个工具叫 Web Search，就去往网上搜索。模型思考之后发出了这样一条指令，然后这个指令由 Claude 客户端这个软件去执行，然后去 Web Search，去网上查到这个东西之后回传给模型。模型理解完之后去生成，生成完之后他再让 Claude 客户端保存到一个比如说 Root 文件夹的报告里面，然后给我们去看。

这就是整个过程。这个过程中 Claude 客户端这个软件以及 Claude 背后的模型 Anthropic 是怎么协同的，以及我们正常的一次请求进来之后到返回结果都发生了什么，希望这条视频能够帮大家去理解这些东西。我是小鑫，关注点击爆炸。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**Claude 客户端是"手脚"、模型是"大脑"，一次请求从发出到返回结果，中间经历了 System Prompt 加载、Memory 读取、Tools 调用、模型 Thinking 四个关键步骤。**

---

### 2.2 知识地图

```
Claude 一次请求的完整链路
│
├── 1. 客户端 vs 模型 的基本关系
│     ├── Claude 客户端（软件）= 手脚，负责执行
│     └── Anthropic 模型（Sonnet/Haiku等）= 大脑，负责思考
│
├── 2. 请求进来后，客户端做的事（打包上下文）
│     ├── 加载 System Prompt（静态指令：语言/风格/详细程度）
│     ├── 加载 Memory（跨会话记忆：之前的竞品、产品信息）
│     └── 加载预定义 Tools（三四十种工具，如 Web Search）
│
├── 3. 上下文打包 → 发给模型
│     └── 用户请求 + System Prompt + Memory + Tools 列表 = 完整上下文
│
├── 4. 模型做的事（Thinking）
│     ├── 分析任务目标
│     ├── 查看 Memory 中的历史信息
│     ├── 决定调用哪个 Tool
│     └── 发出工具调用指令（如：Web Search）
│
├── 5. 客户端执行工具 → 结果回传模型
│     └── 模型生成最终结果 → 客户端保存/展示
│
└── 6. 模型适配性问题
      ├── Claude 原生模型：训练时已适配这些 Tools，效果最好
      └── 第三方模型（如 Gemini）：未必经过 Tools 训练，可能有效果损失
```

---

### 2.3 核心概念精讲

#### 概念一：客户端 vs 模型

**是什么**：Claude 客户端是运行在你电脑上的软件（手脚），Claude 模型（如 Sonnet、Haiku）是 Anthropic 训练的 AI 大脑，两者分工不同、协同工作。

**为什么重要**：很多人以为"Claude"是一个整体，搞不清楚出问题时是软件的问题还是模型的问题，也不知道为什么换了第三方模型效果会变差。

**详细解释**：

| 维度 | Claude 客户端（软件） | Claude 模型（大脑） |
|------|----------------------|-------------------|
| 本质 | 应用程序 | AI 参数/权重 |
| 比喻 | 手脚 | 大脑 |
| 负责什么 | 打包上下文、调用工具、保存结果 | 理解任务、推理、决策、生成内容 |
| 可替换吗 | 固定（Claude 客户端） | 可以换成 Gemini 等第三方模型 |

**实际应用场景**：你在 Claude 客户端里换成 Gemini 模型，工具还是那些工具，但 Gemini 可能不知道怎么用这些工具，导致效果变差。

---

#### 概念二：System Prompt（系统提示词）

**是什么**：在用户发消息之前，Claude 客户端预先注入给模型的一段静态指令，定义模型的行为规则。

**为什么重要**：它决定了模型的"性格"和"工作方式"，是每次对话的隐形基础设置。

**详细解释**：

- 可以设定语言（如：必须用中文回答）
- 可以设定风格（如：简洁 / 详细）
- 可以设定行为规则（如：不要省略步骤，把所有内容都写完整）

**实际应用场景**：你希望 Claude 每次都用中文、回答详细不省略，就在 System Prompt 里写好，之后每次对话都会自动生效，不用反复提醒。

---

#### 概念三：Memory（跨会话记忆）

**是什么**：Claude 客户端保存的历史信息，可以在新的对话中自动加载，让模型"记得"之前发生过的事。

**为什么重要**：AI 模型本身没有记忆，每次对话都是全新的。Memory 机制让客户端帮模型"补上"历史背景，实现跨会话的连续性。

**详细解释**：
- 比如你之前做过竞品分析，记录了竞品 A、竞品 B 的信息以及你自己产品的定位
- 下次新对话再让它做竞品分析，Memory 会把这些背景信息自动打包进上下文
- 模型就能基于之前的信息继续工作，而不是从零开始

**实际应用场景**：做持续性项目时（如长期竞品追踪），Memory 让每次对话都能接续上次的结论，不用重复粘贴背景信息。

---

#### 概念四：Tools 工具调用

**是什么**：Claude 客户端预定义的三四十种工具（如 Web Search、文件保存等），模型可以在 Thinking 过程中决定调用哪个，由客户端负责实际执行。

**为什么重要**：模型只能"想"，不能直接上网搜索或保存文件。Tools 是连接模型思维和真实世界操作的桥梁。

**详细解释**：工具调用的完整流程：

```
模型 Thinking
  → 决定需要 Web Search
  → 发出工具调用指令给客户端
  → 客户端执行 Web Search
  → 搜索结果回传给模型
  → 模型基于结果继续生成
  → 生成完毕后指令客户端保存到文件
```

**实际应用场景**：你让 Claude 做竞品分析，它会自动搜索网上最新信息，而不是只靠训练数据里的旧知识，最后还能把报告保存成文件。

---

### 2.4 速查表

| 环节 | 执行者 | 具体内容 | 类比 |
|------|--------|----------|------|
| 加载 System Prompt | 客户端 | 语言、风格、行为规则等静态指令 | 给员工的岗位说明书 |
| 加载 Memory | 客户端 | 跨会话历史信息（竞品、产品等） | 员工的工作笔记本 |
| 加载 Tools 列表 | 客户端 | 三四十种可用工具清单 | 工具箱目录 |
| 打包上下文 | 客户端 | 以上三项 + 用户当前请求 | 把所有材料装进一个文件夹 |
| Thinking（推理） | 模型 | 分析任务、查看记忆、决定用哪个工具 | 大脑思考 |
| 工具调用 | 客户端执行 / 模型决策 | 如 Web Search、文件保存 | 大脑指挥手脚干活 |
| 生成结果 | 模型 | 基于工具返回结果生成最终输出 | 大脑整合信息写报告 |
| 保存/展示结果 | 客户端 | 存入文件夹或展示给用户 | 手脚把报告递给你 |

**模型选择对效果的影响：**

| 模型类型 | 对 Claude Tools 的适配 | 预期效果 |
|----------|----------------------|----------|
| Claude 原生模型（Sonnet/Haiku） | 训练时已包含 Tools 使用 | 效果最好 |
| 第三方模型（如 Gemini） | 未必经过相同训练 | 可能有效果损失 |

---

### 2.5 延伸思考

1. **Memory 的边界在哪里？** 如果 Memory 里存了大量历史信息，打包进上下文后会不会超出模型的上下文窗口限制？Claude 客户端是如何取舍哪些 Memory 值得被加载的？

2. **换第三方模型到底损失多少？** 视频提到 Gemini 等模型可能没有针对 Claude Tools 做过训练，那在实际使用中，这种"效果损失"会体现在哪些具体场景？是完全不会用工具，还是只是用得不够准确？

3. **System Prompt 和 Memory 都是"背景信息"，区别是什么？** 两者都会被打包进上下文，但一个是静态规则、一个是动态历史，在实际使用中应该把什么信息放 System Prompt、什么信息交给 Memory 管理？

---

### 2.6 一句话记住它

> Claude 客户端是手脚，模型是大脑，你发的每一句话背后，是一套"加载记忆→打包上下文→模型思考→调用工具→返回结果"的完整流水线在默默运转。
