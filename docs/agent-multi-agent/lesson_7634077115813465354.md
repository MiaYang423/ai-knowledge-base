# 教案：多 Agent 路由架构选型方法论

> 来源视频：[抖音博主 · 抖音](https://www.douyin.com/video/7634077115813465354)
> 适合人群：正在设计或面试多 Agent 系统的 AI 工程师和大模型应用开发者

---

## 一、视频原文

在构建多 Agent 的协同架构时，任务分发是一个核心环节。你如何评价基于 LLM 的语义路由与基于固定规则的确定性路由？在实际复杂场景中你又是如何权衡和选型的？

好的，面试官这是一个工程实践的问题。在分布式 Agent 架构中，路由层的设计直接决定了系统的稳定性与生产性。先说我的核心观点：固定规则决定系统的下线，而 LLM 路由决定系统的上线。在生产环境下，我们追求的既不是纯粹的灵活性，也不是死板的稳定性，而是确定性逻辑为主、语义为补充的协同治理。

我们要从底层信息论的角度来看。固定规则路由本质上是对业务逻辑的结构化投影，它要求输入特征空间是必要的，追求的是低延迟和幂等性。LLM 路由本质上是基于高维向量空间的语义映射，它利用了大模型的零样本学习能力去处理开放条件下的意图泛化问题，它解决的是从非结构化输入到原则化推理能力的对齐。

在实战中两者的差异主要体现在三个核心维度：

稳定性：规则分发不存在抖动，但在应对语义模糊时会失效；LLM 路由具备极强的推理泛化能力，但存在随机性。

系统工程：规则越多系统越难维护，成本成指数级上升；LLM 路由将逻辑结构化到 Prompt 中，降低了代码耦合，但增加了调优复杂度。

计算效能：规则路由是常数级或对数级的计算开销，而 LLM 路由引入了显著的推理延迟和 Token 成本。

在实际大厂落地中，我推崇的是漏斗式的分发模型：第一层，静态预备，利用正则或关键词哈希快速拦截高频标准化指令，要减少确定性，通过 Vector 设计，将查询与 Agent 的能力描述做余弦相似度计算，这是一种高性能的准语义路由；第二层，任务知角色，当前两层置信度不足时才调用 LLM，通过一个专门的调度 Agent，结合上下文信息进行最终裁决。这里一定要做 Agent 的 Retries 水，动态维护各个 Agent 的 Meta Data，让 LLM 路由有据可依。

最后总结一下我的技术哲学：规则是系统的骨架，保证它不犯低级错误；LLM 是系统的神经，让它产生感知。一个优秀的 Agent 开发工程师不应该试图用 LLM 去替代所有逻辑，而应该用 LLM 去覆盖那些传统逻辑覆盖不到的边界。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**多 Agent 路由设计的大厂标准答案是：固定规则守下线、LLM 语义路由守上线，漏斗式分层才是工程落地的最优解。**

```
多 Agent 路由架构选型
├── 两种路由方式
│   ├── 固定规则路由
│   │   ├── 本质：业务逻辑的结构化投影
│   │   ├── 优势：低延迟、幂等性、零抖动
│   │   └── 劣势：语义模糊时失效，规则多了维护成本指数级上升

│       └── 劣势：随机性、推理延迟、Token 成本高

    └── 第三层：LLM 调度 Agent（最终裁决）
```

---

### 2.3 核心概念精讲

#### 固定规则路由

**是什么**：用预定义的规则（正则、关键词匹配、哈希等）将用户请求分发到对应 Agent 的路由方式。

**为什么重要**：它是系统稳定性的底线，保证高频、标准化请求的准确性和低延迟，成本几乎为零。

**详细解释**：
- 优点：不存在随机性，可预测，计算开销极低（常数/对数级）
- 缺点：对语义模糊的输入无能为力；规则积累越多，维护成本指数级上升

**实际应用场景**：拦截"查邮件""发周报"这类高频、语义固定的指令，直接命中对应 Agent，不消耗 LLM 算力。

---

#### LLM 语义路由

**是什么**：利用大模型的语义理解能力，将非结构化的用户输入映射到对应 Agent 的路由方式。

**为什么重要**：它是系统能力的上限，能处理规则覆盖不到的模糊、复杂、长尾意图。

**详细解释**：
- 优点：零样本泛化能力强，能理解上下文和语义歧义
- 缺点：存在随机性（置信度不稳定），引入推理延迟和 Token 成本

**实际应用场景**：用户说"帮我看看上个月谁没交周报，顺便发邮件催一下"——这是查数据+发邮件两个意图，规则无法拆解，需要 LLM 理解。

---

#### 漏斗式分发模型

**是什么**：将路由拆成多层漏斗，从计算最轻的规则层到最重的 LLM 层，按置信度逐层过滤的分发架构。

**为什么重要**：这是大厂实际落地的标准打法，既保证准确率，又控制成本和延迟。

**详细解释**：

| 层级 | 方式 | 适用场景 | 成本 |
|------|------|----------|------|
| 第一层 | 静态规则（正则/关键词） | 高频标准化指令 | 极低 |
| 第二层 | Vector 余弦相似度 | 中频、有规律的意图 | 低 |
| 第三层 | LLM 调度 Agent | 模糊、复杂、长尾意图 | 高 |

**实际应用场景**：前两层拦截大部分流量后，只有少量真正复杂的请求才进入 LLM 层，整体成本大幅下降。

---

### 2.4 速查表

| 维度 | 固定规则路由 | LLM 语义路由 |
|------|-------------|-------------|
| 稳定性 | 无抖动，可预测 | 有随机性 |
| 延迟 | 常数/对数级，极低 | 推理延迟，较高 |
| 成本 | 几乎为零 | Token 消耗，较高 |
| 泛化能力 | 弱，语义模糊时失效 | 强，零样本泛化 |
| 维护复杂度 | 规则多了指数级上升 | 调优复杂，但代码耦合低 |
| 适用场景 | 高频标准化指令 | 模糊、复杂、长尾意图 |

### 2.5 延伸思考

1. 视频提到"动态维护各个 Agent 的 Meta Data，让 LLM 路由有据可依"——如果某个 Agent 的能力描述写得不准确，会对整个漏斗式路由产生什么影响？如何设计 Agent Meta Data 的更新机制？

2. 漏斗式分发模型中，第二层 Vector 余弦相似度和第三层 LLM 调度 Agent 的置信度阈值应该怎么设定？阈值设太高或太低分别会带来什么问题？

3. 视频说"规则越多系统越难维护，成本成指数级上升"——在实际工程中，什么时候应该停止增加规则、转而依赖 LLM 路由？有没有可量化的判断标准？

---

### 2.6 一句话记住它

> 规则是系统的骨架保证不犯低级错误，LLM 是系统的神经感知边界——漏斗式分层，让两者各司其职。
