# 教案：如何量化 Agent 性能

> 来源视频：[抖音博主 · 抖音](https://www.douyin.com/video/7618515523780840747)
> 适合人群：正在备战大模型 Agent 岗位面试、或需要在工程中落地 Agent 评测体系的开发者

---

## 一、视频原文

面试官问：你如何量化一个 Agent 的性能？大家注意啊，这道题可是现在大模型 Agent 面试里真正的分水岭。如果你开口只会说"看他任务成功率"，那面试官心里肯定在想：这个人估计还没真正做过复杂的业务落地。AI 大模型学习资料可以在主页置顶群里拿。

大家好，我是彭宇。为啥呢？因为传统的大模型评估只是说，你只要看它生成的文本质量就行。但 Agent 是要做事的，它涉及到了多轮交互、工具调用，还有环境的反馈。所以咱绝对不能用老一套的标准去量化。今天我就对着这张 Agent 评估全景图，把这套专家级的指标体系给大家彻底盘清楚。

咱先看第一层。咱得先想明白一件事，如果你是老板，你雇个 Agent 帮你干活，你最关心啥？首先肯定看结果嘛，也就是 Task level，任务到底成功没成功，这叫任务成功率。比如你让 Agent 帮你订张机票，它最后票订成了没？数据库里的状态改了没？这叫状态匹配。这是咱的底线。

但光看结果就够了吗？咱引导大家思考一下：Agent A 用了三步就把票定好了，Agent B 绕了 50 步，调了一堆没用的 API 才勉强定好，你能说他俩性能一样吗？显然不一样，对吧？所以咱必须得看过程维度，也就是轨迹评估。这里面有几个硬核指标：第一是工具调用准确率，API 选对了吗？参数传对了吗？第二是轨迹效率，看他有没有走弯路。第三是自我纠错率，这个非常关键，遇到报错反馈，它是直接摆烂，还是能自我反思，搞个 self-refine 然后重试成功？

最后咱做工程的还得算一笔账，就是系统维度。咱得看端到端的延迟，快不快？Token 消耗大不大？要是订张票花了 10 块钱的 Token 成本，那这 Agent 也就别干了，对吧？

好，指标订好了，那咱怎么实现自动化评测呢？总不能天天靠人肉盯着屏幕看，那得累死。

第一种，最硬核的就是基于规则的代码检验。比如 Agent 写了一段代码，咱就直接跑单元测试，过了就是 1，没过就是 0。这种办法最客观，只要是逻辑确定的任务，咱首选这个。

第二种，看环境的变化，也就是状态机比对。这个在 RPA 或者数据库 Agent 里特别常用。Agent 操作完，咱去比对一下数据库记录变没变，文件对不对，这能真实还原 Agent 对外部操作的影响。

但问题来了，要是任务没有标准答案咋办？比如你让 Agent 给客户写封安抚邮件，这就涉及到第三种——模型裁判，也就是 LLM-as-judge。咱找个更牛的模型，比如参数量更大的模型来当裁判，给它一套打分准则，让它从逻辑连贯性、安全性、语气这几个维度去主观打分。这就解决了无标准答案的痛点。

划重点了。面试官如果想考你有没有真实的实战经验，一定会追问："你觉得评测 Agent 最难的地方在哪？"这时候你得把这三个坑给他抛出来。

第一个坑：级联错误。啥意思？Agent 干活是连贯的，第一步规划错了，后面全盘崩溃。你很难判断到底是它"脑子"不行，还是"手"（也就是 API）太差。怎么填坑呢？咱得做模块化拆解评估，搞个 mock API，把环境反馈固化下来，专门压测它的规划层，这样才能精准归因。

第二个坑：非确定性。大模型这玩意儿，同样的问题问两遍，回答可能不一样；今天网络慢点，它可能就超时了。怎么填坑？咱得建沙盒，用容器每次测之前快照还原环境，让 Agent 永远在同一条起跑线上。这样测出来的数据才稳。

第三个坑：裁判失效。这就很尴尬了，Agent 可能在骗裁判，假装干完了；或者裁判模型自己看走眼了，给错误轨迹打了高分。怎么填坑？咱搞多路裁判机制，让几个模型一起辩论交叉验证，再配合人工定期抽检校准。

好，最后总结一下。以后面试再遇到这个问题，千万别只说一个准确率就完了。你要按照这三层逻辑一层层地来：先说立体化的指标体系，结果、过程、系统一个都不能少；再聊主客观结合的评估手段，代码检验配上模型裁判；最后一定要秀一把你的工程经验，聊聊怎么用沙盒环境、快照和多路裁判去解决实际落地里的那些脏活累活。只要你这套逻辑甩出来，面试官绝对会觉得你是一个真正带队打过仗的 Agent 专家。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**量化 Agent 性能不能只看任务成功率，要用"结果 + 过程 + 系统"三层指标体系，配合代码检验、状态比对、LLM 裁判三种评测手段，并直面级联错误、非确定性、裁判失效三大工程难题。**

---

### 2.2 知识地图

```
Agent 性能量化体系
├── 三层指标体系
│   ├── 结果维度（Task Level）
│   │   ├── 任务成功率
│   │   └── 状态匹配（数据库/文件状态）
│   ├── 过程维度（轨迹评估）
│   │   ├── 工具调用准确率（API 选择 + 参数）
│   │   ├── 轨迹效率（有无绕路）
│   │   └── 自我纠错率（self-refine 能力）
│   └── 系统维度
│       ├── 端到端延迟
│       └── Token 消耗成本
├── 三种评测手段
│   ├── 基于规则的代码检验（确定性任务首选）
│   ├── 状态机比对（RPA / 数据库 Agent）
│   └── LLM-as-judge（无标准答案任务）
└── 三大工程难题
    ├── 级联错误 → 模块化拆解 + mock API
    ├── 非确定性 → 沙盒容器 + 环境快照
    └── 裁判失效 → 多路裁判 + 人工抽检
```

---

### 2.3 核心概念精讲

#### 三层指标体系

**是什么**：将 Agent 性能从"结果、过程、系统"三个维度全面量化，而非只看最终任务是否完成。

**为什么重要**：同样完成任务的 Agent，过程效率可能相差十倍，系统成本可能相差百倍，只看结果会掩盖巨大差异。

**详细解释**：

| 维度 | 指标 | 示例 |
|-----|-----|-----|
| 结果 | 任务成功率、状态匹配 | 机票是否订成、数据库状态是否正确 |
| 过程 | 工具调用准确率、轨迹效率、自我纠错率 | 用 3 步 vs 50 步完成同一任务 |
| 系统 | 延迟、Token 消耗 | 订票花了 10 元 Token 成本 |

**实际应用场景**：Agent 产品上线前的性能基准测试，以及持续迭代时的回归评测。

---

#### LLM-as-judge（模型裁判）

**是什么**：用参数量更大、能力更强的大模型作为裁判，对 Agent 的输出按照预设维度进行主观打分。

**为什么重要**：大量 Agent 任务（如写邮件、生成报告）没有标准答案，无法用规则或状态比对评估，只能靠模型裁判。

**详细解释**：
- 给裁判模型一套打分准则（逻辑连贯性、安全性、语气等）
- 裁判模型输出分数和理由
- 风险：裁判本身可能失效 → 需要多路裁判 + 人工抽检

**实际应用场景**：客服邮件质量评估、开放式问答质量评测。

---

#### 三大工程难题

**是什么**：在实际落地 Agent 评测时必然遇到的三个系统性挑战：级联错误、非确定性、裁判失效。

**为什么重要**：这三个问题是区分"纸上谈兵"和"真正做过 Agent 工程"的分水岭，面试官必考。

**详细解释**：

| 难题 | 问题描述 | 解决方案 |
|-----|---------|---------|
| 级联错误 | 第一步错导致全盘崩，难以归因 | mock API + 模块化拆解评估 |
| 非确定性 | 同问题不同答，环境不稳定 | 沙盒容器 + 环境快照还原 |
| 裁判失效 | Agent 骗裁判或裁判打分错误 | 多路裁判辩论 + 人工定期抽检 |

**实际应用场景**：构建 Agent 自动化评测流水线时，这三个问题是必须在架构层面解决的核心挑战。

---

### 2.4 速查表

| 评测手段 | 适用场景 | 优点 | 局限 |
|---------|---------|-----|-----|
| 代码检验（规则） | 逻辑确定性任务 | 最客观，0/1 判断 | 只适合有标准答案的任务 |
| 状态机比对 | RPA / 数据库操作 | 真实还原外部影响 | 需要可观测的环境状态 |
| LLM-as-judge | 开放性任务 | 解决无标准答案问题 | 裁判本身可能失效 |

---

### 2.5 延伸思考

1. 自我纠错率（self-refine 能力）如何设计测试用例？如何区分"真正理解了错误并修正"和"随机重试碰巧成功"？

2. 多路裁判机制中，当不同裁判模型给出相互矛盾的评分时，如何设计仲裁规则？是投票、加权还是引入人工？

3. Token 消耗成本作为系统维度指标，如何与任务质量做 tradeoff 建模？什么情况下允许高成本换高质量，什么情况下必须控制成本？

---

### 2.6 一句话记住它

> 评测 Agent 就像考核一个员工：不只看结果，还要看做事过程和效率成本，更要能识别他是真干活还是在糊弄你。
