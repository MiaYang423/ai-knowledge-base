## 一、视频原文

一起来看看字节的 AI Agent 实习二面面经，看看你能答上来几道。这套题整体难度偏难，但是前面是基础，后面才是真正区分水平的地方。我是小哲，点赞收藏加关注，我们马上开始。

**第一道题：项目被整体考打时应该怎么讲？**

不要只备业务背景，核心是讲清楚你解决了什么问题，系统链路怎么走，指标怎么衡量，线上遇到过什么坑。面试官更想听干货，比如延迟、成本、稳定性、可观测性，你有没有真的做过。

**第二道题：你对多模态大模型有哪些了解？**

可以从输入能力讲起，比如图像、语音、视频和文本统一理解。关键不是报模型名，而是讲落地边界。多模态适合内容理解、内容减少、检索增强，但工程上要关注输入预处理、推理成本和结果校验。

**第三道题：Agent Skills 是怎么设计和使用的？**

Skills 本质是把可复用能力封装成工具，说明参数、约束和执行逻辑。设计时要让模型知道什么时候调用、怎么传参、失败怎么兜底。工程上一般还要做权限隔离、调用日志和结果校验。

**第四道题：如何加强大模型的记忆机制和多轮对话能力？**

短期靠上下文窗口，长期靠摘要、向量库和结构化用户画像。关键是不要把所有历史都塞回去，而是按任务检索相关记忆。多轮对话还要维护状态、意图和追问结果，避免越聊越偏。

**第五道题：多 Agent 执行策略的智能选择和切换机制怎么设计？**

可以把任务先拆成规划、执行、校验几类角色，再用路由器根据任务类型、置信度和失败次数切换 Agent。核心是要有退出条件和裁决机制，否则多个 Agent 很容易循环讨论，成本高还不稳定。

**第六道题：SSE 的局限性有哪些？**

SSE 适合服务端到客户端的单向流，做大模型打字机效果很方便。但它不适合双向通信，也有连接数代理超时和断线重连的问题，影响复杂交互场景。复杂场景里 WebSocket 往往更灵活。

**第七道题：如果 LoRA 效果不好你会怎么排查和改进？**

先看数据质量和任务分布，再看基座模型是否匹配，最后看 Rank、学习率、训练轮次这些参数。LoRA 的问题是表达能力有限，容易过拟合小数据。改进可以做数据清洗、增量评测和合并策略对比。

**第八道题：RAG 的动态知识更新怎么做？**

核心是把文档更新、切片、向量化、索引刷新做成流水线。增量更新要记录版本和时间戳，旧知识要能失效。线上还要考虑缓存召回延迟和一致性，不然用户问到的可能还是过期答案。

**第九道题：RAG 的评估方案怎么设计？**

不要只看最终回答好不好，可以拆成检索召回率、上下文相关性、答案忠实度和端到端满意度。工程上要准备标准问题集，记录命中文档、引用片段和模型输出，这样才能定位是检索错还是生成错。

**第十道题：如何评估复杂任务执行准确率的提升？**

复杂任务要拆成子步骤评估，比如规划是否合理、工具调用是否正确、最终结果是否达标。只看成功率太粗，更好的方式是建立任务级人工标注，关键检查点在对比改动前后的成功率、耗时和成本。

**第十一道题：MCP 和 Function Calling 的区别是什么？**

Function Calling 更像模型调用单个函数的协议，重点是参数结构化。MCP 更像一套连接外部工具和资源的上下文协议，适合把文件、数据库、服务能力统一暴露给 Agent。一个偏调用格式，一个偏生态连接。

### 2.3 核心概念精讲

#### Agent Skills 设计

**是什么**：将可复用的 Agent 能力封装成标准化工具，包含参数说明、约束条件和执行逻辑。

**为什么重要**：Skills 是 Agent 能力复用和模块化的基础，设计好坏直接影响 Agent 的稳定性和可维护性。

**详细解释**：
- 封装内容：参数定义、调用约束、执行逻辑、失败兜底
- 工程要求：权限隔离、调用日志、结果校验
- 让模型知道：什么时候调用、怎么传参、失败了怎么处理

**实际应用场景**：搜索 Skill、发邮件 Skill、查数据库 Skill——每个 Skill 独立封装，Agent 按需调用。

---

#### 多 Agent 切换机制

**是什么**：通过路由器根据任务类型、置信度和失败次数，在不同 Agent 之间动态切换执行策略的机制。

**为什么重要**：多 Agent 协作最常见的问题是循环讨论、成本失控，必须有明确的退出条件和裁决机制。

**详细解释**：

| 设计要素 | 说明 |
|----------|------|
| 角色分工 | 规划 Agent、执行 Agent、校验 Agent |
| 路由依据 | 任务类型、置信度、失败次数 |
| 切换时机 | 置信度不足或失败次数超阈值 |
| 退出条件 | 必须明确定义，防止死循环 |
| 裁决机制 | 冲突时由裁决层决定最终执行方向 |

**实际应用场景**：复杂任务中多个 Agent 分工协作，规划 Agent 拆任务，执行 Agent 完成子任务，校验 Agent 检查结果。

---

#### MCP vs Function Calling

**是什么**：两种不同层次的工具接入机制——Function Calling 是模型调用单个函数的协议，MCP 是连接外部工具和资源的上下文协议。

**为什么重要**：这是高频面试题，混淆两者说明对 Agent 工具层理解不到位。

**详细解释**：

| 维度 | Function Calling | MCP |
|------|-----------------|-----|
| 定位 | 模型调用单个函数 | 连接外部工具和资源的上下文协议 |
| 重点 | 参数结构化 | 生态连接，统一接入标准 |
| 适用场景 | 单次工具调用 | 文件、数据库、服务能力统一暴露给 Agent |
| 类比 | 打一个电话 | 接入整个通信网络 |

**实际应用场景**：Function Calling 适合简单的单次工具调用；MCP 适合 Agent 需要访问多种外部资源的复杂场景。

### 2.5 延伸思考

1. 视频说"多 Agent 很容易循环讨论"——如何设计一个有效的退出条件？最大循环次数、时间限制、置信度阈值，哪种方式更适合生产环境？

2. RAG 评估拆成检索和生成两层后，如果检索召回率很高但最终答案质量差，问题出在哪里？如何设计实验来定位是生成模型的问题还是上下文拼接的问题？

3. LoRA 表达能力有限、容易过拟合小数据——在什么情况下应该放弃 LoRA 转而做全量微调？两者的选择依据是什么？
