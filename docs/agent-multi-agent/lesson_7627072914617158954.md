# 教案：多 Agent 协同工作流实践

> 适合人群：非技术背景的产品经理或创业者，希望用多 Agent 协作工具实现低代码建站和产品设计的实践者

## 一、视频原文

哈喽大家，我是在自己干了五年多的产品经理，但是完全不懂代码，全文科背景的我。我的视频都是从零小白的视角来分享外部 Coding 的。

之前我跟大家分享过一套我外部 Coding 的工作流程，在小红书上有了非常不错的反响，有很多用户都觉得这个方法非常好用。这套 Workflow 分我两步：第一步是我会在 Claude 里面跟它去聊需求，让它根据我的诉求来画 Askma（原型图）；第二步呢，我会把 Askma 丢到像 Coding Agent 或者是 Lovable 这种网站里面，让它根据我的 Askma 来做网站。好处是什么？非常地还原，它不会给你乱画网站，你要什么样的功能，它基本就是按照 Askma 的样子把这个功能给实现出来，Token 消耗得很少，很省钱。

但问题也是很多朋友会跟我说我没有梯子。所以我一直在帮大家探索有没有可能有一款软件，能像 Claude 万能一样，我直接在这个软件里面就能够把我说的这两步流程包圆。这个软件就是腾讯的 QClaw。它们做了一个桌面端的应用，在这个应用里面内置了三个 Agent：一个叫悟部颜，一个叫代码行，一个是零语漫。这个代码行就是一个程序员的 Agent，我们一会儿会用到。

我自己 DIY 的 Agent 叫"套娃"，它是一个产品方案的大管家。其实我们类比来看，套娃就相当于原来的 Claude，而代码行就是原来我的 Coding Agent。这个套娃和这个代码行，它俩的分工是非常明确的。我会在套娃这个 Agent 里面跟它去聊产品方案，然后让它帮我产生 Askma，再到代码行的这个 Agent 里面跟它说按照这个 Askma 做一个 HTML 的网站。

有了这两个 Agent 彼此协同之后，我就不需要再去跨应用来完成我的需求设计了。而且你是可以用它们的默认大模型，或者是对接你自己的模型 API，或者是 Coding Plan 的。我们可以来看看效果，它会根据我给它的 Askma 来产出这样的一版，四个 Tab 之间可以交互的，点到某一个单级里面去，它还可以直接帮我跳转到小鱼桌里面去。听整个流程下来，我没有接触其他的 App，而且并不需要任何的梯子。

这就是多 Agent 的好处：因为 Agent 跟 Agent 之间是彼此隔离的，它们相当于每个人只做自己最擅长的事情，而不是一个 Agent 干所有的事情。就好像什么呢？好像你在上课的时候，每门课的老师它都是分开的，语文的教语文，美术的教美术。你可以给你的产品老师赋予它的设定，比如你告诉它是一名非常丰富经验的产品方案的顾问，这个 Prompt 是你自己给这个产品大管家的 Agent 设定的。然后你也可以在这里面给它起名字和上传头像。

我为什么会推荐这种多 Agent 之间协作的模式呢？我真心觉得让每个 Agent 各司其职，它并不是一种小技巧，而是未来的一种方法论。我不知道大家有没有最近刷到在小红书上很火的一个开源项目，叫三省六部制。它的核心就是像唐朝的三省六部制一样，皇上下达指令，太子来负责判断这个事情到底要不要做，然后中书规划，门下审议，尚书执行，然后最终到六部各司其职，执行自己的模块，最后汇总，把整件事情完成。

像我们刚刚在那套流程里面，有产品的 Agent 也有研发的 Agent，那后面我们其实可以扩展更多的 Agent，并且做到让 Agent 跟 Agent 之间彼此协同。你们可以看到在三省六部制的总控台里面，可以去监控到它的进度，让它们彼此之间真正做到自动协同的。我觉得这可能是未来的一种趋势，和一人公司真正能够实现的很核心的要素。那如果你喜欢今天的这期视频的话，欢迎点赞收藏评论哟，拜拜。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**非技术背景的产品经理可以在 QClaw 这类工具中，通过"产品 Agent + 开发 Agent"的多 Agent 协作模式，无需梯子、无需跨 App，完成从需求聊天到原型设计再到网站实现的全流程，这背后的"各司其职"理念将成为未来 AI 工作流的核心方法论。**

### 2.3 核心概念精讲

#### 多 Agent 各司其职模式

**是什么**：将不同职能（产品设计、代码开发等）分配给不同的专属 Agent，每个 Agent 只处理自己擅长的任务，通过协作完成完整工作流。

**为什么重要**：单一 Agent 处理所有任务会导致上下文混乱、能力稀释；专职 Agent 通过明确的职能边界和 Prompt 设定，输出质量更高、更可控。

**详细解释**：
- 类比：语文老师教语文，美术老师教美术，不会混淆
- 实现：通过系统 Prompt 给每个 Agent 设定专属角色和职能边界
- 优势：Token 消耗少，输出还原度高，流程可控

**实际应用场景**：产品 Agent 负责需求分析和原型设计，开发 Agent 负责按原型实现代码，两者协同完成从 idea 到产品的全链路。

---

#### 三省六部制（多 Agent 协同架构参考）

**是什么**：一个开源的多 Agent 协同架构项目，借鉴唐朝三省六部制的分工模式，设计 Agent 之间的层级协作关系。

**为什么重要**：提供了一个可参考的多 Agent 治理框架：决策层（判断）→ 规划层（中书）→ 审议层（门下）→ 执行层（尚书/六部），各层职责清晰，可监控进度。

**详细解释**：

| 角色类比 | 功能 |
|---------|-----|
| 皇上 | 用户下达最终指令 |
| 太子 | 判断任务是否执行 |
| 中书 | 规划任务方案 |
| 门下 | 审议方案合理性 |
| 尚书/六部 | 执行具体任务模块 |

**实际应用场景**：复杂多步骤任务的 Agent 协同架构设计，尤其适合需要审核和纠偏的高风险任务流程。

### 2.4 速查表

| 对比维度 | 原工作流（Claude + Lovable） | 新工作流（QClaw 多 Agent） |
|---------|--------------------------|------------------------|
| 是否需要梯子 | 需要 | 不需要 |
| 跨 App 操作 | 需要（Claude → Lovable） | 不需要（一个软件内完成） |
| Agent 数量 | 2 个工具分开用 | 多个 Agent 协同，可扩展 |
| 自定义程度 | 有限 | 可自定义 Prompt / 名字 / 头像 |
| 适合人群 | 能翻墙的用户 | 国内用户友好 |

---

### 2.5 延伸思考

1. 视频中的"套娃"和"代码行"是两个独立 Agent 手动协作，而三省六部制项目实现了 Agent 之间的自动协同——这两种模式的边界在哪里？什么时候需要自动协同，什么时候人工中转更合适？

2. 给 Agent 设定专属 Prompt（如"你是一名丰富经验的产品顾问"）对输出质量影响有多大？如何设计和优化这类角色 Prompt？

3. 随着多 Agent 协同系统越来越复杂（如三省六部制），系统的可解释性和可调试性如何保证？当输出结果不符合预期时，如何快速定位是哪个 Agent 的问题？

---

### 2.6 一句话记住它

> 多 Agent 协作不是技巧，是方法论——就像唐朝三省六部制，让每个 Agent 各司其职，比让一个全能 AI 包打天下更可靠、更高效。
