### 2.3 核心概念精讲

#### 概念一：Codex Plugin CC

**是什么**：OpenAI 官方发布的 Claude Code 插件，允许在 Claude Code 环境中直接调用 Codex 的能力。

**为什么重要**：
此前开发者想在 Claude Code 里用 Codex，只能绕道通过 subagent 间接调用，流程繁琐。现在官方插件直接打通，降低了使用门槛。

**详细解释**：
这是 OpenAI 主动为竞品（Claude Code）生态做的插件，说明两家工具之间的互操作性正在增强，开发者可以在同一个工作流里混用不同厂商的 AI 能力。

**实际应用场景**：
在 Claude Code 会话中，无需切换工具或手动调 API，直接唤起 Codex 完成特定任务。

---

#### 概念二：三大功能模式

**是什么**：Codex Plugin CC 提供三种不同的调用模式，对应不同的编程场景。

**为什么重要**：
不同场景下需要 AI 扮演不同角色——有时需要生成代码，有时需要审查设计，有时需要"接盘"一个烂摊子，三种模式各司其职。

**详细解释**：

| 模式 | 名称 | 核心能力 | 适用场景 |
|------|------|----------|----------|
| 模式一 | 标准代码生成（Standard） | 生成代码、修复 bug | 日常开发、快速出代码 |
| 模式二 | 对抗性测试（Adversarial） | 修复 bug + 审查架构设计 | 代码 review、上线前质量把关 |
| 模式三 | 救援模式（Rescue） | 接管代码库、完成后续工作 | bug 死活干不掉、项目陷入僵局时 |

**实际应用场景**：
- 新功能开发 → 用标准模式生成代码
- 准备上线 → 用对抗性测试模式做最后审查
- 卡了很久的 bug → 直接开救援模式，让 Codex 接管收尾

---

#### 概念三：救援模式（Rescue Mode）

**是什么**：当开发者遇到无法解决的 bug 时，Codex 接管整个代码库并自主完成后续工作的特殊模式。

**为什么重要**：
这是三个模式里最激进的一个——它不只是给建议，而是直接"接盘"，对于卡壳的项目有极高的实用价值。

**详细解释**：
救援模式的逻辑是：当你的 Claude Code 会话已经陷入死循环、bug 反复出现无法收敛时，交给 Rescue 模式，它会读取整个代码上下文，重新规划并执行后续步骤。

**实际应用场景**：
用 Claude Code 开发到一半，某个 bug 来回修了十几次还是不对，这时候与其继续消耗 token 和时间，不如直接触发 Rescue，让 Codex 接手。

### 2.5 延伸思考

1. **救援模式的边界在哪里？** Rescue 模式会"接管整个代码库"，那么对于大型项目，它的上下文窗口是否足够？接管后的代码风格和原有代码能保持一致吗？

2. **OpenAI 为什么要给竞品 Claude Code 做插件？** 这背后是商业策略还是生态开放的信号？如果 Codex 能在所有主流 AI 编程工具里被调用，对 OpenAI 意味着什么？

3. **对抗性测试模式能替代人工 Code Review 吗？** 它审查架构设计合理性的深度如何？对于复杂的业务逻辑和领域知识，AI 的对抗性测试和资深工程师的 review 差距在哪里？
