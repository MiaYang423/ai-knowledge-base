## 一、视频原文

很多AI产品经理面试都会被问到Vibe Coding能力，那么作为AI产品经理，我们的Vibe Coding能力到底要达到一个什么样的程度呢？今天我就来详细地展开跟大家讲一下。

首先我们要知道，我们作为AI产品经理，并不是需要去成为一个程序员，而是要把我们的想法快速变成可以验证的产品。所以我们更重要的不是说去替代研发，而是借助AI去完成我们从需求承接、原型实现，到跑通流程，到具体一些单独验证。所以我们真正应该去做的，不是去讨论一些概念，而是去把我们整体的一个产品的想法快速推进到可验证和可讨论的状态。

那么对于AI产品经理从入门到进阶所有的资料我也都准备好了，大家可以老地方来拿。

那么首先第一个，我们需要明确Vibe Coding能力它并不是说去瞎写代码。其次我也会来讲解AI产品经理为什么需要Vibe Coding能力，还有就是我们具体要达到什么样的程度。

那么第一块，Vibe Coding能力并不是说我们去写代码，并不是说去建立一套开发意识，而是我们需要知道一个功能为什么要做、具体的流程怎么设计、最终的结果怎么样去验证它。是由AI帮我们把需求、交互、接口、数据全部去做测试、跑通，而不是说我们只会用这个工具而已。所以我们核心还是要知道它需要给我们解决什么样的问题。

然后进一步，就是为什么我们需要这样的一个能力。首先因为目前AI在整体的研发节奏上已经产生了变化。以前的产品经理它更多是需要交付一些PRD、原型图等等，但是现在很多AI产品的早期阶段，它最大的问题不是说我写的PRD完不完整，而是这个想法到底能不能用、到底靠不靠谱，以及这个想法到底能不能跑、用户它到底能不能用，以及我们模型调用到底靠不靠谱。所以如果我们只停留在文档层面的话，其实很难在短期之内快速得到验证。所以越来越多的团队才会看这种AI产品经理的Vibe Coding能力——你能不能把你的想法快速变成可验证的产品，这个就是核心的能力要求。

那么我们需要达到一个什么样的程度呢？

首先第一层，我们需要能够由AI做出一个可交互的原型形态。比如说我们需要具备基础的交互设计能力，也就是说我们需要对于页面整体的一些设计、图片识别、链表等等这块，有一些真实的前后端的内容展现，并且能够有完整的产品逻辑。

然后进一步，我们能够理解前后端的基本逻辑。我们不仅仅需要做动态的页面，我们还要去观察前后端的一些复杂的内容——它们接口什么、数据变化是怎么传的、状态是怎么变化的等等。所以我们需要不断地去挖掘我们在整个流程当中的一些核心逻辑，这不仅仅是产品逻辑，还有具体的一些交互和连发的逻辑。

然后再进一步，我们需要用AI快速去搭建整体的小demo。所以我们这个demo它不一定需要达到一个生产级别，但它需要能够跑通核心流程。所以我们在具体要去做的时候，也需要去把对应当中最基础的主链路——比如说用户的输入、模型的处理、结果的输出等等——去做完整的分析和设计。然后再进一步去设计一些异常的case、兜底策略等等。

然后我们最后再用工程化的思维去管理我们的AI coding，这也是最重要的一步。很多人啊，用AI写代码最大的一个问题就是一开始特别特别爽，但是后面就全部都乱了——代码乱、交互乱等等。所以我们需要知道的是，我们在整体的设计当中，我们一定要由AI去驱动开发，而不是由AI完全代替开发内容。

所以以上就是AI产品经理Vibe Coding需要达到的一些核心能力，如果说大家有什么其他问题也可以随时来找我咨询。

### 2.3 核心概念精讲

#### 概念一：Vibe Coding（对AI产品经理而言）

**是什么**：AI产品经理借助AI工具，将产品想法快速转化为可交互、可验证原型的工作方式，而非自己手写代码。

**为什么重要**：
面试中越来越多被考察；团队评估AI产品经理的核心标准之一正在从"PRD写得好不好"转向"能不能快速出可验证的东西"。

**详细解释**：

| 常见误解 | 正确理解 |
|---|---|
| Vibe Coding = 学会写代码 | Vibe Coding = 用AI驱动产品验证 |
| 目标是替代研发 | 目标是加速从想法到验证的过程 |
| 只要会用工具就够了 | 核心是懂功能逻辑、流程设计和验证标准 |
| PRD写完整就是交付 | 想法能否跑通才是早期最大问题 |

**实际应用场景**：
早期AI产品立项阶段，不等研发排期，产品经理自己用AI快速搭出一个能跑通主流程的demo，拿去和用户验证或和团队讨论。

---

#### 概念二：可验证原型 vs 传统PRD交付

**是什么**：可验证原型是指能够实际运行、让用户或团队操作体验的产品雏形；传统PRD交付是以文档、原型图为核心的交付方式。

**为什么重要**：
AI产品早期最大的不确定性不是需求写没写清楚，而是"这个想法到底能不能用、模型调用靠不靠谱"，文档无法回答这些问题，只有跑起来才能验证。

**详细解释**：

| 维度 | 传统PRD交付 | 可验证原型 |
|---|---|---|
| 核心产物 | 文档、原型图 | 可运行的demo |
| 验证方式 | 评审会讨论 | 实际操作体验 |
| 验证速度 | 慢，依赖研发排期 | 快，PM自己用AI搭建 |
| 能回答的问题 | 需求是否完整 | 想法是否可行、用户能否使用 |
| 适用阶段 | 需求明确后 | 早期探索阶段 |

**实际应用场景**：
产品经理有一个AI功能新想法，不写PRD，而是直接用Cursor等工具让AI搭出一个能跑的小demo，当天拿给用户试用，第二天拿到反馈再迭代。

---

#### 概念三：主链路优先原则

**是什么**：在用AI搭建demo时，优先保证核心主链路跑通（用户输入→模型处理→结果输出），而非追求功能完整或生产级别的质量。

**为什么重要**：
demo不需要达到生产级别，但必须能跑通核心流程，否则无法完成有效验证；过早追求完整会浪费时间且容易让代码陷入混乱。

**详细解释**：

主链路的基本结构：
```
用户输入 → 模型处理 → 结果输出
    ↓
跑通后再考虑：
- 异常case处理
- 兜底策略设计
- 边界情况覆盖
```

**实际应用场景**：
做一个AI写作助手的demo，先只做"用户输入文本→调用模型→返回改写结果"这条主链路，跑通后再考虑输入为空、网络超时、模型返回异常等情况的处理。

---

#### 概念四：工程化思维管理AI Coding

**是什么**：在用AI写代码的过程中，由人来主导整体架构和开发节奏，AI负责执行具体实现，而不是把所有事情都甩给AI随意生成。

**为什么重要**：
视频明确指出，用AI写代码"一开始特别爽，但后面全部都乱了"是最常见的问题——代码乱、交互乱。工程化思维是避免这个问题的关键。

**详细解释**：

| 做法 | 结果 |
|---|---|
| 完全让AI代替开发，随意生成 | 代码越来越乱，交互混乱，难以维护 |
| 由AI驱动开发，人来管理节奏和结构 | 代码可控，能持续迭代 |

核心原则：**AI驱动开发 ≠ AI完全代替开发**

**实际应用场景**：
每次让AI写一个模块前，先明确这个模块的输入输出和边界，写完后做检查再进行下一个，而不是一次性让AI把整个项目都生成出来。

### 2.5 延伸思考

1. **能力边界在哪里？** 视频提到AI产品经理不需要成为程序员，但又需要理解前后端逻辑、接口和数据传递——这个"理解"的深度应该到哪里？什么情况下产品经理的Vibe Coding能力会开始和初级工程师的能力产生重叠，这个边界应该如何界定？

2. **工程化思维如何落地？** 视频指出"代码越来越乱"是Vibe Coding最大的问题，并给出了"由AI驱动开发而非完全代替"的原则，但具体怎么做才算工程化管理？比如是否需要版本控制、模块拆分、文档记录？产品经理在没有工程背景的情况下如何建立这套意识？

3. **验证之后呢？** Vibe Coding帮助产品经理快速做出可验证原型，但原型验证通过后，这份AI生成的代码是直接交给研发继续迭代，还是推倒重来？AI产品经理用Vibe Coding做出的demo和研发团队的正式代码之间，协作和交接的边界应该怎么划？
