## 一、视频原文

上次做的小调查发现很多新手小白不知道 Vibe Coding 到底应该从哪一步开始，那今天我们就用一分钟时间来想清楚，市面上这些主流的开发工具，哪些是适合非技术人员去进行 Vibe Coding 的工具。我已经准备好了，想要用的小白可以去评论区，评论给大家，直接看张参考图。

那我们可以从两个维度去说。第一个维度是全能型和垂直型的。那全能型的工具呢，他就是可以什么都做，写代码、改 bug、大运行都可以，如果大家什么都要去试一试的话，可以偏全能型的工具。那像垂直型的工具呢，其实他只是专注于某一方面，像我现在就是专门去做界面的，这个工具呢他是专门去做某一块的。那如果我们是卡在某一个集中的步骤，不太擅长或者说不太好，想要去精进的话，那我建议大家可以去找这个专家类型的工具，也就是垂直型的工具。

那第二个维度就是半自动和全自动。那比如说助手型的，就是那种我们点一下可以动一下，控制权还是在我们手里的。那 Agent 型的呢，就是数字员工，我们告诉他我们想干什么，他就可以自己去规划，然后从头到尾一跑通之后，最后直接交给我们产品。

那如果是纯小白的话，推荐这五个工具。第一个呢就是 Cursor，之前我已经说很多遍了，他是目前我觉得最适合小白的，因为他对网络没有任何的要求，他的配置也比较简单，并且他是完全免费的，如果大家最新手想去试一试，改成一下这个逻辑化，是完全够用的。第二个工具呢就是 Lovable，他就是最懂外行人的 UI 搭建神器，那 UI 在一个产品中其实也是非常重要的一个部分。然后呢就是 Bolt，如果我们想做 MVP，就是最小可行性产品，用他就可以直接一步到位。第四个工具呢就是 Vercel，是 Next.js 出品的，就是把我们所有的想法可以变成非常高级的网页前端。第五个呢就是可能需要对他的专业知识能力有一定要求，就是 Claude，我们用他做 Vibe Coding，你就可以把他当成另一个伙伴，我们只跟他边聊边对话，他就直接帮我们把有差别的、非常完美的东西做出来了。当然前提也是基于我们跟他的沟通中，非常有逻辑、条理清晰地把我们的需求列出来，那他给的东西基本上是不太会出错的。

那大家如果有时间的话，可以试一下吧！

### 2.3 核心概念精讲

#### 概念一：全能型 vs 垂直型工具

**是什么**：按工具的功能覆盖范围划分的两类工具。

**为什么重要**：选错工具类型会导致学习成本高、效率低，明确自己的需求才能选对工具。

**详细解释**：

| 类型 | 能做什么 | 适合谁 |
|------|----------|--------|
| 全能型 | 写代码、改 bug、运行部署都能做 | 想全面尝试 Vibe Coding 的新手 |
| 垂直型 | 只专注某一个环节，如界面设计或测试 | 卡在某一步、想针对性突破的用户 |

**实际应用场景**：如果你完全不知道从哪里下手，先用全能型工具（如 Cursor）整体跑通流程；如果你已经能大致开发但界面很丑，就用垂直型工具（如 Lovable）专门打磨 UI。

---

#### 概念二：半自动（助手型）vs 全自动（Agent 型）

**是什么**：按工具的自动化程度划分的两类工作模式。

**为什么重要**：决定了你在开发过程中需要投入多少精力和判断，也影响出错时你能否及时介入。

**详细解释**：

| 类型 | 工作方式 | 控制权 | 适合场景 |
|------|----------|--------|----------|
| 半自动（助手型） | 用户指令 → 工具执行一步 → 用户再指令 | 在用户手里 | 想边学边做、保持掌控感 |
| 全自动（Agent 型） | 用户说目标 → 工具自动规划并跑通全流程 → 交付产品 | 在工具手里 | 想快速出产品、信任工具能力 |

**实际应用场景**：做第一个产品时建议用半自动模式，能看清楚每一步在干什么；有经验之后可以用 Agent 型工具提升效率。

---

#### 概念三：MVP（最小可行性产品）

**是什么**：用最少的功能验证一个产品想法是否可行的原型版本。

**为什么重要**：非技术人员资源有限，先做 MVP 可以快速验证方向，避免在错误的产品上浪费时间。

**详细解释**：MVP 不追求完美，只追求"能跑、能用、能验证核心假设"。视频中提到 Bolt 就是专门为快速做出 MVP 设计的工具，一步到位。

**实际应用场景**：你有一个 App 想法，先用 Bolt 做出能点击的基础版本，拿给真实用户试用，收到反馈后再决定要不要继续深做。

---

#### 概念四：用 Claude 做对话式开发

**是什么**：把 Claude 当成开发伙伴，通过持续对话描述需求，让它直接生成代码或产品。

**为什么重要**：对于不会写代码的人，"聊天"是最低门槛的交互方式，但质量高度依赖用户的表达能力。

**详细解释**：视频特别强调了一个前提条件——用户需要**逻辑清晰、条理分明**地描述需求，Claude 才能给出准确的结果。这意味着 Claude 对用户的"需求整理能力"有一定要求，不适合完全没想清楚就开始聊的情况。

**实际应用场景**：在动手之前，先把你想做的产品的功能、用户、流程写成一份清单，然后带着这份清单去和 Claude 对话，效果会好很多。

### 2.5 延伸思考

1. **视频推荐 Cursor 完全免费且对小白最友好，但真实使用中免费版有哪些限制？当项目复杂度提升后，Cursor 还够用吗？**

2. **Claude 做 Vibe Coding 的前提是"逻辑清晰、条理分明地描述需求"——对于非技术人员来说，如何训练自己把模糊的产品想法整理成 Claude 能读懂的需求？**

3. **全自动 Agent 型工具听起来很省力，但控制权完全交给工具，如果中途出错或方向跑偏，非技术人员有能力发现并纠正吗？这种风险该怎么管理？**

---

### 2.6 一句话记住它

> 选 Vibe Coding 工具前先想清楚两件事：你要做什么（全能 vs 垂直），你想管多少（半自动 vs 全自动）——想清楚了，工具自然就选对了。
