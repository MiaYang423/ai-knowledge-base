今天想跟大家分享一下 Vibe Coding 是如何帮我这一个没有开发背景的人打开了三个技术界的新世界大门的。

第一个是关于 GitHub 和开源。虽然这两个词我以前就听过，但是我从来都没有用过，因为我觉得它们离我的世界真的是太远了，只有很专业的那种开发人员才会去用它们。但是当我开始 Vibe Coding 之后，我必须去学如何用 GitHub 的时候，我才发现这个东西没有我想象的复杂。并且因为我学过了一些基础的 GitHub，包括如何去控制一个 repo，所以现在我看到网上一些开源的好玩的东西我可以直接去用。例如最近很火的 G-Stack，还有 Mirofish，还有 OpenClaw，这些都是人人都可以去 GitHub 上克隆下来并且尝试一下的。感觉 GitHub 上有这么多好玩的东西，感觉我以前大概是错过了一个宇宙吧。当然当我开始了解开源社区之后，我也很开心地去参与，并且也把我的一些项目放到了 GitHub 上来开源给大家。

第二个就是 Markdown 文档，所谓的 `.md` 文件。在 Vibe Coding 之前我从来没有听说过 Markdown 或者 `.md` 文档，但是现在我知道了，它就是一个文字文档加上一点点格式而已，并没有什么复杂的地方。当我们开始一个新项目的时候，我们可能会写一个 `project.md` 或者是 `Claude.md`，把我们所有的产品需求都写在里面。我们现在听说的各种 Skills，它们最主要最核心的文档也是一个 `.md`。还有如果你有 OpenClaude 的话，你知道它有 `CLAUDE.md`、有 `agents.md`、有 `memory.md`，这些都是你可以打开它直接读的。而且还有像 Claude Code 这样的工具，它所有的文档都是以 `.md` 的方式来写的。

第三个就是听得懂并且可以参与技术型对话了。以前学 Vibe Coding 我一直摸不透我具体都学到了什么东西、什么知识，感觉就是一边在尝试一边在玩。但是我最近发现我不知不觉还是学到了很多技术知识。有时候最近我跟一个团队来讨论知识管理，我就想到了 OpenClaude 的 `memory.md` 还有下面的每天的 Daily Log。当我们讨论到语音助理不用保存录音的时候，就想到了我用了好多个月的 Granola，只是把当下的信息都提取出来，然后不会保留任何录音。再或者当我们讨论如何保存一些敏感信息的时候，我听得懂 `.gitignore`，听得懂 Environment Variable。所以虽然我没有意识到我具体都学到了哪些内容和单词或者是概念，但是在跟别人的沟通和交流中，我发现我在前一段时间中确实累积了很多新的知识。

所以这个让我更坚定地想继续一直不停地用新的 AI 产品、新的 AI 科技，和尝试自己的 Vibe Coding 项目。

### 2.3 核心概念精讲

#### 概念一：GitHub 与开源社区

**是什么**：GitHub 是一个代码托管平台，开源是指代码公开、人人可用的软件开发模式。

**为什么重要**：
大量优质 AI 工具和项目（如 G-Stack、Mirofish、OpenClaude）都托管在 GitHub 上，能用 GitHub 意味着能直接获取和使用这些资源，而不必等待别人打包好的版本。

**详细解释**：

| 操作 | 含义 | 难度感知（实践前→后） |
|------|------|----------------------|
| 克隆 repo | 把别人的项目下载到本地 | 很难 → 其实不复杂 |
| 开源自己的项目 | 把自己的代码公开到 GitHub | 专业行为 → 普通人也能参与 |
| 浏览开源项目 | 发现并尝试社区里的新工具 | 看不懂 → 能直接上手 |

**实际应用场景**：
看到网上推荐的开源 AI 工具时，可以直接去 GitHub 克隆下来本地运行，而不是只能干等官方发布安装包。

---

#### 概念二：Markdown 文档（.md 文件）

**是什么**：一种轻量级文本格式，用极少的符号（如 `#`、`-`、`**`）给纯文字加上结构和格式，文件后缀为 `.md`。

**为什么重要**：
几乎所有主流 AI 开发工具的核心配置和文档都用 `.md` 格式书写，读懂 `.md` 文件等于读懂工具的"说明书"。

**详细解释**：

| 文件名 | 用途 |
|--------|------|
| `project.md` | 写清楚一个项目的所有产品需求 |
| `CLAUDE.md` | 告诉 Claude 这个项目的背景和规则 |
| `agents.md` | 定义 AI 智能体的行为方式 |
| `memory.md` | 存储 AI 的长期记忆和上下文 |
| `Daily Log.md` | 每日知识/进度记录 |

**实际应用场景**：
用 Claude Code 或 OpenClaude 时，直接打开 `CLAUDE.md` 就能理解这个工具在这个项目里的角色设定，不需要靠猜。

---

#### 概念三：在实践中自然习得技术词汇

**是什么**：通过长期动手使用 AI 工具和 Vibe Coding，在不刻意记忆的情况下，把技术概念内化为真实理解。

**为什么重要**：
传统学编程的方式是"先学后用"，容易放弃。Vibe Coding 的路径是"先用后懂"，技术词汇在真实场景中反复出现，自然就记住了，而且记住的是"用法"而不是"定义"。

**详细解释**：

视频中提到的具体案例：

- **讨论知识管理** → 自然联想到 `memory.md` 和 Daily Log 的设计逻辑
- **讨论语音助理隐私** → 联想到 Granola 工具"只提取信息、不保留录音"的机制
- **讨论敏感信息存储** → 听懂了 `.gitignore`（告诉 Git 忽略某些文件）和 Environment Variable（把密钥存在环境变量里而不是代码里）

**实际应用场景**：
在和技术团队开会时，不会因为听不懂术语而脱节，甚至能用自己的实践经验参与讨论。

### 2.5 延伸思考

1. **"先用后懂"和"先学后用"哪种路径更适合非技术背景的人入门 AI 工具？** 视频里的经历说明实践可以驱动学习，但这种方式会不会留下知识盲区，在某个阶段需要系统补课？

2. **Markdown 作为"人类可读的配置语言"正在成为 AI 工具的标准界面，这意味着什么？** 如果越来越多的 AI 工具用 `.md` 文件来定义行为，普通用户和开发者之间的边界是否正在消失？

3. **视频里说"也许过程比结果更重要"，在 Vibe Coding 的语境下，这个"过程"具体积累的是什么？** 是可迁移的技术素养，还是只对特定工具有效的操作经验？两者有什么区别？
