## 一、视频原文

她叫 Claudio。她的前端就是一个播放器和简洁的界面，我的 Claude Code 和音乐 API，一会儿我讲一下原理。你可以把它理解成一个24小时在线的 AI 电台主播。

可以给大家看一下它的主脑——这应该算是我的音乐品味本体吧。我把最近14年里不同平台上的三十多个歌单，每个阶段的音乐回忆都结构化地喂给了它。它可能比我自己都懂我爱听什么歌。

除了这些音乐数据，它还可以根据每天的天气、日程、心情给我定制歌单。比如说今天周一，我们来看看它做了什么：早上的时候它连接了我的智能音响，给我放的是轻柔舒缓的歌，因为它知道早上我一般爱听粤语歌；上午推荐的都是纯音乐，因为我工作的时候不怎么听有歌词的歌。它就像贴身管家一样，会提前规划我今天一整天的音乐体验，也会自己判断什么时候该说点话。

当然这些歌单也不是完全固定的，因为在聊天界面我可以随时跟它说话，它也会随时响应我。比如说今天晚上我就跟它讨论了一下做内容时选一个 BGM，它推荐了这五首歌，然后我听到其中一首 EVE 的时候觉得推荐得很准，它还给我介绍了一下这首歌的特点。

你可能会好奇，为什么我会给自己做一个"听音乐"这件事的专属 AI？最近两个月我基本上每天都是和 Claude Code 多线程协作，就是同时开好多会话一起跑。刚开始的时候很爽，但时间长了会感觉这种不断切换任务的工作方式非常耗注意力，时间长了可能会把脑子搞废。所以我就觉得得做点"反消耗"的东西。

现在我就一直把它放在我的侧屏幕，在等 Claude Code 跑代码的时候，我就可以和我的 Claudio 聊聊音乐、听听歌，让自己不用一直处于那么紧绷的状态。

我一直都想用 AI 帮我们放大对美好事物的感受。最近我常常想到 Andrej Karpathy 在2月份发的那条推特，他说：所有的传统数据都可以在 AI 时代变成一个接口，让大语言模型原生且轻松地使用它们。我觉得这会是一个更丰富的世界——我们可以排列组合手头的各种设备和数据，比如音响、灯光、音乐数据等等，去创造一些原来没有的体验。Claudio 就是我在这个小方向里先自己试着做出来的一小块东西。

如果你也想做这样一个音乐 AI Agent，原理其实很简单，大致分为三个部分：第一个部分是前端的播放器界面；它跑在我本地搭的服务器上；主要是连了几个 API 让它们协同工作。你可以截图把这个结构和这个示意图发给 Claude，它基本上一看都懂。你需要配合它拿到几个 API Token 和权限。

很期待大家也能做一个符合自己音乐喜好的 AI Agent。那在这期视频的结尾，我想回到音乐，和 Claudio 一起给大家推荐一首歌。

### 2.3 核心概念精讲

#### 概念一：音乐品味本体（个人数据结构化喂给模型）

**是什么**：把自己多年积累的歌单数据，按结构化方式整理后输入给 AI，让模型真正"理解"你的音乐品味，而不只是随机推荐。

**为什么重要**：
AI 的推荐质量取决于它对你的了解程度。用自己的真实历史数据训练上下文，远比用通用推荐算法更贴近个人偏好。

**详细解释**：

| 数据维度 | 具体内容 |
|---|---|
| 时间跨度 | 14年 |
| 数据来源 | 多个音乐平台 |
| 数据量 | 30+ 个歌单 |
| 组织方式 | 按人生阶段结构化整理 |
| 喂给对象 | Claude（作为长上下文/系统提示） |

**实际应用场景**：
早上 Claudio 自动播放粤语歌（因为它从歌单里学到作者早晨的偏好），工作时切换纯音乐，晚上能根据聊天内容推荐合适的 BGM。

---

#### 概念二：AI Agent 的"反消耗"设计思路

**是什么**：专门用来对抗高强度 AI 协作带来的注意力损耗，而不是追求更高生产力的 AI 应用设计方向。

**为什么重要**：
当我们大量使用 AI 提升效率时，多线程、不断切换任务本身会带来认知负担。作者提出，AI 也可以反过来做"减压阀"，而不只是"加速器"。

**详细解释**：
- 作者每天同时开多个 Claude Code 会话协作（多线程 VibeCoding）
- 等待 AI 跑代码的碎片时间，原本是"空转焦虑"
- Claudio 把这段时间变成：听歌 + 和 AI 聊音乐 → 主动切换到低耗能状态
- 放在侧屏幕，不需要主动操作，降低切换成本

**实际应用场景**：
等 Claude Code 输出结果时，侧屏幕的 Claudio 自动播放音乐，作者可以短暂脱离紧绷状态，而不是刷手机或继续盯着代码。

---

#### 概念三：传统数据变成 AI 接口（Karpathy 的观点）

**是什么**：Andrej Karpathy 在2025年2月提出的观点——在 AI 时代，所有传统数据（歌单、日历、健康数据等）都可以变成大语言模型可以原生调用的接口。

**为什么重要**：
这个观点给出了一个框架：我们手边已有的数据和设备，不需要等大公司整合，个人就可以通过 VibeCoding 把它们连接起来，创造新体验。

**详细解释**：

| 传统形态 | AI 时代的接口形态 |
|---|---|
| 歌单（静态列表） | AI 可读取并推理的音乐品味数据库 |
| 智能音响（独立设备） | 被 AI Agent 调度的播放终端 |
| 日历/天气（工具数据） | 影响 AI 决策的上下文输入 |
| 聊天记录（碎片信息） | AI 实时响应的情绪/需求信号 |

**实际应用场景**：
Claudio 就是这个理念的一个小实验：歌单数据 + 音响设备 + 天气日程 + 聊天交互，四类"传统数据"被 AI 统一调度，产生了原来不存在的"个人电台"体验。

### 2.5 延伸思考

1. **"反消耗"AI 应用还有哪些可能？** 作者用音乐对抗多线程协作的注意力损耗，那么其他高强度 AI 使用场景（如长时间提示词调试、反复迭代设计稿）是否也需要类似的"缓冲层"？这类应用和生产力工具在设计逻辑上有什么本质区别？

2. **个人数据的"蒸馏"边界在哪里？** Claudio 用14年歌单来理解音乐品味，那么如果把日记、购物记录、浏览历史也结构化喂给模型，AI 对"你是谁"的理解会更准确，还是会产生新的问题？个人数据蒸馏的收益和风险如何权衡？

3. **Karpathy 说的"传统数据变成接口"，普通人能做到什么程度？** 作者用 VibeCoding 实现了音乐+音响+日程的联动，但更复杂的数据（如健康数据、财务数据）要变成 AI 接口，技术门槛和数据安全门槛是否还在普通人可触达的范围内？
