## 一、视频原文

在构建多 Agent 的协同架构时，任务分发是一个核心环节。你如何评价基于 LLM 的语义路由与基于固定规则的确定性路由？在实际复杂场景中你又是如何权衡和选型的？

好的，面试官，这是一个工程实际的问题。在分布式 Agent 架构中，路由层的设计直接决定了系统的鲁棒性与生产性。

先说我的核心观点：固定规则决定系统的下线，而 LLM 路由决定系统的上线。在生产级环境下，我们追求的既不是纯粹的灵活性，也不是死板的稳定性，而是确定性逻辑为主、语义路由为补充的协同治理。

我们要从底层信息论的角度来看。固定规则路由本质上是对业务逻辑的结构化投影，它要求输入特征空间是必急的，追求的是低延迟和幂等性。LLM 路由本质上是基于高维向量空间的语义映射，它利用了大模型的零样本学习能力去处理开放条件下的意图泛化问题，它解决的是从非结构化输入到原则化路由能力的对齐。

在实战中，两者的差异主要体现在三个核心维度：

稳定性与一致性：规则分发不存在抖动，但在应对语义模糊时会失效；LLM 路由具备极强的推理泛化能力，但存在随机性。

系统工程商职：规则越多，系统越碎、维护成本成指数级上升；LLM 路由将逻辑收敛到 Prompt 中，降低了代码耦合，但增加了调优复杂度。

计算效能：规则路由是常数级或对数级的计算开销，而 LLM 路由引入了显著的推理延迟和 Token 成本。

在实际大厂落地实践中，我推崇的是三层漏斗的分发模型：

第一层，静态匹配层：利用正则或关键词哈希快速拦截高频标准化指令，要减少确定性强的通过。

第二层，通过 Vector 设计，将用户请求与 Agent 的能力描述做余弦相似度计算，这是一种高性能的准语义路由。

第三层，知识角色层：当前两层置信度不足时，才调用 LLM，通过一个专门的调度 Agent，结合上下文信息进行最终裁决。这里一定要做 Agent 的 Retry 水动态维护各个 Agent 的元数据，让 LLM 路由有据可依。

最后总结一下我的技术哲学：规则是系统的骨架，保证它不崩；LLM 是系统的神经，让它产生感知。一个优秀的 Agent 开发工程师，不应该试图用 LLM 去替代所有逻辑，而应该用 LLM 去覆盖那些传统逻辑覆盖不到的边界。

最后我也整理了一份某大厂要求 AI 产品必须了解的文档，里面就包含了整个 AI 产品的研发流程、大模型的未来发展方向以及 AI 产品当前存在的一些问题，想要的我可以分享给你。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**多 Agent 路由架构中，固定规则保下线稳定、LLM 语义路由提上线能力，最优解是"静态匹配→向量相似→LLM 裁决"的三层漏斗模型。**

---

### 2.2 知识地图

```
多 Agent 路由架构选型
├── 核心观点
│   ├── 固定规则 → 决定系统下线（稳定性保障）
│   └── LLM 路由 → 决定系统上线（泛化能力上限）
├── 两种路由方式对比
│   ├── 固定规则路由
│   │   ├── 本质：业务逻辑的结构化投影
│   │   ├── 优点：低延迟、幂等、无随机性
│   │   └── 缺点：语义模糊时失效，规则多则维护成本指数级上升
│   └── LLM 语义路由
│       ├── 本质：高维向量空间的语义映射
│       ├── 优点：零样本泛化，处理开放意图
│       └── 缺点：有随机性，推理延迟高，Token 成本高
├── 三个核心对比维度
│   ├── 稳定性与一致性
│   ├── 系统工程维护成本
│   └── 计算效能（延迟 + 成本）
└── 三层漏斗分发模型（最优实践）
    ├── 第一层：静态匹配（正则/关键词哈希，拦截高频标准指令）
    ├── 第二层：向量相似度（余弦相似度，准语义路由）
    └── 第三层：LLM 裁决（调度 Agent + 上下文 + Agent 元数据）
```

---

### 2.3 核心概念精讲

#### 固定规则路由 vs LLM 语义路由

**是什么**：两种完全不同的 Agent 任务分发策略——前者用确定性规则（正则/关键词）匹配，后者用大模型理解语义后路由。

**为什么重要**：路由层的设计直接决定整个多 Agent 系统的鲁棒性和生产可用性，选错了要么系统不稳定，要么无法处理复杂意图。

**详细解释**：

| 对比维度 | 固定规则路由 | LLM 语义路由 |
|---|---|---|
| 本质 | 业务逻辑的结构化投影 | 高维向量空间的语义映射 |
| 稳定性 | 无抖动，确定性强 | 有随机性 |
| 泛化能力 | 弱（语义模糊时失效） | 强（零样本学习，处理开放意图） |
| 延迟 | 常数级/对数级 | 高（推理延迟） |
| 成本 | 低 | 高（Token 成本） |
| 维护成本 | 规则越多越难维护（指数级上升） | 集中在 Prompt 调优 |

**实际应用场景**：
- 固定规则：用户说"查订单"→直接路由到订单查询 Agent（高频、明确）
- LLM 语义：用户说"我上周买的那个东西到哪了"→LLM 理解意图后路由到物流追踪 Agent

---

#### 三层漏斗分发模型

**是什么**：大厂落地实践中推荐的多 Agent 路由架构——静态匹配→向量相似度→LLM 裁决，三层依次兜底，兼顾效率和泛化能力。

**为什么重要**：纯规则路由无法处理复杂意图，纯 LLM 路由成本高且不稳定；三层漏斗让高频简单任务走最快的路，只有真正复杂的任务才调用 LLM。

**详细解释**：

| 层次 | 技术方案 | 处理对象 | 特点 |
|---|---|---|---|
| 第一层：静态匹配 | 正则 / 关键词哈希 | 高频标准化指令 | 最快，零延迟，确定性强 |
| 第二层：向量相似度 | 余弦相似度（用户请求 vs Agent 能力描述） | 中等复杂度意图 | 高性能准语义路由 |
| 第三层：LLM 裁决 | 调度 Agent + 上下文 + Agent 元数据 | 前两层置信度不足的复杂意图 | 最强泛化，成本最高，作为兜底 |

**实际应用场景**：
- 第一层拦截："帮我查余额"→余额查询 Agent（直接命中关键词）
- 第二层路由："我想了解一下我的账户情况"→向量相似度匹配到账户管理 Agent
- 第三层裁决："上个月我花了多少，感觉有点多，能不能帮我分析一下"→LLM 理解复杂意图后分发给账单分析 Agent

---

#### 技术哲学：规则是骨架，LLM 是神经

**是什么**：多 Agent 系统设计的核心原则——规则保稳定性下限，LLM 提泛化能力上限，两者不是替代关系而是协同关系。

**为什么重要**：很多工程师走极端，要么全用规则（上线能力有限），要么全用 LLM（系统不稳定且成本高）；这个原则给出了正确的协同框架。

**实际应用场景**：
- 支付场景：核心支付链路用固定规则（绝对不能出错）
- 客服场景：标准问题用规则，复杂投诉用 LLM（需要理解情绪和上下文）
- 原则：用 LLM 覆盖传统逻辑覆盖不到的边界，而不是替代所有逻辑

| 对比维度 | 固定规则路由 | LLM 语义路由 |
|---|---|---|
| 延迟 | 常数/对数级（极快） | 高（推理延迟） |

| 适用场景 | 高频标准化指令 | 开放意图、复杂语义 |

| 三层漏斗 | 技术 | 适用对象 |
|---|---|---|
| 第一层 | 正则/关键词哈希 | 高频标准化 |
| 第二层 | 向量余弦相似度 | 中等复杂度 |
| 第三层 | LLM 调度 Agent | 复杂开放意图 |

### 2.5 延伸思考

1. 三层漏斗模型中，第一层和第二层的"置信度阈值"如何设定？阈值过高会导致太多请求落到 LLM 层（成本高），阈值过低会导致误路由（准确率低），实际工程中如何动态调整？
2. 视频中提到"要做 Agent 的 Retry 水动态维护各个 Agent 的元数据"，如果某个 Agent 频繁失败，路由系统应该如何自动降级或切换到备用 Agent？
3. 随着 LLM 推理成本持续下降，固定规则路由的成本优势会逐渐缩小，未来是否会出现"全 LLM 路由"的可行方案？固定规则路由会消失吗？
