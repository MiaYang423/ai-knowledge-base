第一道题，AI Agent 和 Function Calling 的区别是什么？Function Calling 是模型调用工具的一种能力，AI Agent 是围绕目标完成任务的系统，可能包含规划、记忆、工具调用、知识库和反馈循环。简单说，Function Calling 是零件，AI Agent 是整套执行流程。

第二道题，AI coding 工具如何辅助开发 AI Agent？它可以帮你生成工具 Schema、写编排样例、补测试、解释框架代码和排查错误。但它不理解完整业务逻辑，所以核心流程、权限、安全和异常处理仍然要人工把关。提效可以，甩锅不行。

第三道题，LangChain、LangGraph 等 Agent 框架适合什么场景？LangChain 组件丰富，适合快速搭建 RAG 和工具调用；LangGraph 更适合有状态、可分支、可回放的 Agent 工作流。真正落地时要看团队是否需要框架生态，还是更需要可控和可观测。

第四道题，AI Agent 落地时常用哪些云服务能力？常用模型服务、向量数据库、对象存储、消息队列、函数计算、日志监控、权限管理和内容安全。Agent 不是一个模型接口，线上还要有弹性、审计、告警、灰度和成本控制。

第五道题，常见大模型推理引擎有哪些？推理关注哪些指标？常见有 vLLM、TensorRT-LLM、TGI 和 SGLang。推理指标主要看吞吐、首 token 延迟、总延迟、显存占用、并发能力和模型兼容性。不同业务对吞吐和延迟的权重不一样。

第六道题，为什么需要 PD 分离？业界方案通常如何实现？PD 分离就是把 Prefill 和 Decode 两个阶段拆开优化。Prefill 更吃算力和带宽，Decode 更关注持续生成和缓存管理。分离后可以独立扩缩容、提高资源利用率。常见方案是分层调度和 KV Cache 管理。

第七道题，AI Agent 单独落地的技术挑战和发展趋势是什么？挑战是稳定性、成本、可解释性、权限安全、评测困难和长任务可靠性。趋势是更强的工具生态、更标准的协议、更细的模型路由，以及从聊天框走向业务流程自动化。

好，这套面经我们就分享到这里。这套题表面看是在问 Agent 的框架、云服务和推理引擎，其实真正考的是解决方案落地能力。希望在评论区看到你的答案。

### 2.3 核心概念精讲

#### 概念一：Function Calling vs AI Agent

**是什么**：Function Calling 是大模型调用外部工具的单一能力；AI Agent 是以目标为导向、整合多种能力的完整任务执行系统。

**为什么重要**：面试中混淆两者是最常见的失分点，也是判断候选人是否真正理解 Agent 架构的关键。

**详细解释**：

| 维度 | Function Calling | AI Agent |
|------|-----------------|----------|
| 定位 | 零件 / 单一能力 | 整套执行流程 |
| 组成 | 模型 + 工具调用接口 | 规划 + 记忆 + 工具调用 + 知识库 + 反馈循环 |
| 目标导向 | 否（执行单次调用） | 是（围绕目标持续推进） |
| 典型场景 | 查天气、调 API | 自动完成多步骤复杂任务 |

**实际应用场景**：回答"你们系统是 Agent 还是只用了 Function Calling"时，能清晰说明两者边界，体现架构认知深度。

---

#### 概念二：Agent 框架选型（LangChain vs LangGraph）

**是什么**：LangChain 和 LangGraph 是两种主流 Agent 开发框架，适用场景不同。

**为什么重要**：框架选型直接影响开发效率、系统可控性和后期维护成本，是落地方案设计的核心决策之一。

**详细解释**：

| 维度 | LangChain | LangGraph |
|------|-----------|-----------|
| 核心优势 | 组件丰富、生态完善 | 有状态、可分支、可回放 |
| 适合场景 | 快速搭建 RAG、工具调用 | 复杂工作流、需要状态管理 |
| 选型关键 | 看重开发速度和生态 | 看重可控性和可观测性 |

**实际应用场景**：面试被问"为什么选这个框架"时，不能只说"文档多、社区活跃"，要结合业务是否需要状态管理、分支逻辑来回答。

---

#### 概念三：推理引擎与 PD 分离

**是什么**：推理引擎是部署大模型进行在线推理的服务化框架；PD 分离是将推理过程中的 Prefill（预填充）和 Decode（逐 token 生成）两个阶段拆开独立优化的架构方案。

**为什么重要**：这是大模型工程化部署的核心性能优化手段，也是阿里等大厂面试中考察候选人工程深度的高频题。

**详细解释**：

常见推理引擎对比：

| 引擎 | 特点 |
|------|------|
| vLLM | 高吞吐、PagedAttention、社区活跃 |
| TensorRT-LLM | NVIDIA 官方、推理性能极致优化 |
| TGI | HuggingFace 出品、易集成 |
| SGLang | 结构化生成、高并发场景表现好 |

推理核心指标：

| 指标 | 说明 |
|------|------|
| 吞吐（Throughput） | 单位时间处理的请求数 |
| 首 token 延迟（TTFT） | 用户发请求到收到第一个 token 的时间 |
| 总延迟 | 完整响应耗时 |
| 显存占用 | 影响可部署的模型规模 |
| 并发能力 | 同时处理多少请求 |
| 模型兼容性 | 支持哪些模型架构 |

PD 分离核心逻辑：
- **Prefill 阶段**：处理输入 prompt，计算量大、吃算力和带宽
- **Decode 阶段**：逐 token 生成，关注持续生成速度和 KV Cache 管理
- **分离收益**：两阶段可独立扩缩容，资源利用率更高

**实际应用场景**：高并发在线服务场景下，通过 PD 分离 + KV Cache 管理，可以在不增加硬件的情况下显著提升系统吞吐。

---

#### 概念四：AI Agent 落地的云服务体系

**是什么**：Agent 上线不只是调一个模型接口，而是需要一套完整的云服务能力支撑。

**为什么重要**：很多候选人只会讲模型调用，忽略工程化支撑，这是落地能力薄弱的典型表现。

**详细解释**：

| 云服务类型 | 作用 |
|-----------|------|
| 模型服务 | 提供推理接口 |
| 向量数据库 | 支持 RAG 知识检索 |
| 对象存储 | 存储文件、历史记录 |
| 消息队列 | 异步任务解耦 |
| 函数计算 | 弹性执行工具调用 |
| 日志监控 | 可观测性、排查问题 |
| 权限管理 | 控制 Agent 操作边界 |
| 内容安全 | 过滤有害输入输出 |

线上还需要具备：弹性伸缩、审计追踪、告警机制、灰度发布、成本控制。

**实际应用场景**：设计 Agent 系统架构时，能主动提到这些支撑层，是区分"能聊概念"和"能做落地"的关键信号。

### 2.5 延伸思考

1. **边界在哪里**：AI coding 工具"提效可以，甩锅不行"——在你的实际项目中，哪些环节是 AI 工具真正帮不上忙、必须人工介入的？如何在团队中建立清晰的人机协作边界？

2. **框架 vs 自研**：视频提到落地时要看团队"是否需要框架生态还是更需要可控和可观测"——什么规模的团队、什么类型的业务，值得放弃 LangChain 的生态去自研 Agent 调度层？

3. **从聊天框到业务流程**：视频把"从聊天框走向业务流程自动化"列为 Agent 的核心趋势——这个转变在工程上意味着什么？权限管理、异常处理、人工审核节点应该如何设计才能让 Agent 真正可信任地接管业务流程？
