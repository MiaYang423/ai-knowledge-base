# 教案：LLM Wiki 替代 RAG 构建个人知识库

> 来源视频：[抖音博主 · 抖音](https://www.douyin.com/video/7629015871251287347)
> 适合人群：有一定 AI 工具使用经验、正在搭建或考虑搭建个人知识库的开发者和研究者

---

## 一、视频原文

放弃 RAG，LLM Wiki 才是个人知识库的正确方向。

OpenAI 前科学家 Andrej Karpathy 亲自下场说了一句很扎心的话：很多人搭知识库的思路从根上就错了。今天这期我们就来把他的核心观点彻底讲清楚。

先说现状。一提到 AI 知识库，大多数人第一反应就是 RAG——把文档上传、切片、向量化、存进向量库，用户提问的时候召回相关片段，再塞进上下文生成答案。这套流程能用，而且确实是目前最主流的方案。

但 Karpathy 指出它有三个本质缺陷。

第一，没有知识积累。不管你问多少次，LLM 每次都在零时拼凑片段，永远不会真正学会什么，相当于一个每次都会失忆的助手。

第二，跨文档推理很弱。向量检索靠相似度，召回的是零散段落，一旦问题需要跨多篇文档综合推理，逻辑很容易断裂。

第三，工程链路太重。向量库、嵌入、重排序，对个人开发者来说维护成本极高。

那 Karpathy 提出了什么替代方案？他叫它 LLM Wiki。核心思想就一句话：把 LLM 当成编辑而不是检索工具，让它把你的原始资料一次性编辑成一个结构化、可维护的个人 Wiki，后续反复复用，不用每次重新检索。

整个系统只有三个文件夹。Raw，存放原始资料，只进不出，是知识的源头。Wiki，是 LLM 自动维护的结构化知识库，全部是 Markdown 文件。Templates，存放模板和提示词，规范生成逻辑。就这三个文件夹，极简。

具体工作流分四步。

第一步，Ingest 导入。看到好文章、好资料，直接丢进 Raw，不用手动分类整理，越快越好。

第二步，Compile 编译。让 LLM 读取新内容，提炼关键概念，撰写词条，建立页面间的链接，更新总索引。这一步只做一次，后续反复复用。

第三步，Query 查询。提问时直接从 Wiki 里读取结构化内容回答，不再反复扫描原始文档。

第四步，Lint 校验。定期让 LLM 扫描整个 Wiki，自动处理信息不一致、孤立无链接的页面、过时内容。

整个过程没有向量库，没有 Embedding，没有复杂检索。

拍脑袋来看两者的直接对比：在知识积累上，RAG 每次零时拼凑，LLM Wiki 持续沉淀，形成复利；可读性上，RAG 是黑盒向量库，LLM Wiki 全部 Markdown，全透明；工程复杂度上，RAG 需要向量库加重排序，LLM Wiki 就是文件夹加 LLM；跨文档推理上，RAG 碎片召回容易断裂，LLM Wiki 结构化上下文更连贯；个人可落地性上，RAG 较重成本高，LLM Wiki 极轻，开箱即用。对个人开发者和研究员来说，这个差距是非常明显的。

Karpathy 自己的工具栈也非常接地气，完全可以照搬。本地文件夹，纯文本，用 Obsidian 做知识 IDE，利用它的反向链接、知识图谱和 Dataview 查询；LLM 用 Claude 这类大上下文模型；Prompt 模板大幅降低 Token 消耗；用 Git 做版本管理，记录所有 Wiki 修改。

一天的流程是：网页收藏存入 Raw，执行编译更新 Wiki，从 Wiki 直接获取答案，把好的解释保存为新词条，最后定期让 LLM 自检修复。

总结一下，Karpathy 这波分享本质是在纠正一个行业惯性——一提到知识库就 RAG，但 RAG 不一定是最优解。LLM Wiki 给我们提供了一条更简单、更接近人类学习方式的路径：不是每次都临时翻书找答案，而是先把知识整理成体系，再基于体系回答问题。知识库会越用越聪明，真正形成复利。

用 Karpathy 自己的比喻来结尾：Obsidian 是知识 IDE，LLM 是程序员，Wiki 是代码库。不需要花里胡哨的技术，只需要一个持续生长的知识体系。

### 2.3 核心概念精讲

#### 概念一：RAG 的本质缺陷

**是什么**：RAG（检索增强生成）是目前最主流的知识库方案，核心流程是"切片→向量化→存库→检索→生成"。

**为什么重要**：理解 RAG 的缺陷，才能理解为什么 Karpathy 要提出替代方案。

**详细解释**：

| 缺陷 | 具体表现 | 后果 |
|------|----------|------|
| 没有知识积累 | 每次提问都从零拼凑片段 | LLM 永远不会"真正学会"，相当于失忆助手 |
| 跨文档推理弱 | 向量检索靠相似度，召回零散段落 | 需要综合多篇文档时逻辑容易断裂 |
| 工程链路太重 | 需要向量库、Embedding、重排序 | 个人开发者维护成本极高 |

**实际应用场景**：当你用 RAG 搭建的知识库回答复杂问题时，经常出现答案片面或前后矛盾，根本原因就在于此。

---

#### 概念二：LLM Wiki 的核心思想

**是什么**：把 LLM 当成"编辑"而非"检索工具"，让它将原始资料一次性整理成结构化的个人 Wiki，后续直接基于 Wiki 回答问题。

**为什么重要**：这是 LLM Wiki 和 RAG 最根本的哲学差异——RAG 是"临时翻书"，LLM Wiki 是"先建体系再答题"。

**详细解释**：

- **RAG 的逻辑**：每次提问 → 实时检索 → 拼凑片段 → 生成答案（知识不沉淀）
- **LLM Wiki 的逻辑**：导入资料 → LLM 编辑整理（一次）→ 形成结构化 Wiki → 后续提问直接从 Wiki 读取（知识持续积累）

**实际应用场景**：研究员积累某领域文献时，不再是每次提问都重新检索，而是让 LLM 把文献整理成一个不断生长的 Wiki，越用越完善。

---

#### 概念三：LLM Wiki 四步工作流

**是什么**：Ingest（导入）→ Compile（编译）→ Query（查询）→ Lint（校验），构成完整的知识管理闭环。

**为什么重要**：这四步是 LLM Wiki 落地的具体操作路径，缺少任何一步都会影响知识库质量。

**详细解释**：

| 步骤 | 操作 | 关键特点 |
|------|------|----------|
| Ingest 导入 | 看到好资料直接丢进 Raw | 不分类、不整理，越快越好 |
| Compile 编译 | LLM 提炼概念、写词条、建链接、更新索引 | **只做一次**，后续反复复用 |
| Query 查询 | 从 Wiki 读取结构化内容回答 | 不再扫描原始文档 |
| Lint 校验 | LLM 定期扫描，修复不一致/孤立页面/过时内容 | 保持知识库健康 |

**实际应用场景**：日常学习流程——早上收藏文章存入 Raw，晚上执行 Compile 更新 Wiki，提问时直接从 Wiki 获取答案。

---

#### 概念四：推荐工具栈

**是什么**：Karpathy 自己使用的一套极简本地工具组合，完全可以个人复用。

**为什么重要**：证明 LLM Wiki 不需要复杂基础设施，普通人也能落地。

**详细解释**：

| 工具 | 角色 | 用途 |
|------|------|------|
| 本地文件夹 + Markdown | 存储介质 | 全透明，无黑盒 |
| Obsidian | 知识 IDE | 反向链接、知识图谱、Dataview 查询 |
| Claude | LLM | 大上下文模型，适合处理长文档 |
| Prompt 模板 | 规范生成逻辑 | 降低 Token 消耗 |
| Git | 版本管理 | 记录所有 Wiki 修改历史 |

**实际应用场景**：完全本地化部署，无需订阅任何向量数据库服务，个人开发者开箱即用。

### 2.5 延伸思考

1. **LLM Wiki 的 Compile 步骤"只做一次"是理想状态吗？** 当原始资料持续大量涌入时，如何控制每次编译的范围和成本，避免 Compile 步骤本身变成新的瓶颈？

2. **LLM Wiki 和 RAG 真的是非此即彼的关系吗？** 对于需要实时检索海量外部文档（如企业级知识库）的场景，LLM Wiki 的结构化编辑思路是否可以和 RAG 结合，而不是完全替代？

3. **"知识越用越聪明"的复利效应依赖什么前提？** 如果 LLM 在 Compile 阶段就引入了错误的理解或偏差，这个错误会不会随着 Wiki 的生长被不断放大，反而比 RAG 更难纠正？

---

### 2.6 一句话记住它

> RAG 是每次临时翻书，LLM Wiki 是先把书读透再答题——知识库应该越用越聪明，而不是永远失忆。
