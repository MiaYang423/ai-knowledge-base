## 一、视频原文

面试官问：知识图谱和向量数据库到底有什么区别？在工程落地的时候，你该怎么选？又该怎么把他们融合在一起？

大家听到这个问题先别急着去背那些枯燥的定义。大家好，我是彭宇。

咱们先想一个场景。假如你问大模型：某公司被收购前，他的前任 CEO 投过哪些初创企业？如果你只用向量数据库，他可能搜出来一堆关于这家公司、关于 CEO 的新闻片段，看着挺多，但很难直接给你投资列表。为啥？因为他在玩"连连看"，看谁长得像。但如果你有知识图谱，他就像个真探，会顺着"公司→收购→CEO→投资→初创企业"这条线，轻轻松松地把答案扣出来。

所以一句话定调：向量数据库玩的是语义相似度的模糊匹配，而知识图谱玩的是结构化关系的精确推理。一个是感性的模糊派，一个是理性的逻辑派。

好，咱们先通过这组对比把底层逻辑搞透。大家看屏幕左右两侧。

左边是向量数据库，我们称之为"语义匹配的血肉"。他的底层是高维空间的距离计算。他擅长什么？擅长模糊召回，你说"我很热"，他能帮你关联到"冰块"。优点是构建成本低，你把文档 embedding 一下就进去了。但缺点是他是黑盒，你不知道他为什么觉得这两者像。

右边是知识图谱，这是"逻辑推理的骨架"。他的底层是点和线组成的图网络。他擅长的是严谨的事实查询，你说"A 是 B 的爸爸"，他绝不会告诉你"B 是 A 的爸爸"。虽然构建成本高，需要设计 Schema，但他路径清晰，是绝对的白盒。

那问题来了，既然一个有肉、一个有骨头，咱们能不能把他们合体呢？这就是现在工业界最顶尖的 Hybrid RAG 混合检索架构。咱们顺着这个流程走一遍，看看一个请求进来后台到底发生了什么。

第一步，用户提问，当一个逻辑复杂的问题进来时。第二步，智能路由。看到这个黄色的框没？这就是大脑的"交警"。为什么要先路由呢？因为全量检索不仅费钱费时，大模型还容易被无关杂讯带偏。这个交警会先判断：这个问题是查个定义，还是查个复杂的逻辑链条？

接下来的双路并行是重头戏。大家看左右两边。左手边是向量链路，它把非结构化的文本变成向量存进向量库，负责宽泛召回，比如你想找点背景资料，它能给你拉出一大片。右手边是我们的秘密武器知识图谱，我们先用大模型提炼三元组存进图数据库，它不看"长得像不像"，它看的是颜色和缝的点和线，你问多条逻辑，它就去游走匹配。

最后一步，重排序。两边搜出来的东西不一样怎么办？咱们把向量找出来的"肉"和图谱找出来的"骨架"合在一起，经过重排剔除废话，最后喂给大模型。这时候生成的答案既有广度，又绝对不会一本正经地胡说八道。这就是现在业界最前沿的 GraphRAG。

讲到这，大家可能会想：既然区别这么明显，我是不是二选一挑个顺手的就行了？或者简单把他们拼在一起就万事大吉了？真没那么简单。在真实的工程落地里，无论你是单链向量、纯图谱，还是双剑合璧，都有各自绕不开的硬骨头。咱们看这三个最致命的痛点。

第一个，向量的幻觉。向量检索最怕逻辑颠倒，比如你搜"A 打败了 B"，他可能把"B 打败了 A"的文档也召回来。这时候必须靠图谱的拓扑结构来拨乱反正。

第二个，是大家最头疼的图谱构建成本。以前做图谱得请专家写规则，费时费力。现在的解决方案是 OTOKG，讲究本体自动化，直接用大模型来当"搬砖专工"，自动抽关系，成本一下就下来了。

第三个，就是路由怎么分。你得有个聪明的分类器：简单的问答走向量，省钱又快；复杂的涉及统计或查清楚关系的，走图谱。这种"看人下菜碟"的策略，才是高手和小白的区别。

好了，讲了这么多，咱们来个总结。如果面试官最后让你总结，你就拿我准备好的这三点去决杀。

第一点，定调子：向量是黑盒，玩的是高维空间的模糊搜索；图谱是白盒，玩的是离散网络的精确路径。

第二点，说优劣：向量数据库够检快、泛化能力强，但逻辑弱；图谱逻辑强、可解释性强，但构建难。

第三点，谈未来：不要做选择题，我们要全都要。现在的趋势是 GraphRAG 融合架构，这不只是为了应付面试，更是现在所有大厂做私有化部署时的标配。

如果你能把这个融合思想带进简历，那含金量完全不一样。记住，大模型不是万能的，只有给它配上最强的外挂知识库，它才能真正成为专家。这一套架构要是能给面试官讲清楚，那你的 offer 基本就稳了一半了。

### 2.3 核心概念精讲

#### 概念一：向量数据库

**是什么**：将文本转化为高维向量后存储，通过计算向量间距离来衡量语义相似度，从而实现模糊召回的数据库。

**为什么重要**：
构建门槛低，只需对文档做 embedding 即可入库，是目前 RAG 系统最常用的检索底座。

**详细解释**：
- **召回方式**：看"谁长得像"，即语义相近就召回，不管逻辑关系是否正确
- **典型场景**：搜背景资料、找相关段落、模糊问答
- **致命弱点**：搜"A 打败了 B"，可能把"B 打败了 A"的文档也一并召回——方向搞反了它不知道

**实际应用场景**：
用户问"最近有哪些关于大模型的新闻"，向量库能快速拉出一批语义相关的文章片段，适合宽泛的信息聚合。

---

#### 概念二：知识图谱

**是什么**：用"节点（实体）+ 边（关系）"构成的图网络来存储和表达知识，支持沿关系链条进行精确推理。

**为什么重要**：
它是逻辑推理的骨架，能回答"A 和 C 之间通过什么关系连接"这类多跳问题，是向量数据库无法替代的能力。

**详细解释**：
- **存储单元**：三元组（主体 → 关系 → 客体），例如"某 CEO → 投资 → 某初创公司"
- **查询方式**：沿图的边游走，路径清晰可追溯，是完全的白盒
- **典型场景**：多跳逻辑查询，如"某公司被收购前，前任 CEO 投过哪些初创企业"
- **构建痛点**：以前需要专家手写规则，现在可用 OTOKG（本体自动化）让大模型自动抽取关系，大幅降低成本

**实际应用场景**：
企业知识库中查询复杂的人事关系、供应链关系、法律条款引用链等，需要精确推理的场景。

---

#### 概念三：GraphRAG（Hybrid RAG 混合检索架构）

**是什么**：将向量检索和知识图谱检索并联，通过智能路由分配任务、最终重排序融合结果后交给大模型生成答案的架构。

**为什么重要**：
单独用向量有幻觉风险，单独用图谱召回面窄，两者融合才能让大模型的回答既有广度又有逻辑准确性，是当前大厂私有化部署的标配方案。

**详细解释**：

| 流程步骤 | 做什么 | 关键作用 |
|---|---|---|
| 用户提问 | 接收自然语言问题 | 触发整个流程 |
| 智能路由 | 分类器判断问题类型 | 避免全量检索浪费资源 |
| 向量链路 | 文本→向量→向量库→宽泛召回 | 提供背景信息和语义覆盖 |
| 图谱链路 | 大模型抽三元组→图数据库→精确推理 | 保证逻辑关系正确 |
| 重排序融合 | 合并两路结果，剔除冗余 | 提升最终答案质量 |
| 大模型生成 | 基于融合结果生成回答 | 输出既有广度又有逻辑的答案 |

**实际应用场景**：
大厂私有化知识库部署，如内部合规查询系统、企业级智能客服，既需要搜到相关背景，又不能出现逻辑错误。

---

#### 概念四：智能路由（路由分类器）

**是什么**：在 Hybrid RAG 中负责判断用户问题类型，决定将请求分配给向量链路、图谱链路还是两者并行的调度模块。

**为什么重要**：
全量检索既浪费算力，又容易让大模型被无关信息带偏。路由策略是区分工程高手与新手的核心细节。

**详细解释**：

| 问题类型 | 走哪条路 | 原因 |
|---|---|---|
| 简单定义类、背景信息类 | 向量链路 | 省钱、够快、语义匹配足够 |
| 复杂逻辑、多跳关系、统计查询 | 图谱链路 | 需要精确推理，向量容易出错 |
| 两者都需要 | 双路并行 | 兼顾广度与准确性 |

**实际应用场景**：
用户问"什么是 RAG"走向量就够；用户问"A 公司的 CEO 在哪些公司担任过独立董事并且这些公司都在纳斯达克上市"就必须走图谱。

### 2.5 延伸思考

1. **路由分类器本身如何实现？** 如果用大模型来做路由判断，会不会引入额外的延迟和成本，有没有更轻量的替代方案？

2. **OTOKG 自动抽取三元组的准确率如何保证？** 大模型自动抽关系可能引入错误的三元组，错误的图谱结构对最终推理结果的影响有多大，工程上该如何做质量校验？

3. **GraphRAG 的重排序阶段，如何处理向量结果和图谱结果之间的冲突？** 当两条链路给出相互矛盾的信息时，系统应该优先信任哪一侧，依据是什么？
