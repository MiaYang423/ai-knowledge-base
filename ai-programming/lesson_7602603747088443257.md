## 一、视频原文

这么多 Coding 模型到底应该怎么去选？2026 年人人都在玩 Vibe Coding，但用对模型才是关键，模型选错效果不好，又分分钟烧光你的钱包。很多人还停留在一个模型从头写到尾的阶段，但这样做真的效率非常低。毕竟模型也有术业有专攻的问题。

我现在用的是分层写代码模式，让不同的模型做自己最擅长的内容。

第一层，我的前端 Demo 使用 Gemini 2.5 Pro。这个模型有个特点，就是 UI 的界面做得特别漂亮，效果设计也很棒。我一般会用它去快速搭一个原型出来，把界面的效果和基础的交互跑通。但它有个明显短板，逻辑能力不行，复杂的后端逻辑容易出问题。

所以到第二层，我会切到 Claude 系列。这里有分两种情况：如果是常规的编码，我会用 Claude 3.5，执行效率很高，性价比也很不错；但遇到复杂架构设计或者奇怪的 bug，我会直接上 Claude Opus 4.5，贵一点，但解决复杂问题的能力确实要强很多。

举个例子，三个月前我在做我们 AI 视频深层工具的后台管理系统。我先用 Gemini 画了个管理界面，花了半个小时，有了不错的视觉效果。然后用 Claude Opus 去设计整体的架构，包括数据库的设计、API 接口规划等等。架构定义好以后，具体的代码实现就交给 Claude 3.5 去跑，跑了一天时间，一整个系统就跑下来了。

另外，如果你不差钱，你全程用 Claude Opus 开 Sonnet 模式，效果肯定是最好的。但对大部分人来说，性价比还是需要考虑的。

这里我再推荐一个平替方案：Gemini 2.0 Flash。这个模型我最近用下来感觉很不错，水平跟 Claude 3.5 差不多，有些代码的处理甚至更好，关键是特别便宜，主打一个量大管饱。唯一的缺点就是有时候你可能会多跑几轮验证，但问题都能解决。

而 Gemini 2.5 Pro 其实也挺有潜力的，价格也很划算，但 Anthropic 的安全能力我觉得还是有待提高，跟 Claude 的代码质量相比还是会有一些差距。

至于 GPT 的话，我真的基本不用它去写代码。不是说质量不好，就是它的整体性价比较低，同样的钱能买到更好的服务，那为什么还要选它呢？

总结一下我自己的模型选用策略：前端原型用 Gemini 2.5 Pro，快速出效果，UI 设计美观；架构设计用 Claude Opus，把方向定好，推理能力很强；具体编码在 Claude 3.5 和 Gemini 2.0 Flash 之间看预算去决定。

另外也别迷信某一个模型，AI 工具更新太快了，根据具体需求来选择才是最务实的做法。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**不同 AI Coding 模型各有专长，用"分层写代码"策略——前端原型用 Gemini、架构设计用 Claude Opus、具体编码用 Claude 3.5 或 Gemini Flash——可以大幅提升效率并节省成本。**

### 2.3 核心概念精讲

#### 分层写代码模式

**是什么**：根据任务类型（UI 原型 / 架构设计 / 具体编码）选择最适合的 AI 模型，而不是用同一个模型完成所有工作。

**为什么重要**：不同模型在 UI 设计、逻辑推理、代码生成方面能力差异显著；用错模型不仅效果差，还会白白烧钱。

**详细解释**：

| 层级 | 任务 | 推荐模型 | 优势 |
|------|------|----------|------|
| 第一层 | 前端 UI 原型 | Gemini 2.5 Pro | 界面美观，快速出原型 |
| 第二层（常规） | 具体编码实现 | Claude 3.5 | 执行效率高，性价比好 |
| 第二层（复杂） | 架构设计 / 疑难 Bug | Claude Opus 4.5 | 复杂问题解决能力强 |
| 平替 | 具体编码实现 | Gemini 2.0 Flash | 极低价格，质量接近 Claude 3.5 |

**实际应用场景**：构建一个完整的后台管理系统，先用 Gemini 画界面，再用 Claude Opus 定架构，最后用 Claude 3.5 写代码，一天内完成整个系统。

---

#### 性价比决策原则

**是什么**：在模型能力和使用成本之间做权衡，根据任务复杂度选择"够用"的模型。

**为什么重要**：顶级模型（如 Claude Opus）确实能力最强，但价格也最贵；对大多数任务来说，性价比更高的模型完全够用。

**详细解释**：
- **不差钱**：全程用 Claude Opus + Sonnet 模式，效果最佳
- **普通预算**：Claude 3.5 处理常规编码，Opus 仅用于复杂问题
- **预算有限**：Gemini 2.0 Flash 作为平替，价格极低，多跑几轮也能解决问题

**实际应用场景**：遇到奇怪 Bug 时才切换到 Claude Opus，平时用 Claude 3.5 或 Gemini Flash 完成日常编码任务。

---

#### 模型能力边界认知

**是什么**：了解每个模型的"擅长"和"短板"，避免用错场景。

**为什么重要**：Gemini 2.5 Pro UI 能力强但后端逻辑弱；GPT 系列性价比低；Gemini Flash 便宜但需要多轮验证——知道边界才能用对模型。

**详细解释**：
- Gemini 2.5 Pro：UI/UX 强，复杂后端逻辑弱
- Claude 3.5：常规编码稳定，性价比高
- Claude Opus 4.5：复杂推理强，价格贵
- Gemini 2.0 Flash：极低成本，需多轮验证
- GPT 系列：质量可以，但性价比不如同档位竞品

**实际应用场景**：不要用 Gemini 2.5 Pro 写复杂的后端 API，也不要用 Claude Opus 画简单的 UI 原型。

### 2.5 延伸思考

1. **分层策略的边界在哪里？** 当项目规模变大、模型切换成本（上下文同步、Prompt 重写）超过节省的费用时，分层策略是否还值得？如何评估这个临界点？

2. **模型更新如何影响分层策略？** AI 模型迭代极快，今天的"性价比之王"可能三个月后就被新模型取代，如何建立一套动态调整模型组合的机制？

3. **分层策略对团队协作的影响？** 在多人开发团队中，不同成员使用不同模型完成不同层级的任务，如何保证代码风格一致性和上下文传递的准确性？

---

### 2.6 一句话记住它

> 别用一个模型从头写到尾——Gemini 画界面、Claude Opus 定架构、Claude 3.5 写代码，分层用模型才是 Vibe Coding 的正确姿势。
