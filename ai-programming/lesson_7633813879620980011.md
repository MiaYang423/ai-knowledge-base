## 一、视频原文

一张图让你看懂小龙虾（AI Agent）的核心流程，明明白白的养虾。

大家好，我是汉山。我们在微信、飞书里发的一句话，小龙虾为什么能够理解并且还能够去执行？今天我们就用这张图给大家讲清楚。

所以我们从最开始发消息这里，比如微信、飞书，从这边发消息，发的消息会通过出口流走到我们的网关的位置里，就是这个路由或者网关。这是我们的一个数据的入口，它负责一些鉴权、限流，或者一些安全检测。比如我们这个飞书或者微信怎么能够接进来，或者我们的网页是怎么能够接进来的，它需要看一些权限，都是通过这个进行检测。

然后网关收到的数据回到这里进行组装，它会拼上我们的历史的上下文。通常会需要我们 System Prompt 的说明，如果我们有哪些技能、有哪些能力、能够做什么，都要在这里进行合成。如果我们 MCP 的外部接口能力有哪些接口可以调用，都要在这里整理进来。我们这个历史上下文其实就是一种记忆，我们这种 Memory 的记忆，这个记忆短期的一般就存在我们内存里面，但是长期的一般通过文件或者说有一个数据库来存它。这个时候会把我们历史记忆也会读进来，一同进行组装。

组装之后的上下文会送到交互决策模块，我们这里是一个核心模块，它拿到我们组装的上下文之后，从这里最终生成的 Prompt 送给我们这个大模型，大模型进行一个处理，回复过来的内容会给到交互决策模块。这样的，这个交互决策模块它会对这个回复的内容进行一个初步的分析——这个内容是最后的回复，还是回复的一些中间步骤。回复的内容如果有一个工具的调用，它这个组装就会从这里去调工具，比如发消息，比如建一个任务，比如搜一个网页，或者读一个文件，或者执行一个命令，都是可以的。这些工具都是有返回的，就是这些执行结果，比如搜网页搜到的内容要回来，回过来之后这个内容它会再进行整理，这个内容以后我们组装的这些上下文，它会进行一个分析，分析完之后如果需要模型进行再次处理，它会送到大模型里面，然后再次处理。如果还需要接着调用工具，那就是再去调工具，再处理，就这么一个反复的过程，继续执行。

这就是我们说的 Agent 为什么能够持续地去做一些事情，一直做。因为它不停地去调模型，回来之后需要调什么工具，工具回来之后这内容再送给模型，再送给工具，就是这么反复地一直去调，直到得到我们最终想要的结果。这里有一个最终的答案，最终的回复会回到网关这里，通过网关再回到我们消息的发送者，微信或飞书回来，这样我们一个完整的数据流就结束了。

我们给 Agent 下了一个命令，然后它执行完之后我们收到一个结果，这就是我们这个 Agent 的一个完整的数据流和工作过程。好了，今天的分享就到这里，更多的玩法内容我们下一期再见。

### 2.2 知识地图

```
AI Agent 完整数据流
│
├── 1. 消息入口
│   ├── 来源：微信 / 飞书 / 网页
│   └── 经过：网关 / 路由
│       ├── 鉴权
│       ├── 限流
│       └── 安全检测
│
├── 2. 上下文组装
│   ├── System Prompt（技能说明 / 角色定义）
│   ├── MCP 外部接口列表（可调用工具清单）
│   └── Memory（历史记忆）
│       ├── 短期记忆 → 内存
│       └── 长期记忆 → 文件 / 数据库
│
├── 3. 交互决策模块（核心）
│   ├── 接收组装好的上下文
│   ├── 生成 Prompt → 发送给大模型
│   ├── 接收大模型回复
│   └── 判断回复类型
│       ├── 最终答案 → 直接返回
│       └── 中间步骤 → 触发工具调用
│
├── 4. 工具调用（Tool Use）
│   ├── 发消息
│   ├── 建任务
│   ├── 搜索网页
│   ├── 读取文件
│   └── 执行命令
│       └── 工具返回结果 → 重新进入决策模块
│
├── 5. 循环执行机制
│   └── 模型 → 工具 → 模型 → 工具 → ... → 最终结果
│
└── 6. 结果返回
    └── 最终答案 → 网关 → 用户（微信 / 飞书）
```

### 2.3 核心概念精讲

#### 概念一：网关 / 路由（Gateway）

**是什么**：Agent 系统的数据入口，所有外部消息进来、所有结果出去都经过这里。

**为什么重要**：
没有网关，任何人任何请求都能直接打到系统内部，既不安全也无法管控。

**详细解释**：

| 功能 | 说明 |
|------|------|
| 鉴权 | 验证请求来源是否合法（如飞书 / 微信的 Token 校验） |
| 限流 | 控制单位时间内的请求数量，防止过载 |
| 安全检测 | 过滤恶意内容或非法调用 |

**实际应用场景**：
用飞书机器人给 Agent 发一条消息，飞书会带上签名信息，网关验证签名合法后才放行，否则直接拒绝。

---

#### 概念二：上下文组装（Context Assembly）

**是什么**：在把消息发给大模型之前，把所有相关信息拼在一起，形成完整的 Prompt。

**为什么重要**：
大模型是"无状态"的，它只能看到你这次给它的内容。要让它"记得"历史、"知道"自己能干什么，必须每次都把这些信息拼进去。

**详细解释**：

| 组成部分 | 内容 | 存储位置 |
|----------|------|----------|
| System Prompt | Agent 的角色定义、技能说明、行为规则 | 代码 / 配置文件 |
| MCP 工具列表 | 当前可调用的外部接口清单 | 动态加载 |
| 短期记忆 | 本次对话的历史消息 | 内存 |
| 长期记忆 | 跨会话的用户信息或任务记录 | 文件 / 数据库 |

**实际应用场景**：
你第二天再问 Agent"昨天那个任务做完了吗"，它能回答，是因为长期记忆把昨天的记录存到了数据库，今天组装上下文时又读了进来。

---

#### 概念三：交互决策模块（Interaction & Decision Core）

**是什么**：Agent 的"大脑"，负责把组装好的上下文发给大模型，并判断模型返回的内容该怎么处理。

**为什么重要**：
大模型的回复不一定是最终答案，可能只是"我需要先去搜一下网页"——决策模块负责识别这种差异并做出对应动作。

**详细解释**：

```
收到大模型回复
        ↓
  判断回复类型
   ↙          ↘
最终答案      包含工具调用指令
   ↓                ↓
返回给用户      执行对应工具
                    ↓
              拿到工具结果
                    ↓
              重新组装上下文
                    ↓
              再次发给大模型
```

**实际应用场景**：
你让 Agent 帮你查今天的天气并写进日历，它第一次调模型得到"需要调用天气接口和日历接口"的指令，决策模块依次执行这两个工具，最后再调一次模型生成最终回复。

---

#### 概念四：循环执行机制（Agentic Loop）

**是什么**：Agent 在"调用大模型 → 调用工具 → 再调用大模型"之间不断循环，直到任务完成。

**为什么重要**：
这是 Agent 和普通聊天机器人最本质的区别——普通对话一问一答就结束，Agent 可以自主地把一件复杂任务拆成多步连续执行完。

**详细解释**：

```
用户发指令
    ↓
模型分析 → 需要工具？
    ↓是              ↓否
调用工具          输出最终结果
    ↓                  ↓
结果回来          返回给用户
    ↓
再次送给模型
    ↓
继续判断 → 需要工具？
    ↓是              ↓否
继续循环          输出最终结果
```

**实际应用场景**：
让 Agent 帮你整理竞品资料：它会先搜索 A 竞品 → 读取内容 → 搜索 B 竞品 → 读取内容 → 汇总写报告，整个过程可能循环十几次，全程不需要你干预。

---

### 2.4 速查表

| 环节 | 模块名称 | 核心职责 | 关键技术点 |
|------|----------|----------|------------|
| 1 | 网关 / 路由 | 数据入口与安全管控 | 鉴权、限流、安全检测 |
| 2 | 上下文组装 | 拼装完整 Prompt | System Prompt、Memory、MCP 工具列表 |
| 3 | 交互决策模块 | 调模型 + 判断结果类型 | 工具调用识别、结果路由 |
| 4 | 工具层（MCP） | 执行具体动作 | 发消息、搜网页、读文件、执行命令 |
| 5 | 循环机制 | 持续执行直到任务完成 | 模型 ↔ 工具反复调用 |
| 6 | 结果返回 | 把最终答案送回用户 | 经网关回到微信 / 飞书 |

---

### 2.5 延伸思考

1. **循环会不会停不下来？** Agent 的循环机制依赖大模型判断"任务是否完成"，如果模型判断失误或任务描述不清，理论上会陷入死循环。实际工程中会怎么设计"终止条件"来避免这个问题？

2. **长期记忆存什么、不存什么？** 视频提到长期记忆存在文件或数据库里，但如果什么都存，上下文会越来越长、成本越来越高。应该用什么策略决定哪些记忆值得长期保留？

3. **MCP 工具越多越好吗？** 视频说可调用的工具列表要在上下文组装时全部放进去，但工具列表本身也占 Token。当工具数量很多时，如何让模型准确选择正确的工具，而不是被大量选项干扰？

---

### 2.6 一句话记住它

> Agent 的本质就是一个不停转的循环：把消息组装好发给大模型，模型说要用工具就去用，工具结果再喂给模型，反复直到任务完成，然后把结果原路送回给你。
