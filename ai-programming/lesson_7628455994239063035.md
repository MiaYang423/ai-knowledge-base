## 一、视频原文

让我来解释一下什么是 MCP。MCP 是 Model Context Protocol（模型上下文协议）。它是一个为 AI Agent 提供工具使用能力的新标准。

我们先看最简单的大语言模型交互方式：你有一个提示词，把它发给大语言模型，大语言模型给你一个回复。这在两年前很不错，但我们知道大语言模型的能力远不止于此，它们只是需要工具。

我说的工具，指的是像网页搜索、记忆、访问你的邮件、访问你的文档这类东西，基本上任何你能在网上做的事情，都可以成为你的 Agent 的工具。

于是我们进入了这个阶段：我们开始给大语言模型提供工具调用能力。突然间，它们拥有了所有这些额外的、令人难以置信的能力。比如一个提示词进来，大语言模型会调用网页搜索，调用图像生成，从你的文档中获取信息，然后给你回复。

但是有一个大问题——所有这些工具都是用不同的方式编写的，没有统一标准。所以大语言模型会感到困惑。

然后我们有了 MCP。MCP 为大语言模型的工具调用实现了标准化。它允许任何人创建一个 MCP 服务器，以标准化的方式向大语言模型暴露他们的工具。现在有大量的工具可用，每个主要公司都有一个 MCP 服务器可用，Google、AWS、GitHub、Brave Browser、FireCrawl，所有这些令人难以置信的工具。

### 2.3 核心概念精讲

#### MCP（Model Context Protocol）

**是什么**：模型上下文协议，一个为大语言模型工具调用制定的统一标准，允许任何人以标准化方式创建 MCP 服务器，向 AI Agent 暴露工具能力。

**为什么重要**：在 MCP 之前，每个工具都有自己的接入方式，大语言模型面对五花八门的工具接口会产生混乱；MCP 就像 USB 接口标准，让所有工具"即插即用"。

**详细解释**：
| 对比维度 | MCP 之前 | MCP 之后 |
|---|---|---|
| 工具接入方式 | 各自为战，没有标准 | 统一标准，任何人可创建 |
| LLM 理解工具 | 困惑，难以一致调用 | 清晰，按标准协议调用 |
| 工具生态 | 碎片化 | 统一生态，主要公司均有 MCP 服务器 |

**实际应用场景**：Google、AWS、GitHub、Brave Browser、FireCrawl 等主流工具均已提供 MCP 服务器，AI Agent 可以通过统一协议调用这些工具。

---

#### 大语言模型工具调用的演进

**是什么**：大语言模型从纯文本问答，到能调用外部工具，再到通过 MCP 标准化调用工具的三个发展阶段。

**为什么重要**：理解这个演进过程，能帮助我们理解为什么 MCP 是必要的，以及它在 AI 能力发展中的位置。

**详细解释**：
| 阶段 | 能力 | 局限 |
|---|---|---|
| 阶段1：纯问答 | 提示词→LLM→回复 | 只能处理文本，无法访问外部信息 |
| 阶段2：工具调用 | 可调用搜索/邮件/文档等工具 | 工具各自标准，LLM 容易混乱 |
| 阶段3：MCP | 标准化工具调用，统一接口 | — |

**实际应用场景**：用户向 AI Agent 提问"帮我搜索最新新闻并总结"，Agent 通过 MCP 调用 Brave Browser 的搜索工具，获取结果后生成回复。

---

#### MCP 服务器

**是什么**：按照 MCP 协议规范创建的服务端程序，将某个工具（如搜索、邮件、文件系统等）的能力以标准化方式暴露给大语言模型。

**为什么重要**：MCP 服务器是工具接入 AI 生态的"适配器"，任何开发者都可以为自己的工具创建 MCP 服务器，从而让 AI Agent 能够使用该工具。

**实际应用场景**：GitHub 创建了自己的 MCP 服务器，AI Agent 可以通过 MCP 直接操作 GitHub 仓库（创建 PR、查看 issue 等）。
