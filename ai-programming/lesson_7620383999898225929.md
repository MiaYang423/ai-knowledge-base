# 教案：知识图谱与向量数据库的工程选择

> 适合人群：准备 AI 大模型相关面试、或正在做 RAG 系统工程落地的开发者和算法工程师。

## 一、视频原文

面试官问：知识图谱和向量数据库到底有什么区别？在工程落地的时候，你该怎么选？又该怎么把他们融合在一起？

大家听到这个问题先别急着去背那些枯燥的定义。大家好，我是彭宇。咱们先想一个场景：假如你问大模型，某公司被收购前，他的前任 CEO 投过哪些初创企业。如果你只用向量数据库，他可能搜出来一堆关于这家公司、关于 CEO 的新闻片段，看着挺多，但很难直接给你投资列表。为啥？因为他在玩连连看——看谁长得像。但如果你有知识图谱，他就像个真探，会顺着"公司→收购→CEO→投资→初创企业"这条线，轻轻松松地把答案扣出来。

所以一句话定调：向量数据库玩的是语义相似度的模糊匹配，而知识图谱玩的是结构化关系的精确推理。一个是感性的模糊派，一个是理性的逻辑派。

好，咱们先通过这组对比把底层逻辑搞透。大家看屏幕左右两侧。左边是向量数据库，我们称之为"语义匹配的血肉"，他的底层是高维空间的距离计算，他擅长模糊召回——你说"我很热"，他能帮你关联到"冰块"。优点是构建成本低，你把文档 embedding 一下就进去了。但缺点是他是黑盒，你不知道他为什么觉得这两者像。

右边是知识图谱，这是"逻辑推理的骨架"，他的底层是点和线组成的图网络，他擅长严谨的事实查询——你说"A 是 B 的爸爸"，他绝不会告诉你"B 是 A 的爸爸"。虽然构建成本高，需要设计 Schema，但他路径清晰，是绝对的白盒。

那问题来了，既然一个有肉一个有骨头，咱们能不能把他们合体呢？这就是现在工业界最顶尖的 Hybrid RAG 混合检索架构。咱们顺着这个流程走一遍，看看一个请求进来后台到底发生了什么。

第一步，用户提问。当一个逻辑复杂的问题进来时，第二步，智能路由——看到这个黄色的框没，这就是大脑的交警。为什么要先路由呢？因为全量检索不仅费钱费时，大模型还容易被无关杂讯带偏。这个交警会先判断：这个问题是查个定义，还是查个复杂的逻辑链条？

接下来的双路并行是重头戏。左手边是向量链路，它把非结构化的文本变成向量存进向量库，负责宽泛召回，比如你想找点背景资料，它能给你拉出一大片。右手边是我们的秘密武器——知识图谱，我们先用大模型提炼三元组存进图数据库，它不看"长得像不像"，它看的是颜色和缝的点和线。你问多条逻辑，它就去游走匹配。

最后一步，重排序。两边搜出来的东西不一样怎么办？咱们把向量找出来的"肉"和图谱找出来的"骨架"合在一起，经过重排提出废话，最后喂给大模型。这时候生成的答案既有广度，又绝对不会一本正经地胡说八道。这就是现在业界最前沿的 Graph RAG。

讲到这，大家可能会想：既然区别这么明显，我是不是二选一挑个顺手的就行了？或者简单把他们拼在一起就万事大吉了？真没那么简单。在真实的工程落地里，无论你是单链向量、纯图谱，还是尝试双剑合璧，都有各自绕不开的硬骨头。咱们看这三个最致命的痛点：

第一个，向量的幻觉。向量检索最怕逻辑颠倒，比如你搜"A 打败了 B"，他可能把"B 打败了 A"的文档也召回来。这时候必须靠图谱的拓扑结构来拨乱反正。

第二个，是大家最头疼的图谱构建成本。以前做图谱得请专家写规则，现在的解决方案是什么？是 OTOKG，讲究本体自动化，直接用大模型来当"搬砖专工"，自动抽关系，成本一下就下来了。

第三个，路由怎么分。你得有个聪明的分类器：简单的问答走向量，省钱又快；复杂的涉及统计或查清楚关系的走图谱。这种"看人下菜碟"的策略才是高手和小白的区别。

好了，讲了这么多，咱们来个总结。如果面试官最后让你总结，你就拿我准备好的这三点去决杀：

第一点，定调子：向量是黑盒，玩的是高维空间的模糊搜索；图谱是白盒，玩的是离散网络的精确路径。

第二点，说优劣：向量数据库够检快、泛化能力强，但逻辑弱；图谱逻辑强、可解释性强，但够检难。

第三点，谈未来：不要做选择题，我们要全都要。现在的趋势是 Graph RAG 融合架构，这不只是为了应付面试，更是现在所有大厂做私有化部署时的标配。如果你能把这个融合思想带进面试，那含金量完全不一样。

记住，大模型不是万能的，只有给它配上最强的外挂知识库，它才能真正成为专家。这一套架构要是能给面试官讲清楚，那你的 offer 基本就稳了一半。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**向量数据库擅长模糊语义召回、知识图谱擅长精确关系推理，两者融合的 Graph RAG 架构才是工程落地的正确答案，也是面试中的加分利器。**

---

### 2.2 知识地图

```
知识图谱 vs 向量数据库
├── 向量数据库（语义匹配的血肉）
│   ├── 底层：高维空间距离计算
│   ├── 优点：构建成本低、泛化能力强、模糊召回
│   └── 缺点：黑盒、逻辑弱、易幻觉（方向颠倒）
├── 知识图谱（逻辑推理的骨架）
│   ├── 底层：点（实体）+ 线（关系）的图网络
│   ├── 优点：白盒、路径清晰、精确推理
│   └── 缺点：构建成本高、需设计 Schema
└── Hybrid RAG / Graph RAG（融合架构）
    ├── 流程
    │   ├── 1. 用户提问
    │   ├── 2. 智能路由（判断走向量还是图谱）
    │   ├── 3. 双路并行检索
    │   │   ├── 向量链路：宽泛召回背景信息
    │   │   └── 图谱链路：精确游走逻辑关系
    │   └── 4. 重排序合并 → 喂给大模型生成答案
    └── 三大工程痛点
        ├── 向量幻觉（逻辑颠倒）→ 靠图谱拨乱反正
        ├── 图谱构建成本 → OTOKG 大模型自动抽关系
        └── 路由策略 → 简单问答走向量，复杂逻辑走图谱
```

### 2.3 核心概念精讲

#### 向量数据库

**是什么**：将文本转化为高维向量，通过计算向量间的距离（相似度）来检索相关内容的数据库。

**为什么重要**：构建成本极低（文档 embedding 后直接存入），能处理非结构化文本，是当前 RAG 系统最常见的知识库形式。

**详细解释**：
- 核心机制：语义相似度匹配（"模糊派"）
- 优点：构建快、泛化强、适合宽泛召回
- 缺点：黑盒（不可解释）、逻辑推理弱，容易产生"方向性幻觉"（把"A 打败 B"和"B 打败 A"视为相似）

**实际应用场景**：检索背景资料、找相关新闻片段、语义搜索。

---

#### 知识图谱

**是什么**：以"实体（点）+ 关系（线）"构成的图网络，存储结构化的事实关系，支持精确的逻辑推理。

**为什么重要**：能沿着关系链条精确推理（如"公司→收购→CEO→投资→初创企业"），是解决复杂逻辑查询的唯一可靠方案。

**详细解释**：
- 核心机制：三元组（主体-关系-客体）存储，图遍历推理
- 优点：白盒（路径可解释）、逻辑严谨、不会颠倒方向
- 缺点：需设计 Schema，构建成本高；传统方式需领域专家，现可用大模型自动抽取（OTOKG）

**实际应用场景**：复杂关系查询（谁投资了谁）、多跳推理、需要可解释性的业务场景。

---

#### Graph RAG（混合检索架构）

**是什么**：将向量检索（宽泛召回）和知识图谱检索（精确推理）融合的混合 RAG 架构，是当前业界最前沿的大模型知识库方案。

**为什么重要**：单一向量库有逻辑幻觉问题，单一图谱构建成本高且召回窄，两者融合才能实现"既有广度又有精度"的答案生成。

**详细解释**：
| 步骤 | 动作 | 说明 |
|------|------|------|
| 1 | 用户提问 | 接收复杂问题 |
| 2 | 智能路由 | 分类器判断走向量还是图谱 |
| 3a | 向量检索 | 宽泛召回背景文本 |
| 3b | 图谱检索 | 精确游走关系链条 |
| 4 | 重排序合并 | 去噪、整合两路结果 |
| 5 | 大模型生成 | 基于高质量上下文输出答案 |

**实际应用场景**：大厂私有化部署、企业知识库问答、需要同时处理"背景信息+逻辑推理"的复杂问答系统。

---

#### 智能路由

**是什么**：在 Hybrid RAG 系统中，对用户问题进行分类，决定走向量检索还是图谱检索的"交警"模块。

**为什么重要**：全量检索既费钱又容易被无关信息干扰，智能路由是系统效率和精度的关键。

**详细解释**：
- 简单问答（查定义、找背景）→ 走向量，省钱又快
- 复杂逻辑（多跳关系、统计推理）→ 走图谱，精确可靠

**实际应用场景**：生产环境中的 RAG 系统成本控制与精度优化。

---

### 2.4 速查表

| 维度 | 向量数据库 | 知识图谱 |
|------|-----------|---------|
| 核心机制 | 高维空间距离计算（语义相似度） | 图遍历推理（三元组关系） |
| 可解释性 | 黑盒 ❌ | 白盒 ✅ |
| 构建成本 | 低（embedding 即可） | 高（需设计 Schema，可用大模型降低） |
| 擅长场景 | 模糊召回、背景检索 | 精确推理、多跳关系查询 |
| 逻辑能力 | 弱（易方向性幻觉） | 强（严格遵守关系方向） |
| 泛化能力 | 强 | 弱（依赖已有图谱结构） |
| 代表技术 | FAISS、Milvus、Pinecone | Neo4j、知识图谱三元组 |
| 融合方案 | Graph RAG / Hybrid RAG | Graph RAG / Hybrid RAG |

---

### 2.5 延伸思考

1. **智能路由的准确率瓶颈**：如果路由分类器本身判断错误（把复杂逻辑问题误判为简单问答走了向量），会导致什么后果？如何设计 fallback 机制？

2. **OTOKG 的质量问题**：用大模型自动抽取三元组构建知识图谱，降低了构建成本，但大模型本身的幻觉可能导致图谱中存在错误关系。如何做图谱质量校验？

3. **Graph RAG 的实时性挑战**：知识图谱一旦构建，更新成本较高。对于需要实时更新的业务场景（如股票资讯、新闻事件），Graph RAG 架构应如何设计增量更新机制？

---
