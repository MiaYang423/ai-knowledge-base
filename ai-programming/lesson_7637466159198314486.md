### 2.3 核心概念精讲

#### 概念一：Expo Blank TypeScript 模板

**是什么**：Expo 是一套用于开发手机 APP 的工具框架；Blank TypeScript 是其提供的一个最简起步模板，使用 TypeScript 语言，开箱即用。

**为什么重要**：
用一条指令就能让 Codex 在空目录里生成完整的项目骨架，省去手动配置环境的步骤，是 AI 辅助开发的标准起手式。

**详细解释**：

| 名词 | 含义 |
|---|---|
| Expo | 手机 APP 开发工具，支持 iOS 和 Android |
| Blank | 最简模板，没有多余预置功能 |
| TypeScript | 比 JavaScript 更严格的编程语言，有类型检查 |
| 初始化指令 | `请用 Expo Blank TypeScript 模板初始化这个空目录` |

**实际应用场景**：
在 Codex 对话框里直接发送初始化指令，执行后目录里会自动出现 APP 入口文件、配置文件、依赖声明和脚本，可以直接进入开发阶段。

---

#### 概念二：AGENT.md 项目规范文件

**是什么**：一个放在项目根目录下的说明文件，用来告诉 Codex 这个项目的基本规则和约定，让 AI 在整个项目过程中保持一致的行为。

**为什么重要**：
没有规范文件，Codex 每次执行任务时可能做出不一致的选择（比如忽然改用英文、忽然尝试联网）。AGENT.md 相当于给 AI 立的"项目宪法"，一次写好，全程生效。

**详细解释**：

视频中 AGENT.md 只有五行核心内容：

| 字段 | 内容 |
|---|---|
| 项目类型 | Expo 移动应用项目 |
| 技术栈 | （列出所用技术） |
| 开发方式 | 本地优先，不上云 |
| 写作语言 | 全程中文 |

**实际应用场景**：
项目初始化完成后，立刻让 Codex 生成 AGENT.md，把项目类型、技术栈、本地优先、语言要求写进去，后续所有任务 Codex 都会遵守这些规范。

---

#### 概念三：Codex 的模型选择与锁定原则

**是什么**：Codex 对话框右下角可以选择不同的 AI 模型（如 o4、o3 等历史版本）来驱动项目构建。

**为什么重要**：
不同模型的能力和行为有差异。一个项目中途换模型，可能导致代码风格不一致、上下文理解断裂，引发难以排查的问题。

**详细解释**：
- 在项目开始前选定模型
- 确定后，整个项目周期内不要切换到其他模型
- 这是保证项目构建一致性的基本纪律

**实际应用场景**：
新建项目、发出第一条指令之前，先在对话框右下角确认好模型，之后不再改动。

---

#### 概念四：Codex 任务排队与插队机制

**是什么**：当 Codex 正在执行一个主任务时，如果用户插入新消息，Codex 有两种处理方式——排队（等主任务完成再回答）或插队（暂停主任务先回答插入内容）。

**为什么重要**：
理解这个机制，可以避免误以为 Codex 卡住了，也能主动选择适合当前场景的处理方式。

**详细解释**：

| 模式 | 行为 |
|---|---|
| 排队模式 | 主任务执行完毕后，再处理插入的问题 |
| 插队模式 | 跳出主任务，先回答插入内容，再继续主任务 |

切换方式：在 Codex 设置里直接配置。

**实际应用场景**：
初始化项目时，如果对"Expo Blank TypeScript"不理解，可以直接插入一条"请用通俗语言解释 Expo 以及这句话的含义"，Codex 会根据设置的模式来决定何时回答。

### 2.4 速查表

| 操作步骤 | 具体做法 |
|---|---|
| 新建项目目录 | 右键新建空白项目，命名为 Speak to Note |
| 选定模型 | 对话框右下角选择，选好后全程不换 |
| 初始化项目 | 发送：请用 Expo Blank TypeScript 模板初始化这个空目录 |
| 询问不懂的术语 | 发送：请用通俗的语言解释 Expo 并解释这句话的含义 |
| 生成规范文件 | 让 Codex 写 AGENT.md，包含项目类型、技术栈、本地优先、全程中文 |
| 首次设置关键项 | 批准策略 Auto-approve、核心设置 Workspace Write、权限访问默认关闭 |
| 任务插队/排队 | 在 Codex 设置里切换两种模式 |

### 2.5 延伸思考

1. **AGENT.md 写多详细才合适？** 视频里只用了五行，但实际项目中是否应该把代码规范、命名约定、禁止行为也写进去？更详细的 AGENT.md 会让 Codex 更听话，还是反而产生干扰？

2. **中途换模型真的会出问题吗？** Codex 的上下文记忆是跟模型绑定的还是跟项目目录绑定的？如果换了模型但项目文件不变，Codex 能否无缝接续之前的工作？

3. **"本地优先不上云"这个约束，Codex 能真正执行吗？** 写在 AGENT.md 里的规则是软约束还是硬约束？如果 Codex 在某个任务里自动选择了需要联网的方案，开发者应该如何发现和纠正？

---

### 2.6 一句话记住它

> 用 Codex 做项目，先建目录选模型、一句话初始化框架，再用 AGENT.md 立好规矩——框架搭对了，后面的事才能交给 AI。
