# 教案：多Agent协同编程工作流实战

> 适合人群：想用 AI 工具实现产品想法、但没有技术背景的产品经理和文科背景创作者。

## 一、视频原文

哈喽大家，我是在自己干了五年多的产品经理，但是完全不懂代码，纯文科背景的我。我的视频都是从纯小白的视角来分享 Vibe Coding 的。

之前我跟大家分享过一套我 Vibe Coding 的工作流程，在小红书上有了非常不错的反响，有很多的用户都觉得这个方法非常好用。这套工作流分我两步：第一步是我会在 Claude 里面跟它去聊需求，让它根据我的诉求来画 Askma（原型图）；第二步呢，我会把 Askma 丢到像 Coding Agent 或者是 Lovable 这种网站里面去，让它根据我的 Askma 来做网站。它的好处是什么？非常地还原，它不会给你乱做网站，你要什么样的功能，它基本就是按照 Askma 的样子把这个功能给实现出来，token 消耗得很少，很省钱。但问题也是很多朋友会跟我说，我没有梯子。

所以我一直在帮大家探索有没有可能有一款软件，它能像 Claude 一样，我直接在这个软件里面就能够把我说的这两步流程包圆。这个软件就是腾讯的 QClaude（QQ 旗下 AI 应用）。它们做了一个桌面端的应用，在这个应用里面内置了三个 Agent：一个叫无部颜，一个叫代码行，一个是灵感漫。这个代码行就是一个程序员的 Agent，我们一会儿会用到。

我自己 DIY 的 Agent 叫"套"，它是一个产品方案的大管家。其实我们类比来看，"套"就相当于原来的 Claude，而代码行就是原来我的 Coding Agent。这个"套"和这个代码行，它俩的分工是非常明确的：我会在"套"这个 Agent 里面跟它去聊产品方案，然后让它帮我产生 Askma，再到代码行这个 Agent 里面跟它说"按照这个 Askma 做一个 HTML 的网站"。有了这两个 Agent 彼此协同之后，我就不需要再去跨应用来完成我的需求设计了。而且你是可以用它们的默认大模型，或者是对接你自己的模型 API，或者是 Coding Plan 的。

我们可以来看看效果，它会根据我给它的 Askma 来生成这样的一版，四个 Tab 之间可以交互的，点到某一个单页里面去，它还可以直接帮我跳转到小鱼桌里面去。整个流程下来我没有接触其他的 App，而且并不需要任何的梯子。

这就是多 Agent 的好处，因为 Agent 跟 Agent 之间是彼此隔离的，它们相当于每个人只做自己最擅长的事情，而不是一个 Agent 干所有的事情。就好像什么呢？好像你在上课的时候，每门课的老师它都是分开的，语文的教语文，美术的教美术。你可以给你的产品老师赋予它的设定，比如你告诉它是一名非常丰富经验的产品方案顾问，这个 prompt 是你自己给这个产品大管家的 Agent 设定，然后你也可以在这里面给它起名字和上传头像。

我为什么会推荐这种多 Agent 之间协作的模式呢？我真心觉得让每个 Agent 各司其职，它并不是一种小技巧，而是未来的一种方法论。我不知道大家有没有最近刷到在小红书网站上很火的一个开源项目叫"三省六部制"。它的核心就是像唐朝的三省六部制一样：皇上下达指令，太子来负责判断这个事情到底要不要做，然后中书规划、门下审议、尚书执行，然后最终到六部各司其职，执行各自的模块，最后回报，把整件事情完成。

像我们刚刚在那套流程里面，有产品的 Agent 也有研发的 Agent，那后面我们其实可以拓展更多的 Agent，并且做到让 Agent 跟 Agent 之间彼此协同。你们可以看到在三省六部制的总控台里面，可以去监控到它的进度，让它们彼此之间真正做到自动协同。我觉得这可能是未来的一种趋势，也是一人公司真正能够实现的很核心的要素。那如果你喜欢今天的这期视频的话，欢迎点赞收藏评论哟，拜拜。

### 2.3 核心概念精讲

#### 多 Agent 协同工作流

**是什么**：将一个复杂任务拆分给多个专职 Agent，每个 Agent 只负责自己最擅长的环节，Agent 之间传递中间成果，最终完成整体目标。

**为什么重要**：单一 Agent 承担所有任务会导致上下文混乱、能力不专注；多 Agent 分工让每个环节都能发挥最大效能，类似现实中的团队分工。

**详细解释**：
| 角色 | 对应 Agent | 职责 |
|---|---|---|
| 产品大管家 | "套"（自定义） | 聊需求、生成原型图（Askma） |
| 程序员 | 代码行（内置） | 读原型图、生成 HTML 网站 |

**实际应用场景**：用户在"套"里描述产品想法，得到原型图后，把原型图交给代码行，直接生成可交互的网页，全程不需要离开同一个应用。

---

#### 原型图驱动开发（Askma 方法）

**是什么**：先让 AI 根据需求生成原型图（Askma），再把原型图作为输入让 AI 生成代码，而非直接从需求描述跳到代码。

**为什么重要**：原型图是需求和代码之间的"翻译层"，它让 AI 生成的代码更还原需求，减少反复修改，节省 token 消耗。

**详细解释**：
| 对比维度 | 直接描述→代码 | 原型图→代码 |
|---|---|---|
| 还原度 | 较低，AI 可能乱发挥 | 高，按原型图实现 |
| Token 消耗 | 较高（反复修改） | 较低（一次到位） |
| 适合人群 | 有代码基础 | 无代码基础的产品人 |

**实际应用场景**：文科产品经理用自然语言描述需求，AI 先画出原型图供确认，再根据确认后的原型图生成代码，避免"做出来不是我想要的"。

---

#### 三省六部制方法论

**是什么**：一个开源项目，借鉴唐朝三省六部制的组织架构，将 AI Agent 的协同模式设计为层级化的决策-执行体系，并提供总控台监控各 Agent 进度。

**为什么重要**：它将多 Agent 协同从"工具技巧"上升为"组织方法论"，提供了一套可扩展的 AI 团队协作框架，被认为是"一人公司"实现的核心要素。

**详细解释**：
| 角色 | 对应职能 |
|---|---|
| 皇上（用户） | 下达最终指令 |
| 太子 | 判断任务是否执行 |
| 中书 | 规划执行方案 |
| 门下 | 审议方案 |
| 尚书 | 执行 |
| 六部 | 各司其职执行具体模块 |

**实际应用场景**：在三省六部制总控台中，可以实时监控各 Agent 的执行进度，实现真正的自动协同。

### 2.5 延伸思考

1. **多 Agent 各司其职的理念很美好，但 Agent 之间如何传递上下文？** 当产品 Agent 生成的原型图有歧义时，研发 Agent 会如何处理？中间成果的质量如何保证？

2. **三省六部制的"皇上下达指令→层级执行"模式，对用户的需求描述能力有什么要求？** 如果用户的初始需求不够清晰，多层 Agent 的误差会如何放大？

3. **视频说多 Agent 协同是"一人公司的核心要素"，这意味着什么？** 一个人通过多 Agent 体系能完成一个完整产品的哪些环节？还有哪些环节目前 AI 还无法替代？

### 2.6 一句话记住它
