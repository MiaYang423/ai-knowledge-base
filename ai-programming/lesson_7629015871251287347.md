# 教案：LLM Wiki 个人知识库新范式

> 来源视频：[抖音博主 · 抖音](https://www.douyin.com/video/7629015871251287347)
> 适合人群：对 AI 知识库感兴趣、正在使用或考虑搭建 RAG 系统的个人开发者和研究员

---

## 一、视频原文

放弃 RAG，LLM Wiki 才是个人知识库的正确方向。OpenAI 前科学家 Andrej Karpathy 亲自下场说了一句很扎心的话：很多人搭知识库的思路从根上就错了。今天这期我们就来把他的核心观点彻底讲清楚。

先说现状。一提到 AI 知识库，大多数人第一反应就是 RAG——把文档上传、切片、向量化，存进向量库，用户提问的时候，召回相关片段，再拼上下文生成答案。这套流程能用，而且确实是目前最主流的方案。

但 Karpathy 指出它有三个本质缺陷。第一，没有知识积累，不管你问多少次，LLM 每次都在零时拼凑片段，永远不会真正学会什么，相当于一个每次都会失忆的助手。第二，跨文档推理很弱，向量检索靠相似度，召回的是零散段落，一旦问题需要跨多篇文档综合推理，逻辑很容易断裂。第三，工程链路太重，向量库、重排序，对个人开发者来说维护成本极高。

那 Karpathy 提出了什么替代方案？他叫它 LLM Wiki，核心思想就一句话：把 LLM 当成编辑而不是检索工具，让它把你的原始资料一次性编辑成一个结构化、可维护的个人维基，后续反复复用，不用每次重新检索。

整个系统只有三个文件夹。Raw，斜杠，存放原始资料，只进不出，是知识的源头。Wiki 是 LLM 自动维护的结构化知识库，全部是 Markdown 文件。Templates 存放模板和提示词，规范生成逻辑。就这三个文件夹，极简。

具体工作流分四步。第一步，Ingest，导入，看到好文章好资料直接丢进 Raw，不用手动分类整理，越快越好。第二步，Compile，编译，让 LLM 读取新内容，自动提炼关键概念，撰写词条，建立页面间的链接，更新总索引，这一步只做一次，后续反复复用。第三步，Query，查询，提问时直接从 Wiki 里读取结构化内容回答，不再反复扫描原始文档。第四步，Lint，校验，定期让 LLM 扫描整个 Wiki，自动处理信息不一致、孤立无链接的页面、过时内容。整个过程没有向量库，没有 Embedding，没有复杂检索。

拍开来看两者的直接对比。在知识积累上，RAG 每次零时拼凑，LLM Wiki 持续沉淀，形成复利。可读性上，RAG 是黑盒向量库，LLM Wiki 全部 Markdown，全透明。工程复杂度上，RAG 需要向量库加重排序，LLM Wiki 就是文件夹加 LLM。跨文档推理上，RAG 碎片召回容易断裂，LLM Wiki 结构化上下文更连贯。个人可落地性上，RAG 较重成本高，LLM Wiki 极轻，开箱即用。对个人开发者和研究员来说，这个差距是非常明显的。

Karpathy 自己的工具栈也非常接地气，完全可以照着做。本地文件夹，纯文本，Obsidian 做知识 IDE，利用它的反向链接、知识图谱和 Dataview 查询。LLM 用 Claude 这类大上下文模型。Hash prompt 缓存大幅降低 Token 消耗。用 Git 做版本管理，记录所有 Wiki 修改点。

一天的流程是：网页收藏存入 Raw，执行编译更新 Wiki，从 Wiki 直接获取答案，把好的解释保存为新词条，最后定期让 LLM 自检修复。

总结一下，Karpathy 这波分享本质是在纠正一个行业惯性——一提到知识库就 RAG，不一定是最优解。LLM Wiki 给我们提供了一条更简单、更接近人类学习方式的路径，不是每次都临时翻书找答案，而是先把知识整理成体系，再基于体系回答问题，知识库会越用越聪明，真正形成复利。

用 Karpathy 自己的比喻来结尾：Obsidian 是知识 IDE，LLM 是程序员，Wiki 是代码库。不需要花哨的技术，只需要一个持续生长的知识体系。

好，今天的内容就到这里了，你觉得未来 LLM Wiki 会取代 RAG 吗？评论区讨论一下。

### 2.3 核心概念精讲

#### 概念一：RAG 的三大本质缺陷

**是什么**：RAG（检索增强生成）是目前最主流的 AI 知识库方案，但 Karpathy 认为它在设计上存在三个根本性问题。

**为什么重要**：理解这三个缺陷，是判断"什么场景下 RAG 不够用"的前提，也是理解 LLM Wiki 为何被提出的出发点。

**详细解释**：

| 缺陷 | 具体表现 | 类比 |
|------|----------|------|
| 没有知识积累 | 每次提问都重新从向量库召回片段，LLM 不会"记住"任何东西 | 一个每次都会失忆的助手 |
| 跨文档推理弱 | 向量检索靠相似度，召回的是零散段落，跨多篇文档的综合推理容易逻辑断裂 | 临时翻书，拼不出完整逻辑 |
| 工程链路太重 | 需要向量库、Embedding、重排序等组件，个人开发者维护成本极高 | 为了喝水架了一套自来水管道 |

**实际应用场景**：当你发现 RAG 系统回答跨主题问题时答非所问，或者维护向量库让你精疲力竭，这三个缺陷就在直接影响你。

---

#### 概念二：LLM Wiki 核心思想——LLM 作为编辑

**是什么**：不把 LLM 当实时检索工具，而是让它扮演"编辑"角色，把原始资料一次性整理成结构化的个人维基，后续直接复用。

**为什么重要**：这是 LLM Wiki 和 RAG 最根本的哲学差异——"整理在前，回答在后"vs"提问在前，临时拼凑在后"。

**详细解释**：

- **RAG 的逻辑**：提问 → 临时检索 → 拼凑上下文 → 生成答案（每次都从零开始）
- **LLM Wiki 的逻辑**：资料进来 → LLM 编辑整理 → 沉淀为 Wiki → 提问时直接读取结构化知识（一次整理，反复复用）

核心比喻（Karpathy 原话）：
> Obsidian 是知识 IDE，LLM 是程序员，Wiki 是代码库。

**实际应用场景**：研究员积累某个领域的文献时，与其每次问 AI"帮我找相关内容"，不如让 AI 把文献整理成一个持续生长的知识图谱，之后所有问题都基于这个图谱回答。

---

#### 概念三：四步工作流

**是什么**：LLM Wiki 的日常操作流程，分为 Ingest（导入）、Compile（编译）、Query（查询）、Lint（校验）四步。

**为什么重要**：这四步把"知识管理"变成了可执行的具体动作，降低了落地门槛。

**详细解释**：

| 步骤 | 动作 | 关键原则 |
|------|------|----------|
| Ingest（导入） | 看到好资料直接丢进 Raw 文件夹 | 不分类、不整理，越快越好 |
| Compile（编译） | 让 LLM 读取新内容，提炼概念，撰写词条，建立页面链接，更新总索引 | 只做一次，后续反复复用 |
| Query（查询） | 提问时直接从 Wiki 结构化内容中获取答案 | 不再扫描原始文档 |
| Lint（校验） | 定期让 LLM 扫描 Wiki，处理信息不一致、孤立页面、过时内容 | 保持知识库健康可用 |

**实际应用场景**：每天的使用节奏——早上收藏几篇文章存入 Raw，执行一次 Compile 更新 Wiki，工作中遇到问题直接查 Wiki，每周跑一次 Lint 做维护。

---

#### 概念四：推荐工具栈

**是什么**：Karpathy 自己使用的一套极简本地工具组合，用于实现 LLM Wiki。

**为什么重要**：证明这套方案不需要复杂基础设施，个人开发者完全可以用现成工具落地。

**详细解释**：

| 工具 | 角色 | 用途 |
|------|------|------|
| 本地文件夹 + 纯文本 | 存储层 | Raw / Wiki / Templates 三个文件夹 |
| Obsidian | 知识 IDE | 反向链接、知识图谱、Dataview 查询 |
| Claude | LLM | 大上下文窗口，适合一次性读取大量 Wiki 内容 |
| Hash prompt 缓存 | 成本优化 | 大幅降低重复读取 Wiki 的 Token 消耗 |
| Git | 版本管理 | 记录所有 Wiki 修改历史，可回溯 |

**实际应用场景**：零成本起步——建三个文件夹，装 Obsidian，接入 Claude API，就可以开始跑 LLM Wiki 了。

| 对比维度 | RAG | LLM Wiki |
|----------|-----|----------|
| 知识积累 | ❌ 每次零时拼凑，不积累 | ✅ 持续沉淀，形成复利 |
| 跨文档推理 | ❌ 碎片召回，逻辑易断裂 | ✅ 结构化上下文，更连贯 |

| 个人可落地性 | ❌ 较重，维护成本高 | ✅ 极轻，开箱即用 |
| 核心角色定位 | LLM 是检索工具 | LLM 是编辑 |
| 适合场景 | 大规模企业文档检索 | 个人知识积累与研究 |

### 2.5 延伸思考

1. **LLM Wiki 的 Compile 步骤依赖 LLM 的理解质量，如果 LLM 在编辑时引入了错误或偏差，Wiki 里的知识会随时间不断"污染"——这个风险应该如何设计机制来控制？**

2. **RAG 和 LLM Wiki 真的是非此即彼的替代关系吗？对于需要实时检索海量企业文档的场景，LLM Wiki 的"一次编译"模式是否现实？两者有没有可能融合？**

3. **Karpathy 的方案高度依赖大上下文窗口模型（如 Claude）来一次性读取整个 Wiki，随着个人知识库不断增长，上下文窗口的上限和 Token 成本会不会成为这套方案的天花板？**

---

### 2.6 一句话记住它

> RAG 是每次临时翻书，LLM Wiki 是先把书读透再回答——知识库应该越用越聪明，而不是每次都从零开始。
