## 一、视频原文

2026年了，写代码的方式层出不穷，今天从好到差锐评一下主流的开发方式。

首先出场的是我们的老大哥——纯手写编程。这种编程方式使用传统代码生成工具和开发者手写代码进行构建程序。因为模型能力逐渐变强，在24、25年还有大力说自己坚持不使用 AI，但如今已经成为过去，只有盲信和骗门领域的大佬才会使用，出于情怀，给到 NPC 评级。

其次是网页对话式编程。通过网页端对话要 AI 生成代码，因为 AI 无法感知到本地环境，只能靠用户语言描述或截图给 AI，所以很多时候无法快速定位上下文内容，需要手动复制，也是问题。写扫雷、贪吃蛇什么的程序还好，但无法应用于实际生产，直接给到"拉"评级。

接下来是第三种，IDE 插件式编程，比如市面上比较火的 Cursor，也可以使用 VS Code 的插件来实现 IDE 插件式编程。能读取本地环境，AI 自己跑命令验证代码有没有问题，还有很多便捷的功能。因为能看到项目文件，是大部分老程序员会使用的方式，给到顶级评级。

接下来是命令行编程，主流有 Craft Code、Conduct。该方式依赖于大模型强大的能力，不注重代码而注重想法，想要什么功能自己描述一下就好。但因为无法看到项目文件，失去了对代码的掌控感，所以是没有代码基础的人用得多，拿来写大模型和中小项目都合适。擅长使用的人能做得很好，不擅长使用的人一个功能可能需要多次修改，也是给到顶级评级。

接下来看手机编程。通过手机下达命令，让电脑的 Craft Code、Codex 这类命令行工具写代码。适合灵感突发时，就拿出手机让 AI 去做，无论你在健身房、走路、吃饭，都可以拿出手机让 AI 进行编程，完成你的想法。但手机屏幕小，信息量少，请先用于做一些小 demo 验证灵感，因此给到"人少人"评级。

接下来要出场的是命令行自动编程。给 AI 装很多 MCP 工具，让 AI 能自己打开浏览器、程序，自己截图找问题，你写好详细的计划让 AI 自己执行。但这只适合编程大佬，新人提出来的计划可能需求是相冲的，导致 AI 难以实现。但可以预料到这是未来的编码方式，因此给到"人少人"评级。

最后一种是多 Agent 编程，比较流行的概念。使用一个命令行来进行编写代码时，你需要花几分钟等待模型的输出；而多 Agent 没有这个问题，你有了想法就可以快速验证，几个 Agent 写代码，几个 Agent 进行测试，几个 Agent 进行合并冲突，是目前最新潮的编码方式，也是效率最高的，直接给行。

### 2.3 核心概念精讲

#### IDE 插件式编程

**是什么**：在集成开发环境（IDE）中通过插件接入 AI，使 AI 能读取本地项目文件并直接在开发环境中辅助编程。

**为什么重要**：能读取本地环境意味着 AI 能感知完整的项目上下文，自动跑命令验证代码，是目前对专业开发者最实用的方式。

**详细解释**：
- 代表工具：Cursor、VS Code 插件
- 核心优势：读取本地环境、AI 自动运行命令验证代码
- 适合人群：大部分有代码基础的程序员

**实际应用场景**：在 Cursor 中直接编辑大型项目，AI 能理解整个代码库上下文并给出精准建议。

---

#### 命令行编程

**是什么**：通过命令行界面与 AI 交互，以自然语言描述需求，让 AI 生成并执行代码，无需手动操作 IDE。

**为什么重要**：降低了编程门槛，让没有代码基础的人也能通过描述想法来实现功能；同时也是编程大佬高效开发的利器。

**详细解释**：
| 维度 | 内容 |
|---|---|
| 代表工具 | Craft Code、Conduct |
| 核心理念 | 注重想法，不注重代码本身 |
| 适合项目 | 大模型、中小项目 |
| 潜在问题 | 无法看到项目文件，掌控感弱 |
| 适合人群 | 无代码基础者 + 擅长使用的高手 |

**实际应用场景**：新手可以用自然语言描述需求直接生成完整项目；高手可以快速原型验证。

---

#### 多 Agent 编程

**是什么**：多个 AI Agent 并行协作完成编程任务，不同 Agent 分别负责写代码、测试、合并冲突等不同环节。

**为什么重要**：解决了单一命令行编程"串行等待"的效率瓶颈，通过并行化大幅提升开发速度，是目前评级最高的编程方式。

**详细解释**：
| 对比维度 | 单 Agent 命令行 | 多 Agent 编程 |
|---|---|---|
| 任务执行 | 串行，需等待 | 并行，同时进行 |
| 效率 | 较低 | 最高 |
| 适合场景 | 中小项目 | 复杂项目快速验证 |
| 当前成熟度 | 成熟 | 最新潮，快速发展中 |

**实际应用场景**：几个 Agent 同时写不同模块的代码，几个 Agent 同时跑测试，几个 Agent 处理代码合并冲突。

### 2.4 速查表

| 编程方式 | 评级 | 代表工具 | 适合人群 | 核心优势 | 主要限制 |
|---|---|---|---|---|---|
| 纯手写 | NPC | 传统 IDE | 特定领域大佬 | 完全掌控 | 效率最低 |
| 网页对话 | 拉 | ChatGPT 等 | 练手/学习 | 零门槛 | 无法读取本地环境 |
| IDE 插件 | 顶级 | Cursor | 有代码基础的程序员 | 读取本地环境 | 需要一定代码基础 |
| 命令行 | 顶级 | Craft Code | 无基础者/高手 | 注重想法 | 掌控感弱 |
| 手机编程 | 人少人 | 手机+Codex | 灵感验证 | 随时随地 | 屏幕小，信息量少 |
| 命令行自动 | 人少人 | MCP+命令行 | 编程大佬 | AI 自主执行 | 新人难以驾驭 |
| 多 Agent | 行 | — | 追求极致效率者 | 并行最高效 | 最新潮，学习成本 |

### 2.5 延伸思考

1. **视频给命令行编程和 IDE 插件编程都打了"顶级"，但适合人群截然不同，如何判断自己该用哪个？** 一个有一定代码基础但不是资深程序员的人，应该从哪种方式入手？

2. **多 Agent 编程被评为效率最高，但目前成熟度如何？** 多个 Agent 并行时如何协调避免代码冲突？这种方式对项目规模和复杂度有什么要求？

3. **命令行自动编程（配合 MCP）被说是"未来的编码方式"，但目前只适合编程大佬，这个门槛会如何降低？** 随着 AI 能力提升，普通用户什么时候能安全地使用"让 AI 自主执行计划"的编程方式？

---

### 2.6 一句话记住它

> 编程方式的进化路线是：手写→对话→插件→命令行→多 Agent，你现在站在哪一步，决定了你的开发效率上限。
