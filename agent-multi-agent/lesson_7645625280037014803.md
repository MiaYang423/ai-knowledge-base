## 一、视频原文

一起来看看这套美团的Agent面经，看看你能答上来几道。这套题整体难度中等，但是最后一题很多人会答得很虚。我是小哲，点赞收藏加关注，我们马上开始。

第一道题：什么是MCP？它的核心目标是什么？MCP可以理解成Agent连接外部工具和数据源的一套标准协议，核心目标是把工具发现、上下文传递和调用方式标准化，让不同客户端和服务端可以复用，而不是每个项目都重新接一遍。

第二道题：MCP主要解决了什么问题？它解决的是工具接入碎片化的问题。过去每个Agent框架都有自己的工具协议，千奇百怪，接口和描述也不同。MCP想做的是一层标准接口，让工具能力更容易被发现、标用和治理。

第三道题：MCP里的Host是什么？Host是承载用户交互和模型能力的应用，比如IDE、桌面助手或Agent平台。它负责发起连接、管理会话、协调模型和工具结果。简单说，Host是整个MCP使用体验的入口和控制面。

第四道题：MCP里的Client是什么？Client通常运行在Host的内部，负责和某个MCP Server建立连接、发送请求、接收工具列表和执行结果。它像适配层，把Host的需求转换成协议消息，也把Server的能力暴露给上层。

第五道题：MCP里的Server是什么？Server负责提供具体能力，比如文件系统、数据库、搜索、代码仓库或业务系统。它声明有哪些工具、参数怎么传、返回什么结果。工程上Server还要处理鉴权、限流、日志和错误信息。

第六道题：MCP和Function Calling有什么区别？Function Calling更偏模型侧的工具调用格式，重点是让模型按Schema产出参数。MCP更偏系统侧协议，重点是工具如何被发现、连接和治理。两者可以配合，不是简单替代关系。

第七道题：MCP和Plugins有什么区别？Plugins更偏具体平台的扩展机制，通常依赖某个生态。MCP更强调跨应用、跨工具的标准化连接。面试里可以说：Plugins是一种产品形态，MCP更偏底层协议和集成方式。

第八道题：生产环境落地MCP要注意什么？关键是通信安全、权限隔离、工具白名单、参数校验、审计日志和超时容断。工具发现不能等于工具可用，敏感操作要二次确认，还要考虑多租户、版本兼容和Server异常时的降级。

### 2.3 核心概念精讲

#### MCP三层架构：Host / Client / Server

**是什么**：MCP将Agent的工具调用拆分为三个角色，各司其职。

**为什么重要**：清晰的分层让工具能力可以跨应用复用，不同的Host可以接入同一个Server，降低重复开发成本。

**详细解释**：

| 角色 | 位置 | 职责 | 示例 |
|------|------|------|------|
| Host | 最外层 | 用户交互入口，发起连接，管理会话 | IDE、桌面助手、Agent平台 |
| Client | Host内部 | 协议适配，连接Server，转换消息 | 协议适配层 |
| Server | 后端 | 提供具体能力，声明工具和参数 | 文件系统、数据库、搜索服务 |

**实际应用场景**：一个IDE（Host）内置Client，连接多个MCP Server，分别提供代码搜索、文件读写、数据库查询等能力，模型可以统一调用。

---

#### MCP vs Function Calling vs Plugins

**是什么**：三种增强Agent能力的不同机制，层次和侧重点各不相同。

**为什么重要**：面试中经常被混淆，能清晰区分说明对Agent系统架构有深入理解。

**详细解释**：

| 概念 | 层次 | 侧重点 | 关系 |
|------|------|--------|------|
| Function Calling | 模型侧 | 让模型按Schema产出参数 | 可与MCP配合使用 |
| MCP | 系统侧 | 工具发现、连接、治理的标准协议 | 底层基础设施 |
| Plugins | 产品侧 | 特定平台的扩展机制 | 依赖某个生态 |

**实际应用场景**：模型通过Function Calling决定调用哪个工具，MCP负责标准化地把调用请求路由到对应的Server，Plugins是某个平台（如ChatGPT）对外暴露的扩展能力形态。

---

#### 生产落地的安全与工程要点

**是什么**：MCP在真实生产环境中需要额外处理的工程和安全问题。

**为什么重要**：这是第八题"很多人会答得很虚"的原因——概念好答，落地细节才是真正的工程能力体现。

**详细解释**：
- **安全层**：通信安全、权限隔离、工具白名单、参数校验
- **可靠性层**：审计日志、超时容断、Server异常降级
- **工程层**：多租户支持、版本兼容、敏感操作二次确认
- **关键原则**：工具发现 ≠ 工具可用，不能因为发现了工具就直接允许调用

**实际应用场景**：Agent要帮用户发送重要邮件时，必须有二次确认机制，不能让模型自动触发敏感操作。

### 2.5 延伸思考

1. 视频说"工具发现不能等于工具可用"——在实际产品设计中，应该如何设计工具的权限分级和动态白名单机制？

2. MCP的三层架构（Host/Client/Server）和传统的B/S架构有何相似之处？这种分层对AI产品的可扩展性有什么具体帮助？

3. 当多个MCP Server提供相似功能时（如两个不同的搜索Server），模型应该如何决策选择哪个？这个决策逻辑应该放在哪一层处理？
