# 教案：智能体意图识别的工程落地方案

> 来源视频：[抖音博主 · 抖音](https://www.douyin.com/video/7637462285053509113)
> 适合人群：正在准备 AI Agent 面试或从事智能体产品开发的工程师和算法工程师

## 一、视频原文

面试官问你智能体意图识别的方法，你要是直接说写个 Prompt 调用大模型就行，那这场面试基本就凉了。大家好，我是 Jack。

遇到意图识别，很多人第一反应就是把用户的话直接丢给大模型，让它给个分类标签。比如用户说"帮我查邮件"，模型返回"查询邮件"的标签，简单粗暴。但这在面试官眼里就是灾难。为啥呢？用户说话很随意，像老板说"帮我看看上个月谁没交周报，顺便发邮件催一下"，这是查数据和发邮件两个意图，要是系统只能输出一个分类，漏掉哪个都得挨骂。更可怕的是，用户说"把客户资料全删了"，单靠语义模型判断，万一它出错真删了就出大事了。

那到底怎么解决呢？面试官想听的大招，不是一堆模型乱堆，而是一套可落地的意图分类体系。首先你得明白，真实的智能体产品意图一定是个有限集合。比如你做一个办公助手，意图无非就是查邮件、发邮件、查日程、删文件这么十几二十类。定义清楚之后，意图识别就变成了一个分类任务。

怎么让分类又准又快、成本还扛得住？工业界的常用打法是规则加模型并行。安全后置——别一上来就全甩给大模型。像查邮件、发周报这种高频固定的意图，直接用规则做关键词匹配，速度极快，成本几乎为零。只有规则兜不住的模糊场景，才交给语义模型去理解。语义模型可以是一个微调过的小参数模型，也可以是大模型的 few-shot 分类。但调用之前先让规则过一遍，能挡掉大半流量，成本一下子就控制了。

这两路并行跑，结果汇总产生最终意图。如果结果冲突，再用一个简单的裁决逻辑：比如语义模型置信度高就听语义模型的；规则命中并且语义模型置信度低，就走人工兜底或反问澄清。这个思路既保证了准确率，又没有盲目消耗算力。

那上下文里出现"不对不对"这种自我纠错怎么办？这就是多轮对话里意图识别的必备能力了。实际处理时会把前一轮或前几轮发言作为历史信息一起传给意图分类模型，模型自然能学到用户刚刚改口了，现在意图是查风险条款，不再是找合同。这比专门训练一个上下文专家轻量得多，也更符合端到端落地的习惯。

至于安全方面，比如"删客户资料"这种高危操作，它确实不属于意图识别这个环节。工程上安全的做法是：意图识别只管输出"删除资料"这个标签，后续会有一个独立的 Guardrail 安全策略层，在动作真正执行前对指令做全量校验、二次确认和风险拦截。这样职责清晰，不会把安全逻辑硬塞进意图分类模型，导致模型又慢又难调。

咱们还用之前那个例子过一遍：用户说"帮我把昨天的合同找出来，不对不对，还是先查查有没有风险条款吧"。规则引擎瞬间抓到"合同""风险条款"，初步命中"合同风险分析"这个意图；同时语义小模型读取上文，理解到用户已经改口，也给出了"合同风险分析"的结果，而且置信度很高。两路结果一致，意图直接确定为"合同风险分析"，整个识别过程只调用了一次轻量模型，规则又凌成本，快又省。识别完成后，后面才是工具调度：查文件、调法务分析接口，而安全护栏在工具执行前会最后审一遍，"合同风险分析"是读操作，不涉及删除，放行。

你看，这样分阶段设计，每一步都清晰可控。回头看这道题：智能体做意图识别，核心就是垂直有限意图的分类；落地时用规则加小模型并行提效控成本；把安全校验后置到策略层，而不是一开始就塞进意图识别。

最后留个思考题：并行时规则和模型给出的意图标签不一致，你觉得裁决逻辑应该怎么设计？欢迎在评论区聊聊，今天视频就到这，咱们下期见，拜拜。

## 二、核心教案

### 2.1 这个视频在讲什么（一句话）

**智能体意图识别不能只靠大模型，工业界的正确打法是：有限意图分类 + 规则与语义模型并行 + 安全校验后置，三层设计缺一不可。**

---

### 2.2 知识地图

```
智能体意图识别工程方案
├── 核心前提：意图是有限集合
│   └── 先定义清楚意图列表（十几二十类）
├── 识别架构：规则 + 语义模型并行
│   ├── 规则层（第一道）
│   │   ├── 关键词匹配、正则
│   │   ├── 适用：高频固定意图
│   │   └── 成本极低、速度极快
│   ├── 语义模型层（第二道）
│   │   ├── 微调小模型 or 大模型 few-shot
│   │   ├── 适用：模糊、复杂意图
│   │   └── 规则先过滤，只处理漏网场景
│   └── 裁决逻辑（结果冲突时）
│       ├── 语义模型置信度高 → 听语义模型
│       └── 规则命中且语义置信度低 → 人工兜底/反问
├── 多轮对话：历史上下文注入
│   └── 前几轮发言作为历史一起传入分类模型
└── 安全后置：独立 Guardrail 层
    ├── 意图识别只输出标签
    └── 执行前由安全策略层做校验和二次确认
```

---

### 2.3 核心概念精讲

#### 有限意图集合

**是什么**：将智能体能处理的所有意图预先定义为一个有限的分类集合（如查邮件、发邮件、查日程等）。

**为什么重要**：把开放问题转化为有限分类问题，是意图识别从"玄学"变成"工程"的第一步。没有这一步，后续的规则和模型都无从设计。

**详细解释**：
- 真实产品的意图通常只有十几到二十类，不是无限开放的
- 先定义清楚意图列表，再设计识别方案，而不是反过来

**实际应用场景**：设计办公助手时，先列出所有可能的意图类型，再针对每类设计识别逻辑。

---

#### 规则 + 语义模型并行架构

**是什么**：同时运行规则引擎和语义模型，两路结果汇总后通过裁决逻辑得出最终意图。

**为什么重要**：纯规则覆盖不了模糊意图，纯 LLM 成本高且有随机性；并行架构取两者之长，是工业界的标准打法。

**详细解释**：

| 方式 | 适用场景 | 优点 | 缺点 |
|------|----------|------|------|
| 规则引擎 | 高频固定意图 | 极快、成本极低、无随机性 | 覆盖不了模糊场景 |
| 语义模型 | 模糊、复杂意图 | 泛化能力强 | 有延迟、有成本、有随机性 |
| 并行+裁决 | 所有场景 | 兼顾准确率和成本 | 需要设计裁决逻辑 |

**实际应用场景**：规则先过滤高频请求，拦截大半流量后，剩余模糊请求再走语义模型，整体成本大幅下降。

---

#### 安全后置（Guardrail 层）

**是什么**：将安全校验从意图识别中剥离，作为独立的策略层在动作执行前进行全量校验和风险拦截。

**为什么重要**：把安全逻辑塞进意图分类模型会导致模型又慢又难调，职责不清晰。独立的 Guardrail 层让每个环节各司其职。

**详细解释**：
- 意图识别只负责输出意图标签（如"删除资料"）
- 安全策略层在工具真正执行前介入，做二次确认和风险拦截
- 高危操作（删除、修改敏感数据）必须经过 Guardrail 层

**实际应用场景**：用户说"把客户资料全删了"，意图识别输出"删除资料"标签，Guardrail 层拦截后要求二次确认，避免误操作。

---

### 2.4 速查表

| 环节 | 方式 | 核心原则 |
|------|------|----------|
| 意图定义 | 预先枚举有限集合 | 先定义，再识别 |
| 高频固定意图 | 规则引擎（关键词/正则） | 成本极低、速度极快 |
| 模糊复杂意图 | 语义小模型/大模型 few-shot | 规则先过滤再调用 |
| 结果冲突裁决 | 置信度比较 + 人工兜底 | 语义置信度高则优先 |
| 多轮对话 | 历史上文注入分类模型 | 轻量处理上下文改口 |
| 安全校验 | 独立 Guardrail 层 | 后置于意图识别，执行前拦截 |

---

### 2.5 延伸思考

1. 视频说"意图是有限集合"——但随着产品迭代，意图类型会不断增加，如何设计一套可扩展的意图管理机制，避免每次新增意图都要大改系统？

2. 并行架构中，规则和语义模型的裁决逻辑是这道题的思考题——如果两路置信度都不高，应该走人工兜底还是反问用户澄清？各自的适用条件是什么？

3. 视频提到"安全后置到 Guardrail 层"——但 Guardrail 层本身如何保证不被绕过？比如用户通过多轮对话逐步引导 Agent 执行高危操作，Guardrail 层应该如何设计才能防范这类攻击？

---

### 2.6 一句话记住它

> 意图识别不是"丢给大模型猜"，而是有限分类+规则语义并行+安全后置——每一层各司其职，才是工业级的落地方案。
