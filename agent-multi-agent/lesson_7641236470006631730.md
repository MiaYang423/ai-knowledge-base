## 一、视频原文

大家好，我是阿里长长，今天聊一个被 Karpathy 亲口宣判结束的时代——氛围编程。

前几天 Karpathy 跟红杉的 Stephanie Zhan 录了一期对话，标题就一句话：From Vibe Coding to Agentic Engineering。距离他第一次抛出 vibe coding 这个词，刚好一年。

一年前他说，未来你只要描述想要啥，模型就给你写出来。当时整个圈子都嗨了，Cursor、Claude Code、Codex、Windsurf 一个个起飞，所有人都觉得程序员要消失了。

结果一年过去，他亲口说那个时代结束了。

我自己这一年是 Claude Code 的重度用户，跟你说真的，那种我啥也不懂但功能出来的快感是真的爽。打开 Cursor 跟他说一句"帮我做个 dashboard"，回车，啪一下界面就出来了。爽完之后我就开始懵：为啥我的登录流程里有一个叫 Banana.ts 的文件？这是哪个 Agent 的恶趣味？我也不知道，能跑就行。

这种状态，Karpathy 管它叫 vibe coding——靠氛围，靠感觉，靠运气。但他说这玩意儿只配做原型。

接下来他抛了个挺狠的框架，叫 Software 3.0：Software 1.0，人手写代码；Software 2.0，神经网络从数据里学；Software 3.0，LLM 本身变成计算基底，自然语言就是编程接口，Agent 就是 runtime。

听着挺选对的，但你回头琢磨一下，他说的是实话——你现在写代码的时候真的在写代码吗？你大部分时间是在写 spec，在配 context，在跑测试，在审 diff，在管 Agent 的权限。你写的是给模型看的提示词，模型才是真正在敲键盘的那个家伙。

这一年我最大的感受是：老程序员变成新手了。我有个朋友写了十几年，够骨灰级的那种，他前两个月跟我吐槽，用 Claude Code 打了一周，觉得自己跟刚毕业的实习生没两样。不是说写不出代码，是 Agent 一次性给他吐八百行他没写过的东西，他看半天看不懂，但又必须 code review，必须签字背锅。他说他现在不是在编程，是在签字。

这就是核心矛盾：模型越强，传统意义上的"会写代码"越不能保证你真的懂这个系统。

所以 vibe coding 和 agentic engineering 的区别在哪？Vibe coding 是你说做个东西，模型给你做，看起来还行，可能能跑，前端可能是个大型车祸现场，但你看到像素动了，demo 能演了，你就觉得自己很牛。

Agentic engineering 是另一套东西，它要 memory、要 tools、要测试、要多 Agent 协作、要沙箱、要监控、要 code review，还要 spec、安全审计，一堆工程纪律。一个像魔术师从帽子里变戏法，一个像工程纪律，tools 还在，但 tools 身上挂着 CI 了。

Karpathy 说了句让我印象很深的话：一个没有约束的 coding Agent，不是同事，是一个拥有 root 权限、精力无限、偶尔会犯绝的初级工程师。这不是劳动力策略，这是人质危机。

这话听着有点刺耳，但是事实。你给 Agent 放开了，让他能改文件、能装包、能拿 secret、能推送分支、能部署，听着像是给你配了个全能助理，但只要他有一次犯错，你就完了。

所以未来不是一个万能 Agent，是一堆专精 Agent，每个有自己的工具、自己的权限、自己的成功标准，你的活是调度他们。

这事最有意思的是，市场已经在用脚投票了。微软和 GitHub 已经推出一个叫 GitHub Certified Agentic AI Developer 的认证，考试编号 GH600。一年前 vibe coding 听着像个段子，现在已经有证书认证考试了。

但另一边，Replit、Coding base、Cloudflare、IBM、Salesforce、Snap 都在裁人，Cloudflare 一次裁了 1100 多个岗位，名义上说就是为了适配 agentic AI 时代。

听着挺残酷，但我想了想，AI 其实没单纯替代谁，它做的是另一件事——它把那些看起来在工作但其实只是在会议之间传话的岗位暴露出来了。

我有时候觉得，这个行业正在经历一次身份重写。程序员这三个字的含义在变，以前它意味着你会写代码、挑架构、架系统，现在它意味着你会写 spec、管 Agent、做判断，知道什么该交给机器，什么必须人来。

知识越来越不稀缺，判断力越来越稀缺。

Karpathy 最后那句话我记到现在：Software 3.0 不是 slogan，它是程序员从代码作者变成计算劳动力操作者的那一刻。

说真的，我也不知道是该兴奋还是该焦虑。一方面我亲手用 Claude Code 几天搞出来以前要请三个工程师做一个月的东西，那种爽感是真实的。另一方面，我也清楚这种爽感的底层是什么——是我积累的判断力在给 Agent 擦屁股。那些没积累过这些判断力的人怎么办？我也没答案。

但我至少知道一件事：光会用 Claude Code，跟会工程化地使用 Agent，是两件完全不同的事儿。后者比前者难得多，但也值钱得多。

Karpathy 这次还是说对了，氛围编程已经结束了，接下来真正的游戏，才刚刚开始。

好，这期就到这里，觉得这期有用，记得点个关注，收藏一下，我是阿里长长，下一期喜欢听什么，欢迎评论区告诉我。

### 2.3 核心概念精讲

#### Vibe Coding vs Agentic Engineering

**是什么**：两种使用 AI Coding 工具的范式——前者靠感觉和运气快速出原型，后者用工程纪律和系统设计管理 Agent 完成生产级任务。

**为什么重要**：这是判断一个人 AI 编程能力段位的核心分水岭，也是当前市场对 AI 开发者的真实要求。

**详细解释**：

| 维度 | Vibe Coding | Agentic Engineering |
|---|---|---|
| 核心方式 | 描述需求，模型输出，靠感觉接受 | 写 spec，配 context，审 diff，管权限 |
| 适用场景 | 原型验证、demo 演示 | 生产级系统、复杂业务逻辑 |
| 结果质量 | 可能能跑，前端可能是车祸现场 | 有测试、有监控、有工程纪律 |
| 工程要素 | 基本没有 | memory/tools/多Agent/沙箱/CI/安全审计 |
| 程序员角色 | 魔术师（从帽子里变戏法） | 工程师（管理 Agent 团队） |

**实际应用场景**：个人 side project 用 vibe coding 快速验证想法；企业生产系统必须用 agentic engineering，否则 Agent 一次犯错可能导致数据丢失或安全事故。

---

#### Software 3.0

**是什么**：Karpathy 提出的软件发展三阶段框架，3.0 阶段 LLM 本身成为计算基底，自然语言是编程接口，Agent 是 runtime。

**为什么重要**：这个框架解释了为什么程序员的工作内容在变——你写的不再是代码，而是给模型看的 spec 和提示词。

**详细解释**：

| 阶段 | 特征 | 程序员做什么 |
|---|---|---|
| Software 1.0 | 人手写代码 | 直接编写逻辑 |
| Software 2.0 | 神经网络从数据学习 | 准备数据、设计网络结构 |
| Software 3.0 | LLM 为计算基底 | 写 spec、管 Agent、做判断 |

**实际应用场景**：在 Software 3.0 时代，程序员的日常工作变成：写清楚 spec → 给 Agent 配好 context → 跑测试 → 审 diff → 管权限边界。

---

#### 无约束 Agent 的风险

**是什么**：Karpathy 的警告——一个拥有 root 权限、精力无限但偶尔会犯错的 Agent，不是同事，是"人质危机"。

**为什么重要**：这是 agentic engineering 的核心工程原则：Agent 必须有权限边界、工具限制和成功标准，否则一次失误就可能造成不可逆损失。

**详细解释**：
- 危险能力组合：能改文件 + 能装包 + 能拿 secret + 能推送分支 + 能部署
- 正确做法：专精 Agent（每个有自己的工具范围、权限边界、成功标准）
- 人的角色：调度多个专精 Agent，而不是给一个万能 Agent 放开所有权限

**实际应用场景**：代码审查 Agent 只有读权限；部署 Agent 只能操作特定环境；数据库 Agent 只能查询不能写入——权限最小化原则。

### 2.4 速查表

| 对比维度 | Vibe Coding | Agentic Engineering |
|---|---|---|
| 定位 | 原型 / demo | 生产级系统 |
| 核心能力 | 会用 AI 工具 | 会管 Agent、做工程判断 |
| 工程要素 | 基本缺失 | memory/tools/测试/监控/安全审计 |
| Agent 权限 | 通常放开 | 严格最小化 |
| 程序员价值 | 知识（越来越不稀缺） | 判断力（越来越稀缺） |
| 市场认证 | 无 | GitHub GH600 认证 |

### 2.5 延伸思考

1. 视频中说"判断力越来越稀缺"——那些没有足够工程经验积累判断力的初级开发者，在 agentic engineering 时代应该如何建立自己的核心竞争力？
2. Karpathy 说无约束 Agent 是"人质危机"，但过度约束又会限制 Agent 的能力上限，实际工程中如何找到权限边界的最优解？
3. GitHub GH600 认证的出现说明 agentic engineering 已经开始职业化，这类认证体系能否真正衡量一个人管理 Agent 的实际能力？

---

### 2.6 一句话记住它

> 氛围编程只配做原型，真正的 AI 时代工程师是管 Agent 的人——判断力，才是 Software 3.0 时代最稀缺的技能。
